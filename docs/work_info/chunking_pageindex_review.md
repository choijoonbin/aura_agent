## 1. 참고 자료 요약

### 1.1 페이징 인덱스 기법이 하는 일

- **입력**: PDF/MD 문서 → 파싱 후 **페이지(page)** 단위 + **헤딩 트리(section_path)** 가 붙은 블록들.
- **청킹(parsing_v1)**  
  - PDF: 페이지별로 블록 머지 후, **고정 길이(TARGET_CHUNK_CHARS≈1000, OVERLAP≈80)** 로 슬라이딩 분할.  
  - MD: 헤더 트리(최대 레벨 3) 기준으로 구간 나누고, 텍스트는 **문자 수 타깃 + 오버랩**으로 청킹.  
  - 각 청크(leaf)에 **page**, **section_path** 부여.
- **노드 인덱스(node_index)**  
  - 섹션 경로별로 **요약(summary_short)**, **라우팅용 텍스트(router_text)**, **해당 페이지 목록(pages)**, **자식/리프 ID** 등을 가진 트리 구조를 별도로 둠.  
  - 검색 시 “어느 섹션이 관련 있는지”를 이 트리로 먼저 판단.
- **검색 흐름(pageindex.py)**  
  1. 전역 **리프 키워드 검색** → 점수 높은 리프 추출  
  2. 리프가 속한 **페이지**로 후보 페이지 집합 생성 (주변 페이지 확장 가능)  
  3. 후보 페이지에 등장하는 **section_path**로 시드 노드 선정  
  4. 시드에서 **빔 탐색**: 자식 노드를 확장하면서 title/summary/router_text로 노드 점수 계산, 점수·개선량 기준으로 종료  
  5. 최종 빔 노드 하위 **리프만** 모아서 컨텍스트 구성  
  6. 빔 실패 또는 리프/페이지 수 부족 시 **백스톱**: 전역 리프 히트 + (선택) 태그/엔티티로 쿼리 확장 후 재검색

즉, **페이지로 후보를 좁히고 → 섹션 트리를 빔으로 탐색해 관련 구간을 찾은 뒤 → 해당 구간의 리프만 사용**하는 구조다.

## 2. 우리 프로젝트 청킹과의 차이

| 항목 | masterdoc (페이징 인덱스) | AuraAgent (규정집 청킹) |
|------|---------------------------|---------------------------|
| **입력** | PDF/MD, 페이지 존재 | TXT 원문, **페이지 없음** |
| **청킹 단위** | 고정 길이(문자 수) + 오버랩, 페이지·헤딩 경계 반영 | **규정 구조**(조문→항→호) 기준, ARTICLE/CLAUSE/ITEM |
| **위치 정보** | page, section_path, bbox | semantic_group, regulation_article, regulation_clause, node_type |
| **인덱스 구조** | **node_index**(섹션 트리 + 요약 + pages)와 **leaf**(실제 텍스트) 분리 | 청크 한 종류; 각 행이 곧 검색 단위(ARTICLE/CLAUSE/ITEM) |
| **검색** | 페이지 시드 → 빔으로 섹션 확장 → 해당 리프만 사용, 백스톱 시 전역 리프 | semantic_group 등 필터 → BM25+Dense → RRF → rerank → 조문 단위 중복 제거 |

우리 쪽은 **규정 구조 기반**이라 “조·항·호” 단위가 이미 잘려 있고, **페이지** 개념이 없어서, 페이징 인덱스의 “페이지로 후보 좁히기”를 그대로 쓰기 어렵다.

## 3. 도입 검토 요약

### 3.1 그대로 쓰기 어렵거나 부적합한 부분

- **페이지 기반 시드/후보**  
  - 규정집은 TXT만 있어 **물리 페이지가 없음**.  
  - 나중에 PDF 규정집을 넣을 때는 “페이지” 메타를 추가한 뒤, 그때 페이지 기반 로직을 검토하는 편이 자연스러움.
- **고정 길이 + 오버랩 청킹**  
  - 우리는 이미 **구조 기반(ARTICLE/CLAUSE/ITEM)** 이고, 문서/이전 검토에서도 규정 문서에는 구조 기반이 정확도상 유리하다고 정리됨.  
  - 페이징 쪽의 “문자 수 타깃 청킹”은 우리 **인덱싱 전략으로 채택할 필요 없음**.

### 3.2 참고만 하고 선택 도입할 수 있는 부분

| 참고 기법 | 내용 | 우리 적용 시 고려사항 |
|-----------|------|------------------------|
| **노드 인덱스(섹션 트리 + 요약)** | 섹션별 summary_short, router_text로 **1단계: 구간 선택** → 2단계: 해당 구간 리프만 검색 | **장/절 수준 인덱스** 도입 시: 장(또는 장+절)별 요약/라우팅 텍스트를 두고, 쿼리와 매칭해 “관련 장/절”만 먼저 골라 그 하위 ARTICLE/CLAUSE/ITEM만 검색. 정확도·설명 가능성은 올라갈 수 있으나, 인덱스 구축·갱신과 쿼리 2단계 설계가 필요. |
| **빔 탐색(자식 확장)** | 시드 노드에서 자식만 확장하며 점수로 경로 결정 | 우리는 이미 semantic_group 등으로 “구간”을 거르고, BM25+Dense로 세부 청크를 찾음. “ARTICLE → CLAUSE/ITEM”을 빔처럼 명시적으로 확장하면 **어느 조문에서 내려갔는지** 설명에 활용 가능하지만, 현재 파이프라인에 단계가 하나 더 늘어남. |
| **백스톱** | 트리/빔 실패 시 전역 리프 히트로 폴백 | **필터+검색 결과가 너무 적을 때**(예: 청크 수 < N) **필터 완화** 또는 **2차 검색(필터 없이)** 하는 폴백을 두면, 페이징 인덱스의 백스톱과 같은 “실패 시 넓게라도 찾기” 역할을 할 수 있음. |

### 3.3 도입 우선순위 제안

1. **우선 검토 권장 (부담 적고 효과 기대)**  
   - **쿼리 확장(힌트 리젠)**  
     - 1차 검색 점수/결과가 약할 때, 상위 청크의 **semantic_group·regulation_article** 등에서 키워드를 뽑아 쿼리에 추가해 재검색.  
     - masterdoc의 “태그/엔티티로 쿼리 확장”과 동일한 발상.  
   - **백스톱에 해당하는 폴백**  
     - 1차 검색 결과 청크 수(또는 점수)가 임계치 미만이면, semantic_group 필터를 완화하거나 2차 검색으로 재시도.

2. **중기에서 검토**  
   - **장/절 단위 노드 인덱스**  
     - 장(또는 장+절)별 summary_short/router_text를 저장하고, “관련 장/절 선택 → 해당 구간 청크만 검색” 하는 2단계 검색.  
     - 페이징 인덱스의 “node_index로 구간 선택”을 규정 구조에 맞게 이식하는 형태.
   - **빔 스타일 확장**  
     - “상위 ARTICLE(또는 semantic_group) 점수 → 그 하위 CLAUSE/ITEM만 추가 조회”를 명시적으로 두어, 설명용 경로(어느 조문에서 내려갔는지)를 남기는 방식.

3. **입력이 PDF로 확장될 때만**  
   - 페이지 메타 추가 후, “페이지 기반 후보 좁히기”를 참고해 규정집용으로 재해석해 볼 만함.

## 4. 가정: 규정집이 PDF인 경우

규정집이 **TXT가 아니라 PDF**라고 가정하면, 페이징 인덱스와의 정합성이 크게 달라진다.

### 4.1 가능해지는 것

| 항목 | TXT 현재 | PDF 가정 시 |
|------|----------|-------------|
| **페이지 메타** | 없음 | PDF 추출 시 청크별 **page_start / page_end** 부여 가능 (또는 bbox). |
| **페이지 기반 후보** | 적용 불가 | 1차 검색으로 점수 높은 청크의 **페이지 집합**을 구한 뒤, 그 페이지(±인접 페이지)를 후보로 좁히는 흐름 적용 가능. |
| **노드 인덱스 + pages** | 장/절 요약만 가능 | 장·조문별로 **요약 + 해당 페이지 목록(pages)** 까지 저장 가능 → pageindex의 node_index와 동일한 역할. |
| **빔 시드** | semantic_group 등 구조만 | **후보 페이지에 등장하는 장/조문**을 시드로 쓰고, 그 아래로 빔 확장 가능. |
| **인용** | "제23조 ③" 수준 | **"제23조 ③ (p.5)"** 처럼 PDF 쪽 번호로 인용 가능 → 검증·감사 시 유리. |

### 4.2 청킹 전략 (PDF 가정)

- **권장**: PDF에서 텍스트 추출 시 **페이지 경계를 유지**한 채로 한 덩어리(또는 페이지별)로 만든 뒤, **기존 구조 파서**(제n장/제n조/①②/1./가.)를 그대로 돌려 **ARTICLE/CLAUSE/ITEM** 생성.  
  - 각 청크에 **page_start, page_end** (및 필요 시 bbox)만 추가.  
  - 즉, **청킹 단위는 지금처럼 구조 기반 유지**, 페이지는 **메타데이터**로만 붙인다.
- **대안**: masterdoc처럼 PDF 블록 단위로 고정 길이 청킹 + section_path 매핑도 가능하나, “제23조 ③” 같은 규정 단위가 깨질 수 있어 **규정집 목적에는 구조 기반이 더 적합**하다고 보는 것이 타당.

### 4.3 검색 흐름 (PDF 가정)

1. (선택) **노드 인덱스**: 장/조문별 summary_short, router_text, **pages** 저장.
2. **1차 검색**: 기존처럼 BM25+Dense 등으로 청크 검색 (또는 노드 인덱스로 관련 장/조문 선별).
3. **후보 페이지**: 점수 높은 청크들의 **페이지 집합** 계산 → 필요 시 인접 페이지 확장 (pageindex의 `page_expand`와 동일).
4. **빔/필터**: 후보 페이지에 등장하는 **ARTICLE(또는 장)** 을 시드로, 그 하위 CLAUSE/ITEM만 추가 수집. 또는 기존 semantic_group 필터에 **page IN (후보 페이지)** 조건 추가.
5. **백스톱**: 수집 청크/페이지가 부족하면 필터 완화 또는 전역 상위 청크로 폴백.
6. **인용**: 최종 채택 청크에 대해 **"제23조 ③ (p.5)"** 형식으로 표시.

### 4.4 PDF 가정 시 결론

- **페이지 기반 후보 좁히기**, **노드 인덱스에 pages 넣기**, **페이지 기반 인용**은 **그대로 도입 검토 가능**해진다.
- **청킹**은 페이지 단위로 자르기보다 **구조(ARTICLE/CLAUSE/ITEM) 유지 + 페이지 메타 추가**가 규정집에는 맞다.
- 정리하면, 규정집이 PDF라면 **“구조 기반 청킹 + 페이지 메타”**를 전제로, 페이징 인덱스의 **페이지 시드 → 빔/필터 → 백스톱** 흐름을 **상당 부분 참고·도입할 수 있는** 결과가 된다.

## 5. 결론

- **페이징 인덱스**는 **페이지가 있는 문서**를 전제로 한 “페이지 시드 + 섹션 빔 + 리프 검색 + 백스톱” 구조라, **현재 우리 규정집 TXT 청킹(ARTICLE/CLAUSE/ITEM)과 전제가 다름.**  
- **그대로 이식하기보다**,  
  - **쿼리 확장·백스톱(필터 완화/2차 검색)** 은 단기에서 도입 검토하고,  
  - **장/절 노드 인덱스·빔 스타일 확장**은 중기에서 “구조 기반 유지 + 설명 가능성·정확도 보강” 목적으로 선택 도입하는 편이 무난함.  
- **우리 쪽 구조 기반 청킹(조·항·호)** 은 유지하는 것이 좋고, 페이징 인덱스는 “검색 단계에서의 라우팅·폴백·쿼리 보강” 아이디어만 참고하는 수준이 적절함.
- **규정집이 PDF인 경우**는 §4 참고: 페이지 메타를 두면 페이지 기반 후보·노드 인덱스·인용까지 페이징 인덱스 기법을 상당 부분 도입 검토 가능.
