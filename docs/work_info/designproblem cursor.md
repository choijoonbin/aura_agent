AI 워크스페이스
# 화면 전반적으로 디자인 관점에서 검수 요청 드립니다. POC 로 보여줘야 할 우리 프로젝트의 메인(얼굴)임에도 불구하고 너무 부끄럽습니다. streamlit 에서 구현할수 있는 범위내에서
커스텀 컴포넌트들을 활용해서라도 멋지게 디자인하고 사용자 친화성 화면이어야 합니다.

## 케이스 워딩이 적당한지? 그리고 케이스 영역에 보여주는 전표 N건 들의 건별 내용들이 과연 지금과 같이 보여주는게 맞는지?
## 상세 / 선택 버튼이 전표 N건 마다 똑같이 계속 제공해주는게 맞는지, 그리고 이 버튼들의 동작도 애매합니다. 뭘 위한 버튼인건지 눌러보면 똑같은 내용을 보여줍니다.
## 우측 에이전트 대화에서 보여주는 내용이 좌측 전표에서 보여주는 내용과 중복입니다. 에이전트 대화는 SSE를 보여주는게 목적(기능)이라면 기능에 충실하는게 좋은지?
## 분석시작 버튼이 저렇게 길게 한 줄을 차지하는게 디자인 관점에서 맞는지?
## 스크리닝 완료 - .... 이 내용을 보여주는것도 한줄 차지하는게 맞는지와 해당 위치가 맞는지
## 분석 시작 버튼을 클릭하면 SSE 수신을 받는데 계속 추가되면서 사용자가 스크롤을 내리면서 수동으로 보는게 맞는지?
   예를 들면, chatgpt 도 질문을 받을면 추론하고 작업계획 세우는 등 동작하면서 발생하는 메시지를 보여주는데 길게 늘어지면서 밑에 붙여가며 보여주지 않는걸로 알고 있습니다.
## SSE 수신 받을때 텍스트가 타이핑하는 애니메이션 효과가 적용되었다고 하나 전혀 그렇게 보이지 않고 있습니다.
## 에이전트 대화 영역 하단의 4개 탭 영역이 해당 위치에 존재하는게 맞는지 의문입니다. 
## 하단의 4개 탭의 내용에 관한 중요한 확인 사항입니다. 아래 내용을 보면 4개 탭과 에이전트 대화 영역에 조회해서 보여주는 데이터입니다.
   모두 agent_activity_log 테이블 정보이며, 대략 보더라도 결국 중복되는 내용들이 많습니다. POC 참석자(대표, 사용자들) 앞에 보여줄 유용한 데이터인지 판단이 필요합니다.

# 워크스페이스 리팩토링 로드맵

협의·크로스 체크 완료 후 그대로 실행 가능한 수준의 작업계획서이다. Streamlit 한계 내에서 커스텀 CSS/컴포넌트를 활용해 POC 대표 화면으로서 품질을 높인다.

## 1. 목표 및 원칙

| 항목 | 내용 |
|------|------|
| **목표** | 워크스페이스를 POC 얼굴에 걸맞게 정리하고, 정보 중복을 줄이며, SSE·탭·버튼 UX를 개선한다. |
| **범위** | Streamlit + `st.markdown`/CSS/`st.session_state` 기반. 무리한 신규 프레임워크 도입 없음. |
| **원칙** | 한 화면에 같은 데이터를 여러 형태로 반복하지 않는다. 기능별 1차 목적을 명확히 한다. |

---

## 2. Phase 1: 정보 구조 및 데이터 전략

**목적**: agent_activity_log 기반 5개 영역의 역할을 나누고, “보여줄 것”을 정의해 중복을 제거한다.

| 작업 ID | 작업 내용 | 상세 | 완료 기준 |
|---------|-----------|------|-----------|
| P1-1 | **에이전트 대화 vs 4개 탭 역할 정의** | (1) **에이전트 대화**: “지금 이 run에서 무슨 일이 일어나고 있는가” — 실시간 SSE 스트림 전용. 완료 후에는 “마지막 N개 이벤트 요약”만 표시하거나, “결과 보기”로 유도. (2) **4개 탭**: “이 run의 결과를 어떻게 해석할 것인가” — 사고 과정·계획·실행 로그·최종 결과를 **run 완료(또는 HITL 대기) 후**에만 노출. | 역할 정의서 1페이지 확정, 코드 주석/README 반영 |
| P1-2 | **탭 데이터 통합 검토** | 사고 과정·작업 계획·실행 로그가 동일 timeline 파생이므로, “한 번 조회한 timeline → 3가지 뷰”로만 사용. 결과 탭은 RUN_COMPLETED payload만 사용. 불필요한 재조회·중복 렌더 제거. | timeline 1회 로드로 3탭 렌더, 결과 1회 로드 |
| P1-3 | **POC 관점 “유용한 데이터” 목록** | 대표·사용자 앞에서 강조할 지표: (예) 최종 판단·규정 근거·검증 요약·HITL 필요 여부·도구 성공률. “노드별 생각/관찰”은 심화 리뷰용으로만 유지. | 우선 노출 항목 목록 문서화 및 1차 반영 |

## 3. Phase 2: 좌측 케이스(전표) 영역

**목적**: 케이스 워딩 정리, 전표 N건 표시 방식, 상세/선택 버튼 명확화.

| 작업 ID | 작업 내용 | 상세 | 완료 기준 |
|---------|-----------|------|-----------|
| P2-1 | **워딩 정리** | “케이스” 유지 시: 부제 또는 툴팁으로 “전표 단위 분석 대상” 명시. 또는 상단 라벨을 “전표 목록” / “분석 대상 전표”로 변경해 전표 분석 맥락이 드러나도록 한다. | 기획 확정 워딩 적용, 일관되게 표기 |
| P2-2 | **건별 카드 내용** | 현재: 전표키·가맹점·금액·발생일시·상태·심각도·유형·근태·업종·예산 등 전부 노출. 개선: **리스트 뷰**에서는 전표키·가맹점·금액·발생일시·유형(또는 상태) 5개 이하로 요약. “더 보기” 확장 시에만 나머지 필드 표시. | 카드당 5개 이하 기본, 확장 시 전체 |
| P2-3 | **상세 / 선택 버튼** | (1) **상세**: “이 전표의 원본 정보 보기” — 클릭 시 다이얼로그/확장에서 전표키·가맹점·금액·발생일시·상태·심각도·유형·근태·업종·예산 등 전체 KV 표시. (2) **선택**: “이 전표를 우측에서 분석/대화할 대상으로 지정” — 클릭 시 `mt_selected_voucher` 설정 후 우측 패널 갱신. 버튼 라벨을 “상세 보기” / “선택하여 분석” 등으로 구체화. | 두 버튼 역할이 코드·라벨·동작으로 일치 |
| P2-4 | **버튼 배치** | 카드당 “상세 보기”는 아이콘 또는 작은 링크, “선택하여 분석”은 1차 액션으로 강조(예: primary). 한 줄에 두 개 나란히 배치해 공간 낭비를 줄인다. | 디자인 시안에 맞춰 적용 |

## 4. Phase 3: 우측 에이전트 대화 영역

**목적**: 좌측과의 중복 제거, SSE 전용 화면, 스크롤·타이핑 UX 개선.

| 작업 ID | 작업 내용 | 상세 | 완료 기준 |
|---------|-----------|------|-----------|
| P3-1 | **좌측과 중복 제거** | 우측 상단에 “전표키·가맹점·금액·발생일시·상태·심각도·유형·근태·업종·예산” 전체 KV를 다시 쓰지 않는다. **선택된 전표 1줄 요약**만 표시(예: “1000-H000000001-2026 · POC 심야 식대 1 · 97,788원 · 2026-03-07 22:28”). 클릭 시 좌측 카드로 포커스 또는 상세 다이얼로그 호출. | 우측에 KV 그리드 제거, 1줄 요약만 표시 |
| P3-2 | **분석 시작 버튼** | 전체 너비 한 줄 사용 중단. 우측 상단 요약 옆 또는 그 아래 **한 개 버튼**으로 배치(예: 아이콘+“분석 시작”). `use_container_width=True` 제거 또는 컬럼 내 작은 너비로 제한. | 버튼이 한 줄을 독점하지 않음 |
| P3-3 | **스크리닝 완료 배너** | “스크리닝 완료 - …” 한 줄 배너: (1) **위치**: 분석 시작 버튼 위 또는 선택 전표 요약 바로 아래. (2) **형태**: `st.success` 또는 커스텀 작은 배너. 여러 줄로 펼치지 않고 한 줄 + 필요 시 툴팁. | 한 줄 배너, 위치 확정 |
| P3-4 | **SSE 스트림 UX (스크롤)** | “아래로 계속 붙여 가며 수동 스크롤” 대신: (1) **자동 스크롤**: 새 메시지 추가 시 컨테이너를 맨 아래로 자동 스크롤(Streamlit에서는 `st.container` + 키 기반 리렌더, 또는 `st.chat_message` + 마지막 메시지로 스크롤 영역 하단 유지). (2) **접기/요약**: 완료된 run은 “최근 3개 이벤트 요약 + 전체 보기는 결과 탭에서” 형태로 옵션 제공. | 새 메시지 시 사용자가 수동으로 끝까지 스크롤하지 않아도 됨 |
| P3-5 | **타이핑 애니메이션** | 현재 `_stream_card_chunks` 등에서 단어 단위 yield하나 체감이 안 되는 경우: (1) **딜레이**: `time.sleep(0.02~0.05)` 등으로 시각적 타이핑 구간 확보. (2) **표시 단위**: 단어가 아닌 글자 단위로 yield해 더 촘촘히 갱신. (3) **대안**: “스트리밍 중…” 인디케이터 + 메시지 완성 시 한 번에 표시. | 실사용 시 타이핑 효과가 인지되거나, 대안으로 “스트리밍 중” 표시 명확 |

## 5. Phase 4: 하단 4개 탭 영역

**목적**: 탭 위치·구성 결정, 데이터 중복 제거, POC에서 보여줄 핵심만 강조.

| 작업 ID | 작업 내용 | 상세 | 완료 기준 |
|---------|-----------|------|-----------|
| P4-1 | **탭 위치** | 현재: 에이전트 대화 영역 하단(우측 패널 안). 검토 옵션: (A) **현행 유지**하되 “에이전트 대화”와 시각적으로 구분(구분선·다른 배경). (B) **페이지 상단 또는 별도 섹션**으로 이동해 “run 결과 리뷰” 블록으로 묶기. (C) **탭 제거**하고 “결과” 단일 뷰 + “상세(사고 과정·계획·실행 로그)”는 확장기(expander)로 제공. | 위치/구성 옵션 결정 및 반영 |
| P4-2 | **탭 구성 단순화(옵션)** | 4탭 → 2탭 통합 검토: (1) **“결과”**: 최종 판단·규정 근거·검증·점수·HITL. (2) **“실행 상세”**: 사고 과정 + 작업 계획 + 실행 로그를 하나의 스크롤 영역에 순서대로 표시(동일 timeline 1회 로드). | 협의 후 2탭 구조 적용 여부 결정 |
| P4-3 | **결과 탭 우선 노출** | POC 시연 시 “결과”가 먼저 보이도록 기본 선택 탭을 “결과”로 두거나, run 완료 시 자동으로 “결과” 탭으로 전환. | run 완료 후 결과 탭이 기본 노출 |
| P4-4 | **데이터 일원화** | timeline 1회 로드 후 사고 과정·작업 계획·실행 로그가 모두 이를 참조. 결과는 result payload 1회 로드. 중복 API/DB 쿼리 제거. | 동일 run에 대해 timeline/result 각 1회만 로드 |

## 8. 작업 순서 및 의존성

```
P1-1 → P1-2, P1-3     (역할 정의 후 데이터·노출 전략)
P2-1, P2-2, P2-3, P2-4  (케이스 영역, 상대적으로 독립)
P3-1 ~ P3-5            (에이전트 대화, P3-1은 P2와 병렬 가능)
P4-1 ← P1-1            (탭 위치/구성은 P1-1 결정에 의존)
P4-2, P4-3, P4-4       (P4-1 확정 후)
P5-1 ~ P5-4            (구조 변경 후 시각 정리, 마지막에 권장)
```

**권장 스프린트**  
- **1차**: P1-1, P2-1, P2-3, P3-1, P3-2, P3-3 (정의·중복 제거·버튼/배너)  
- **2차**: P4-1, P4-3, P4-4, P1-2 (탭 위치·결과 우선·데이터 일원화)  
- **3차**: P3-4, P3-5, P2-2, P2-4 (SSE UX·타이핑·카드 요약·버튼 배치)  
- **4차**: P4-2(옵션), P5-1~P5-4 (탭 통합 검토·시각 정리)

## 9. 수용 기준 요약

- **에이전트 대화**: 실시간 SSE에 충실하고, 좌측 전표 KV 전체 중복 제거.  
- **4개 탭**: 동일 timeline/result 1회 로드, POC에서 “결과” 강조, 위치/구성 확정.  
- **좌측 케이스**: 카드 요약·상세/선택 버튼 역할 명확·워딩 정리.  
- **전체**: 자동 스크롤 또는 스트리밍 표시 개선, 타이핑 효과 또는 대안 명확화, 시각 일관성.

위 로드맵을 크로스 체크·협의 후 그대로 작업에 반영할 수 있도록 작성하였다. 수정이 필요하면 Phase/작업 ID를 기준으로 이슈·태스크로 관리하면 된다.
