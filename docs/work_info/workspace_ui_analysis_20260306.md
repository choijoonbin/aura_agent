### 1.4 메인 중앙 — Agent Flow 및 결과

| 단계(스크린샷) | 현재 코드 노드/표시 |
|----------------|----------------------|
| Agent Flow | 공통 |
| 이슈 탐지 | **없음**. 전표 플로우: **스크리닝** → 정보 수집 → 계획 수립 → 도구 실행 → 비판 검토 → 검증 및 HITL 판단 → (HITL 대기) → 보고서 생성 → 최종 판정 |
| 이슈 심층 분석 | planner / execute / critic 에 해당 가능 |
| 이슈 해결 계획 수립 | planner (계획 수립) |
| 해결책 코드 생성 / 수정 | **없음**. 전표 감사는 코드 생성 없음 |
| 해결책 PR 생성 | **없음** |
| Pull Request 제목/내용 | **없음**. HITL 다이얼로그는 **담당자 검토**(사업 목적, 참석자, 의견, 승인/보류) |
| 최종 해결책 확인 + "확인" 버튼 | 전표: **최종 판정** 결과 + (HITL 시) "HITL 검토 입력 열기" → `render_hitl_dialog` (승인/보류 등) |

**정리**  
- 스크린샷의 "이슈 탐지 → PR 생성 → 최종 해결책 확인"은 **코드/PR 중심 플로우**로 보이며,  
- 현재 AuraAgent **전표 워크스페이스**는 **스크리닝 → 검증 → HITL(담당자 검토) → 보고서/최종 판정** 구조임.  
- 따라서 **동일 화면이면** "Agent Flow - Pull Request 제목/내용" 팝업은 **다른 제품/모드**의 HITL 또는 확인 단계에 대응하는 UI로 추정됨.

### 1.5 우측 — HITL(확인) 팝업

| 항목 | 내용 |
|------|------|
| **제목** | "Agent Flow - Pull Request 제목/내용" (또는 유사) |
| **내용** | 큰 텍스트 영역에 생성된 PR 제목/내용(한글) 표시 |
| **버튼** | "이전", "완료" |
| **역할** | 사용자 확인 후 다음 단계로 진행 — HITL(인터럽트) 구간으로 해석 가능 |

**코드 대응 (전표 HITL)**  
- `render_hitl_dialog`: 검토 사유, 중단 사유, 질문, 증거 스냅샷 + **담당자/사업목적/참석자/의견/승인·보류** 입력 폼.  
- 제출 시 `POST /api/v1/analysis-runs/{run_id}/hitl` 호출 후 같은 run으로 재개.  
- "이전/완료"가 아니라 **저장(드래프트)** 및 **제출 후 재개** 버튼 구조.

## 2. 현재 문제점 도출

### 2.1 HITL 확인 체크박스 vs 팝업 동작 (이전 이슈와 연계)

- **증상**  
  - "HITL 확인"을 **체크 해제**한 뒤 분석을 돌렸는데도 **HITL 팝업이 뜸**.  
- **가능 원인(이미 수정된 부분)**  
  1. **verify_node**에서 `_enable_hitl=False`여도 **HITL_REQUESTED 이벤트**를 스트림에 보내 타임라인에 남고,  
  2. UI의 `_has_pending_hitl()`이 타임라인 내 `HITL_REQUESTED` 존재만으로 팝업을 띄움.  
- **적용한 수정**  
  - `_enable_hitl=False`일 때 verify에서 `hitl_request=None`으로 두어 이벤트/상태에 HITL을 남기지 않음.  
  - 체크 해제 시 팝업이 더 이상 뜨지 않아야 함.  
- **추가로 확인할 점**  
  - 스크린샷이 **체크된 상태**에서 찍힌 것이라면, 현재는 “의도된 동작”임.  
  - **체크 해제 후에도** 팝업이 뜬다면, **요청 body의 `enable_hitl` 전달**(프론트→API→에이전트)과 **다른 run/캐시** 영향 여부를 점검 필요.

---

### 2.2 체크박스 기본값과 시각적 피드백

- **코드**  
  - "HITL 확인" 기본값은 **`value=False`**(체크 해제).  
- **문제**  
  - 사용자가 “한 번도 건드리지 않았다”고 인지해도, **rerun/세션**에 따라 체크 상태가 달라 보일 수 있음.  
  - “체크 해제 = HITL 없이 끝까지 분석”이라는 의미가 **라벨/툴팁 없이**만 있어, 첫 사용 시 헷갈릴 수 있음.  
- **개선 제안**  
  - 툴팁 또는 라벨 옆 짧은 설명 추가 (예: "체크 시 분석 중 담당자 검토 팝업 표시").  
  - 필요 시 체크박스 옆에 "(기본: 해제)" 등으로 기본 동작 명시.

---

### 2.3 UI 불일치 가능성 (스크린샷 vs 전표 워크스페이스)

- **스크린샷**  
  - 분석 대상 **경로** 입력, **파일 선택**, **이슈 탐지 → PR 생성 → 최종 해결책 확인**, "Pull Request 제목/내용" 팝업.  
- **현재 코드**  
  - **전표 키** 기준 케이스 선택, **경로/파일 입력 없음**, **스크리닝→검증→HITL(담당자 검토)→보고서/최종 판정**, PR/코드 생성 없음.  
- **도출**  
  - 같은 레포라도 **다른 화면(다른 앱/빌드/모드)**이 캡처되었을 가능성이 있음.  
  - 그 경우 "분석 대상 경로", "파일 선택", "분석 취소", "분석 대기/완료 (N)" 등은 **현재 전표 워크스페이스 코드에 없으므로**,  
    - 해당 UI가 **다른 페이지/프로젝트** 것인지,  
    - 또는 **추가 예정/다른 브랜치** 것인지 구분이 필요함.

---

### 2.4 팝업(모달) UX

- **스크린샷**  
  - 우측에 큰 모달, "이전"/"완료" 버튼.  
- **문제 가능성**  
  - 모달이 **메인 플로우를 가리면** 스트림/타임라인을 보며 검토하기 어려움.  
  - "완료"가 **제출인지, 단순 닫기인지** 문구만으로는 다소 모호할 수 있음.  
- **전표 HITL 쪽**  
  - `render_hitl_dialog`에서 **제출 후 재개** 의미가 버튼/메시지에 드러나도록 하면, “완료”와 “재개” 구분이 명확해짐.

---

### 2.5 정보 밀도 및 다음 액션

- **스크린샷**  
  - 상단에 경로, 체크박스, 버튼, 파일 선택, 상태 카드가 한 화면에 배치.  
- **문제 가능성**  
  - 요소가 많을 경우 **실제 다음 액션**(분석 시작 vs 검토 입력 vs 재개)이 눈에 잘 안 들어올 수 있음.  
- **현재 코드**  
  - "다음 액션: 분석 시작 또는 검토 재개" 문구와 실행 상태 메타(현재 상태, 심각도, run_id)로 보완하고 있음.  
  - HITL 대기 시 "이 분석은 담당자 검토가 필요합니다" 경고 + "HITL 검토 입력 열기" 버튼으로 구분 가능.  
- **개선**  
  - 상태에 따라 **한 가지 CTA**를 강조(예: HITL 대기 시 “검토 입력”만 강조)하면 혼동이 줄어듦.

## 4. 다음 액션 제안

1. **HITL 체크 해제** 후 동일 시나리오로 한 건 더 분석해, 팝업이 뜨지 않는지 재현 테스트.  
2. **스크린샷**이 현재 AuraAgent `main`(전표 워크스페이스)인지, 다른 앱/모드인지 확인 후, 경로·파일·PR UI가 필요하면 별도 스펙/이슈로 정리.  
3. "HITL 확인" 체크박스에 툴팁 또는 짧은 설명 추가 반영.  
4. HITL 다이얼로그 버튼 문구를 "제출 후 재개" 등으로 명확화 검토.
