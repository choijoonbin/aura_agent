#워크스페이스 리팩토링 로드맵

## 1. 목표 재정의
AI 워크스페이스는 단순 조회 화면이 아니라, "에이전트가 실제로 어떻게 판단하고 어떤 근거로 결론에 도달하는지"를 한눈에 보여주는 메인 데모 화면으로 재구성해야 한다.

이 메뉴의 목적은 아래 4가지로 고정한다.
1. 사용자가 분석 대상 전표를 빠르게 선택할 수 있어야 한다.
2. 선택한 전표에 대해 에이전트의 현재 진행 상태를 실시간으로 이해할 수 있어야 한다.
3. 분석 완료 후 결과, 근거, 사람 검토 필요 사유를 구조적으로 확인할 수 있어야 한다.
4. 발표/시연 상황에서 "복잡한 내부 로그"가 아니라 "의미 있는 의사결정 흐름"이 보이도록 해야 한다.

## 2. 현재 화면의 핵심 문제
### 2.1 정보 구조 문제
- 좌측 전표 카드와 우측 상단 요약 정보가 사실상 같은 데이터를 반복 노출한다.
- 에이전트 대화와 하단 4개 탭이 모두 `agent_activity_log` 기반이라, 사용자가 보기에 같은 내용을 다른 이름으로 반복하는 구조다.
- 사고 과정, 작업 계획, 실행 로그, 결과의 경계가 약해 각 탭의 존재 이유가 불명확하다.

### 2.2 인터랙션 문제
- `상세`, `선택` 버튼이 동시에 존재하지만 동작 의미가 분리되지 않았다.
- `분석 시작` 버튼이 과도하게 넓고, CTA 위계가 맞지 않는다.
- HITL 발생 시 실시간 흐름과 사람 개입 흐름이 자연스럽게 이어지지 않는다.
- SSE 수신 시 사용자가 계속 아래로 스크롤해야 하므로 "실시간 대화"처럼 느껴지지 않는다.

### 2.3 시각 디자인 문제
- 전표 카드, 상태 배지, CTA, 탭, 스트림 패널이 같은 디자인 시스템을 공유하지 않는다.
- 여백, 카드 높이, 패널 폭, 타이포 스케일이 일관되지 않는다.
- 메인 화면인데도 시선이 머무는 중심 영역이 없다.

### 2.4 데모 메시지 문제
- 현재 화면은 "실행 로그를 많이 보여준다"에 가깝고, "에이전트가 판단한다"는 느낌은 약하다.
- 발표 대상자는 원시 로그가 아니라, 판단 근거와 진행 맥락, 개입 포인트를 보고 싶어 한다.

## 3. 리팩토링 원칙
### 3.1 원칙
- 중복 데이터는 제거한다.
- 한 영역은 하나의 목적만 가진다.
- 실시간 영역과 사후 리뷰 영역을 명확히 분리한다.
- 데모용 화면이므로 "설명 가능한 시각 구조"를 우선한다.
- 내부 컬럼명/로그명보다 사용자 이해 가능한 용어를 우선한다.

### 3.2 금지사항
- 같은 데이터를 카드/요약/탭/스트림에 반복 노출하지 않는다.
- 버튼 의미가 겹치는 CTA를 두지 않는다.
- `agent_activity_log`를 원문 중심으로 그대로 나열하지 않는다.
- 화면 확장으로 해결하지 않고, 정보 구조 재설계로 해결한다.

## 4. 목표 구조
워크스페이스는 아래 3단 구조로 재편한다.

### 4.1 좌측: 분석 대상 선택 영역
이 영역은 "무엇을 분석할지 선택"하는 역할만 수행한다.

구성:
- 상단 필터: `전체`, `검토 필요`, `완료`, `HITL 대기`
- 전표 목록: 카드형 리스트
- 각 카드에 표시할 정보:
  - 전표명 또는 시연명
  - 금액
  - 발생일시
  - 위험유형 배지
  - 심각도 배지
  - 상태 배지
- 카드 액션:
  - `선택`만 유지
  - `상세` 버튼 제거
  - 카드 전체 클릭도 선택으로 통일 가능 여부 검토

전표 카드 설계 원칙:
- 텍스트 정보는 2줄 이내 요약
- 메타 정보는 하단 보조행으로 정리
- 선택된 카드만 보더/배경/그림자로 강조
- 길이가 길어질 경우 줄바꿈보다 잘라서 보조 툴팁 또는 팝업으로 처리

### 4.2 중앙 메인: 실시간 분석 영역
이 영역은 "현재 에이전트가 무엇을 하고 있는지"를 보여주는 핵심 무대다.

구성:
- 상단: 선택된 전표의 compact summary
  - 전표키, 금액, 발생시각, 가맹점, 위험유형, 상태 정도만 compact card로 표시
  - 현재처럼 상세 key-value 표 전체를 길게 반복하지 않음
- 상단 우측 CTA:
  - `분석 시작`
  - HITL 발생 시 `검토 재개`
- 중단 배너:
  - 스크리닝 완료 문구는 배너가 아니라 summary badge 또는 info strip로 축소
- 메인 스트림 패널:
  - 실시간 SSE/에이전트 스트림 전용
  - 자동 스크롤
  - 최근 메시지를 중심으로 보여주고 이전 메시지는 접을 수 있게 설계
  - 메시지 유형별 스타일 구분
    - 생각
    - 행동
    - 도구 호출
    - 도구 결과
    - 검증/게이트
    - HITL 요청
    - 최종 결론

실시간 스트림 UX 원칙:
- "타임라인 누적 로그"가 아니라 "현재 진행 중인 작업 콘솔"처럼 보이게 한다.
- 최신 메시지가 항상 하단 고정되도록 자동 스크롤한다.
- 중복 문장 제거, 이벤트 병합, 중요 메시지 강조를 적용한다.
- 실행 중에는 skeleton 또는 typing indicator를 명확히 표시한다.

### 4.3 하단 또는 우측 보조영역: 사후 리뷰 탭
이 영역은 분석 완료 후 검토용 정보만 보여준다.

권장 탭 재구성:
1. `판단 요약`
   - 최종 상태
   - 점수
   - 심각도
   - 최종 판단 문장
   - HITL 반영 여부
2. `근거 맵`
   - 규정 근거
   - citation 목록
   - 어떤 문장이 어떤 근거를 채택했는지
3. `실행 내역`
   - 도구 호출 내역
   - 실행 성공/실패
   - 핵심 스킬 결과 요약
4. `검토 이력`
   - HITL 요청 사유
   - 사람 검토 응답
   - resume 이후 상태 변화

제거 또는 통합 검토 대상:
- `사고 과정`과 `에이전트 대화`는 중복도가 높다.
- 권장안은:
  - 에이전트 대화 = 실시간 전용
  - 사고 과정 = 제거 또는 `판단 요약` 내부의 압축형 review로 통합
- `작업 계획`은 planner 결과가 명확할 때만 독립 유지
- `실행 로그`는 tool trace 중심으로 정리

## 5. 세부 작업 계획
### Phase 1. 정보 구조 재설계
목표: 중복 제거와 영역 역할 재정의

작업:
- 현재 워크스페이스 내 각 컴포넌트가 어떤 데이터 소스를 참조하는지 표로 정리
- `agent_activity_log`, `latest_bundle.result`, `timeline`, `history` 간 중복 필드 식별
- 좌측 카드, 우측 summary, 하단 탭 간 중복 필드 제거 설계
- 각 탭의 목적과 데이터 소스를 1:1로 확정

완료 기준:
- 화면 각 영역의 목적이 1문장으로 설명 가능해야 함
- 동일 데이터가 2개 이상 영역에 반복되지 않아야 함

### Phase 2. 케이스 선택 UX 리팩토링
목표: 전표 선택이 빠르고 직관적이어야 함

작업:
- 전표 카드 컴포넌트 분리
- `상세` 버튼 제거
- 카드 전체 클릭 선택 또는 단일 CTA로 통일
- 상태/심각도/유형 배지를 디자인 시스템 기준으로 통일
- 목록 영역 고정 높이 + 내부 스크롤 유지
- 선택 상태, hover 상태, focus 상태 명확화

완료 기준:
- 카드 한 장만 봐도 전표 성격 파악 가능
- 클릭 동작 의미가 하나로 명확함

### Phase 3. 실시간 분석 패널 리디자인
목표: 에이전트가 살아 움직이는 느낌을 주는 중심 영역 구현

작업:
- 스트림 전용 패널 레이아웃 재설계
- 최신 이벤트 auto-scroll 적용
- 이벤트 타입별 컴포넌트 스타일 분리
- 중요 이벤트는 카드형 강조
  - `TOOL_CALL`
  - `TOOL_RESULT`
  - `HITL_REQUESTED`
  - `HITL_PAUSE`
  - `RESULT_FINALIZED`
- typing indicator / 진행 중 상태 컴포넌트 명확화
- 실시간 패널과 완료 후 리뷰 탭의 시각적 구분 강화

완료 기준:
- 분석 중 사용자가 스크롤을 거의 수동으로 내리지 않아도 핵심 흐름을 볼 수 있어야 함
- 첫 화면 5초 안에 "AI가 지금 무엇을 하는지" 이해 가능해야 함

### Phase 4. 탭 구조 개편
목표: 분석 완료 후 정보 탐색성을 높임

작업:
- 기존 4탭의 역할 재검토 후 재명명
- `사고 과정` 제거 또는 review summary로 축소
- `작업 계획`은 planner 결과가 존재할 때만 노출
- `실행 로그`는 raw 로그가 아니라 tool trace summary 형태로 구성
- `결과` 탭은 최종 판단 중심으로 재구성
- `근거 맵`은 citation 중심으로 시각화 강화

완료 기준:
- 각 탭에 들어가는 이유가 명확해야 함
- 탭 이름만 보고 무엇을 볼 수 있는지 예측 가능해야 함

### Phase 5. HITL 흐름 시각 통합
목표: 사람 개입이 UX 상 자연스럽게 이어지도록 함

작업:
- 스트림에서 HITL 발생 시 즉시 팝업 연결 유지
- HITL 발생 이유를 LLM/verifier payload 기반으로 노출
- HITL 제출 후 재개 스트림을 현재 패널에서 이어서 표시
- HITL 이력은 `검토 이력` 탭으로 누적 관리
- popup과 메인 스트림 사이의 상태 동기화 정리

완료 기준:
- 사용자가 "왜 멈췄는지"와 "어떤 입력이 필요한지"를 즉시 이해할 수 있어야 함
- 제출 후 재개 흐름이 끊기지 않아야 함

## 6. 권장 레이아웃안

- 에이전트 중심 시연에 유리
n

## 9. 최종 산출물 기준
리팩토링 완료 후 워크스페이스는 아래 상태여야 한다.
- 발표자가 "좌측에서 전표를 고르고, 가운데에서 에이전트가 실시간으로 판단하는 모습을 보여주고, 하단에서 근거와 최종 판단을 설명하는" 흐름으로 바로 시연 가능해야 한다.
- 한 화면 안에서 중복 로그를 반복해서 설명하지 않아도 되어야 한다.
- Streamlit 기반이더라도 제품형 데모 수준의 완성도를 가져야 한다.
- 이후 에이전트 스튜디오, 규정문서 라이브러리, 운영 대시보드에도 같은 디자인 시스템을 확장 적용할 수 있어야 한다.
