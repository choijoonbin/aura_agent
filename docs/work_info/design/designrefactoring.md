# 워크스페이스 리팩토링 최종안

이 문서는 `AI 워크스페이스` 메뉴를 POC 대표 화면 수준으로 끌어올리기 위한 최종 리팩토링 기준 문서다.  
`designproblem.md`의 문제 정의, `designproblem cursor.md`의 실행형 로드맵을 교차 검토해 통합했다.  
합의 후에는 이 문서를 기준으로 바로 구현 작업에 착수할 수 있어야 한다.

## 1. 리팩토링 목표

AI 워크스페이스는 단순 조회 화면이 아니라 아래 4가지를 동시에 보여주는 메인 데모 화면이어야 한다.

1. 어떤 전표를 분석 대상으로 선택했는지
2. 에이전트가 지금 무엇을 하고 있는지
3. 어떤 근거와 검증 과정을 거쳐 판단했는지
4. 언제 사람 검토(HITL)가 개입했고 이후 어떻게 재개되었는지

즉, 이 화면은 “전표 목록 + 로그 뷰어”가 아니라 “선택 → 실행 → 판단 → 검토”의 흐름을 설명하는 무대여야 한다.

## 2. 현재 화면의 핵심 문제

### 2.1 정보 중복
- 좌측 케이스 카드와 우측 상단 요약이 사실상 같은 전표 정보를 반복한다.
- 에이전트 대화와 하단 탭들이 모두 `agent_activity_log` 기반이라, 사용자 입장에서는 같은 데이터를 다른 이름으로 반복 노출하는 구조다.
- `사고 과정`, `작업 계획`, `실행 로그`, `결과`의 역할 경계가 약하다.

### 2.2 인터랙션 불명확
- `상세`, `선택` 버튼의 의미가 겹친다.
- `분석 시작` 버튼은 CTA 위계 대비 과도하게 넓고 무겁다.
- SSE는 실시간 스트림인데, 현재는 누적 로그를 수동 스크롤로 보는 방식이라 대화형 UX가 약하다.
- HITL은 발생과 재개 흐름이 존재하지만, 메인 화면에서 그 흐름이 자연스럽게 이어지지 않는다.

### 2.3 시각 완성도 부족
- 카드, 배지, 탭, 버튼, 스트림, 빈 상태가 하나의 디자인 시스템으로 묶여 있지 않다.
- 여백, 패널 높이, 카드 밀도, 타이포 스케일이 불균형하다.
- 사용자의 시선이 어디에 머물러야 하는지 중심축이 약하다.

### 2.4 POC 메시지 전달력 부족
- 현재는 “많은 로그를 보여준다”는 느낌이 강하고, “에이전트가 실제로 판단한다”는 인상이 약하다.
- 발표 상황에서는 원시 로그보다 “판단 흐름, 개입 포인트, 근거 구조”가 더 중요하다.

## 3. 리팩토링 원칙

### 3.1 설계 원칙
- 같은 데이터는 한 화면에서 한 번만 의미 있게 보여준다.
- 각 영역은 하나의 목적만 가진다.
- 실시간 영역과 사후 리뷰 영역을 분리한다.
- 내부 로그명이 아니라 사용자가 이해할 수 있는 용어를 우선한다.
- Streamlit 기반이라도 제품형 데모 수준의 완성도를 목표로 한다.

### 3.2 금지사항
- `agent_activity_log`를 원문 형태로 그대로 나열하지 않는다.
- 같은 필드를 카드, 요약, 탭, 스트림에 중복 배치하지 않는다.
- 의미가 겹치는 CTA를 동시에 두지 않는다.
- 화면을 키워서 해결하지 않고 정보 구조를 재설계해서 해결한다.

## 4. 목표 화면 구조

워크스페이스는 아래 3영역 구조를 기준으로 재편한다.

### 4.1 좌측: 분석 대상 선택 영역

이 영역의 역할은 오직 “무엇을 분석할지 선택”하는 것이다.

구성:
- 상단 필터: `전체`, `검토 필요`, `완료`, `HITL 대기`
- 전표 목록: 카드형 리스트
- 카드 표시 항목:
  - 전표명 또는 시연명
  - 금액
  - 발생일시
  - 위험유형 배지
  - 심각도 배지
  - 상태 배지
- 카드 액션:
  - `선택`만 1차 액션으로 유지
  - `상세`는 제거하거나, 반드시 별도 의미를 갖는 작은 보조 액션으로 축소

전표 카드 설계 기준:
- 기본 정보는 2줄 이내 요약
- 세부 메타는 하단 보조행으로 배치
- 선택된 카드만 보더/배경/그림자로 강조
- 카드 전체 클릭을 선택으로 통일하는 방안 우선 검토

좌측 카드와 우측 compact summary의 공통 필드 기준:
- 공통 핵심 필드: 금액, 발생일시, 위험유형, 상태
- 좌측 카드 우선 필드: 전표명 또는 시연명, 심각도
- 우측 summary 우선 필드: 전표키, 가맹점
- 원칙: 동일 필드를 좌/우에 모두 길게 반복 노출하지 않는다.

### 4.2 우측 상단: 실시간 분석 영역

이 영역은 “에이전트가 지금 무엇을 하고 있는지”를 보여주는 핵심 무대다.

구성:
- 선택된 전표 compact summary
  - 전표키, 금액, 발생시각, 가맹점, 위험유형, 상태 정도만 compact 노출
  - 현재처럼 전체 key-value 표를 반복하지 않음
- CTA 영역
  - `분석 시작`
  - HITL 중이면 `검토 재개` 또는 `HITL 검토 입력 열기`
- 상태 strip
  - 스크리닝 완료, 검토 필요, 재개 대기 등은 작은 상태 strip 또는 badge row로 축약
- 실시간 스트림 패널
  - SSE/에이전트 스트림 전용
  - 자동 스크롤
  - 최신 메시지 중심
  - 중요 이벤트 강조

스트림 메시지 유형:
- 생각
- 행동
- 도구 호출
- 도구 결과
- 검증/게이트
- HITL 요청
- 최종 결론

UX 기준:
- “누적 로그”보다 “현재 작업 콘솔”처럼 보여야 한다.
- 사용자가 수동으로 계속 스크롤하지 않아도 최신 흐름을 볼 수 있어야 한다.
- 이벤트 병합, 중복 제거, 상태 변화 강조가 필요하다.
- 타이핑 효과가 애매하면 과감히 버리고 `진행 중 인디케이터 + 단계별 카드 갱신` 방식으로 전환한다.
- Streamlit 제약상 완전한 브라우저형 auto-scroll이 불안정하면, 구현 목표를 `최신 이벤트가 항상 보이도록 유지`로 조정한다.
- 구현 우선순위는 `최신 N개 유지` → `새 이벤트 시 최신 블록 강조` → `필요 시 최소 JS 보조` 순으로 둔다.

### 4.3 우측 하단: 사후 리뷰 영역

이 영역은 run 완료 또는 HITL 대기 후 “리뷰용”으로만 사용한다.

권장 탭 구조:
1. `판단 요약`
   - 최종 상태
   - 점수
   - 심각도
   - 최종 판단 문장
   - HITL 반영 여부
2. `근거 맵`
   - 규정 근거
   - citation 목록
   - 문장-근거 연결
3. `실행 내역`
   - TOOL_CALL / TOOL_RESULT / 실행 성공 여부
   - 핵심 스킬 결과 요약
4. `검토 이력`
   - HITL 요청
   - 사람 검토 응답
   - resume 이후 상태 변화

통합/제거 검토:
- `사고 과정`은 `에이전트 대화`와 중복되므로 제거 또는 `판단 요약` 내부의 압축형 review로 흡수
- `작업 계획`은 planner 결과가 실질적으로 설명 가치가 있을 때만 유지
- `실행 로그`는 raw log가 아니라 tool trace summary 중심으로 정리

기존 탭과 신규 탭의 권장 매핑:

| 기존 탭/함수 관점 | 신규 권장 명칭 | 목적 |
|---|---|---|
| 사고 과정 | 판단 요약 내부 review 또는 제거 | 사람이 설명 가능한 핵심 판단 흐름만 요약 |
| 작업 계획 | 작업 계획 또는 실행 내역 내부 섹션 | planner 결과가 존재할 때만 노출 |
| 실행 로그 | 실행 내역 | TOOL_CALL / TOOL_RESULT / 게이트 요약 |
| 결과 | 판단 요약 | 최종 상태, 점수, 심각도, 이유 |
| HITL 관련 별도 표시 | 검토 이력 | 요청, 응답, 재개 이후 상태 변화 |

## 5. 커서 안에서 반영할 가치가 큰 장점

커서 문서에서 유지할 가치가 있는 부분은 아래 4가지다.

1. 작업을 `Phase + 작업 ID`로 나눠 실행 단위와 완료 기준을 분리한 점
2. 각 UI 문제를 “데이터 원천과 렌더 위치” 기준으로 나눈 점
3. 좌측 케이스 영역과 우측 에이전트 영역을 별도 단계로 쪼개 구현 순서를 명확히 한 점
4. 탭 구조를 유지/통합/제거 옵션으로 분기 검토한 점

따라서 최종안은 커서의 “실행형 표 구조”를 살리고, Codex 안의 “정보 구조/UX 원칙”을 상위 기준으로 둔다.

## 6. 세부 작업 계획

### Phase 1. 정보 구조 재설계
목표: 중복 제거와 영역 역할 확정

작업:
- 워크스페이스 각 컴포넌트의 데이터 소스 표 작성
- `agent_activity_log`, `latest_bundle.result`, `timeline`, `history` 간 중복 식별
- 좌측 카드, 우측 summary, 하단 리뷰 탭 간 중복 필드 제거
- 각 탭의 목적과 데이터 소스를 1:1로 확정

완료 기준:
- 화면 각 영역의 목적이 1문장으로 설명 가능
- 같은 데이터가 서로 다른 컴포넌트에 반복되지 않음

### Phase 2. 좌측 케이스 영역 리팩토링
목표: 전표 선택 UX를 단순하고 직관적으로 개선

작업:
- `케이스` 워딩 유지 여부 검토
  - 유지 시 부제로 “전표 단위 분석 대상” 명시
  - 대안: `전표 목록`, `분석 대상 전표`
- 카드 정보 축소
  - 기본은 전표명, 금액, 발생일시, 유형, 상태 정도만 노출
  - 나머지는 hover/툴팁/확장 또는 별도 preview로 이동
- `상세` 버튼 제거 또는 작은 보조 액션으로 축소
- 카드 전체 클릭 또는 `선택하여 분석` 단일 CTA로 통일
- 목록 영역 고정 높이 + 내부 스크롤 유지

완료 기준:
- 카드 한 장만 보고도 전표 성격을 이해할 수 있음
- 클릭 동작 의미가 하나로 명확함

### Phase 3. 실시간 분석 패널 리디자인
목표: 에이전트가 살아 움직이는 느낌을 주는 중심 영역 구현

작업:
- 우측 상단 key-value 반복 정보 제거
- compact summary header 도입
- `분석 시작` 버튼 크기와 위치 재설계
- 스크리닝 완료 문구를 줄이고 strip/badge 형태로 정리
- 스트림 패널을 “현재 진행 중인 작업 콘솔”처럼 재구성
- 자동 스크롤 적용
- 메시지 유형별 시각 스타일 분리
- 중요 이벤트만 카드형 강조
- 빈 상태를 3단계로 정의
  - 선택 전: `전표를 선택하면 이 영역에서 에이전트 실행을 시작할 수 있습니다.`
  - 분석 전: `분석을 시작하면 실시간 스트림이 여기에 표시됩니다.`
  - 완료 후: `실시간 실행은 종료되었습니다. 상세 리뷰는 아래 탭에서 확인하세요.`

완료 기준:
- 첫 화면 5초 내에 “AI가 지금 무엇을 하는지” 이해 가능
- 분석 중 수동 스크롤 부담이 크게 줄어듦

### Phase 4. 리뷰 탭 구조 개편
목표: 분석 완료 후 정보 탐색성을 높임

작업:
- 기존 4탭 재명명 또는 재구성
- `사고 과정` 제거/통합 여부 확정
- `작업 계획`은 planner output 존재 시만 노출
- `실행 로그`는 raw 로그가 아니라 요약형 trace로 구성
- `결과` 탭을 최종 판단 중심으로 재구성
- `근거 맵`은 citation 중심 시각화 강화

완료 기준:
- 각 탭의 존재 이유를 바로 설명 가능
- 사용자가 탭 이름만 보고 내용 예측 가능

### Phase 5. HITL 흐름 시각 통합
목표: 사람 개입과 재개 흐름을 자연스럽게 연결

작업:
- 스트림에서 HITL 발생 시 즉시 팝업 연동
- HITL 발생 이유는 verifier/LLM payload 기반으로 표시
- HITL 제출 후 재개 스트림을 현재 패널에 이어서 표시
- HITL 이력은 `검토 이력` 탭에 누적
- popup과 메인 스트림 상태 동기화 정리

구현 기준:
- SSE에서 `HITL_REQUESTED` 또는 `HITL_PAUSE`를 수신하면 아래 순서를 따른다.
  1. 이벤트 수신
  2. 해당 run의 `session_state`에 HITL open 플래그 기록
  3. 스트림 블록이 제어를 반환할 수 있는 지점까지 종료
  4. 다음 렌더에서 dialog/popup 표시
- 워크스페이스 최초 진입 시 자동 팝업은 금지한다.
- 스트림 중 발생한 HITL 또는 사용자가 수동으로 다시 여는 HITL만 허용한다.

완료 기준:
- 사용자가 왜 멈췄는지 즉시 이해 가능
- 제출 후 재개 흐름이 끊기지 않음

### Phase 6. 디자인 시스템 정비
목표: 화면 전반의 엔터프라이즈급 일관성 확보

작업:
- 색상 토큰 정리
  - primary
  - success
  - warning
  - danger
  - neutral
- 타이포 스케일 정리
  - page title
  - section title
  - body
  - meta
- 공통 스타일 정리
  - card
  - badge
  - tab
  - button
  - panel
  - empty state
- Streamlit custom CSS를 영역 단위로 재편
- 가능하면 `ui/shared.py` 또는 workspace 전용 style 모듈로 구조화

완료 기준:
- 같은 의미의 상태는 항상 같은 색과 형태 사용
- 카드, 버튼, 탭, 배지가 한 제품 안의 컴포넌트처럼 보임

## 7. 레이아웃 권장안

### 안 A. 좌측 목록 / 우측 메인 분석형
권장 1순위

구성:
- 좌 32%: 케이스 목록
- 우 68%: 선택 전표 요약 + 실시간 스트림 + 리뷰 탭

장점:
- 현재 구조를 크게 깨지 않으면서 개선 가능
- 발표 시 설명 순서가 자연스럽다
- 구현 리스크가 낮다

### 안 B. 상단 선택 / 하단 분석 콘솔형
권장 2순위

구성:
- 상단 전체 폭: 전표 선택 카드 grid/carousel
- 하단 좌: 실시간 분석
- 하단 우: 결과/근거/HITL

장점:
- 에이전트 중심 시연에는 강함

단점:
- 선택 목록이 많아질수록 공간 관리가 어려움
- 현 구조 대비 리팩토링 비용이 큼

결론:
- 현재는 안 A를 기본안으로 채택한다.

## 10. 수용 기준

- 좌측 영역은 “무엇을 분석할지 고르는 영역”으로만 기능해야 한다.
- 우측 상단은 “에이전트가 지금 무엇을 하는지”를 보여주는 데 집중해야 한다.
- 하단 리뷰 영역은 “실행 후 검토” 목적만 가져야 한다.
- 에이전트 대화와 리뷰 탭의 데이터 중복은 최소화돼야 한다.
- Streamlit 기반이더라도 제품형 데모 수준의 완성도를 가져야 한다.
- 이후 에이전트 스튜디오, 규정문서 라이브러리, 운영 대시보드에도 같은 디자인 시스템을 확장 적용할 수 있어야 한다.

## 11. 최종 결론

커서 문서의 장점은 “실행형 단계 분해와 완료 기준 명시”에 있고, Codex 문서의 장점은 “정보 구조와 UX 원칙의 명확성”에 있다.  
최종안은 두 방향을 결합해 아래 기준으로 확정한다.

- 설계 기준은 Codex 안을 따른다.
- 구현 단계와 작업 관리 방식은 커서 안의 `Phase + 완료 기준` 방식을 따른다.
- 실제 개발은 `안 A` 레이아웃을 기준으로 진행한다.
- 가장 먼저 손볼 것은 “중복 제거 + 실시간 스트림 패널 재구성”이다.

이 문서를 워크스페이스 리팩토링의 최종 기준 문서로 사용한다.

따라서 본 문서는 추가 메모가 아니라 현재 상태 그대로 구현 기준 문서로 사용한다.

### 12.3 기술적 참고 (Streamlit)

- **리렌더**: 버튼/세션 변경 시 전체 페이지가 리렌더되므로, “스트림만 부분 갱신”은 어렵다. “스트림 표시 = 최근 이벤트 N개를 매 rerun마다 다시 그리기”로 두고, N을 20~30 등으로 두는 방식이 현실적이다.
- **CSS**: `st.markdown(unsafe_allow_html=True)`와 `stylable_container`가 이미 쓰이고 있으므로, 6장 디자인 토큰은 `ui/shared.py`의 `inject_css()` 및 workspace 전용 `<style>` 블록으로 확장하면 된다.
- **데이터 소스**: timeline·result·history는 모두 `fetch_case_bundle` 한 번으로 오므로, “같은 데이터 1회 로드 → 3영역에 역할별로만 분배”하는 현재 방향이 코드와 맞다.

### 12.4 결론

- 문서만으로도 “무엇을 왜 바꾸는지”와 “어디까지 바꾸는지”가 명확하므로, **이 문서대로 작업 진행해도 된다.**
- 위 보완 권장사항은 “착수 전 체크리스트” 1~2개 항목 추가와, Phase 3·4·5 착수 시 “구현 수단 1줄”을 각 Phase에 보강하는 수준으로 반영해 두면, 구현 단계에서 의사결정이 더 줄어든다.
