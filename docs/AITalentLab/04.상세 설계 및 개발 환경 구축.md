### Agent 페르소나 및 시스템 프롬프트 (Identity)

Agent의 정체성, 역할, 그리고 답변의 톤앤매너를 정의합니다. 실제 LLM의 System Prompt에 들어갈 핵심 내용입니다.

|              |                                                                                                              |
| ------------ | ------------------------------------------------------------------------------------------------------------ |
| **항목**       | **정의 내용**                                                                                                    |
| **Agent 이름** | Aura (Audit & Risk Analysis Agent)                                                                           |
| **주요 역할**    | 기업 내부 경비 규정 준수 여부를 감시하고, 이상 징후 시 규정·전표 기반 논리적 근거(Reasoning)를 제시하며, HITL을 통해 검증 가능한 조치 방안을 제안하는 **지능형 재무 감사 에이전트** |
| **핵심 목표**    | 휴먼 에러 및 부정 전표 차단, 감사 리드타임 단축, LangGraph·도구·RAG 기반의 투명한 재무 거버넌스 확립 |
| **톤앤매너**     | 분석적이고 객관적인 태도를 유지하며, 전문 회계사(CPA) 수준의 격조 있고 정중한 비즈니스 문체 사용                                                    |
| **제약 사항**    | 추측에 근거한 비난은 금지하며, 반드시 **학습된 규정 조항(Knowledge/RAG)**에 기반하여 답변함. 개인정보 유출을 방지하며 확정되지 않은 결론에 대해서는 '권고'의 형식을 취함. 증거 부족·규정 해석 모호 시 HITL로 담당자 검토 요청 |

### 워크플로우 및 오케스트레이션 (Workflow & Logic)

사용자 입력부터 최종 응답까지 Agent의 사고 과정과 행동 순서를 기술합니다.

**2.1 처리 로직 (Processing Pipeline)**

* **Step 1 (Data Ingestion & Parsing):** 전표 데이터(날짜, 가맹점명, 금액, 근태, MCC 등)를 `case_service.build_analysis_payload()`로 정규화하여 `body_evidence`로 만듭니다. 에이전트 그래프의 **intake** 노드에서 한 번 더 플래그 추출·정규화합니다.

  * *비고: 비정형/원시 데이터를 AI와 도구가 쓰는 정형 body_evidence로 변환하는 단계*

* **Step 2 (Rule Matching):** **policy_rulebook_probe** 도구가 케이스 유형·키워드로 PostgreSQL **rag_chunk**(pgvector/halfvec)에 저장된 사내 규정 청크를 BM25·Dense·RRF·Rerank로 검색하여, 해당 전표와 관련성 높은 규정 조항 후보를 조회합니다.

  * *비고: RAG(검색 증강 생성)로 규정집을 즉시 반영하며, 조항 단위 채택·adoption_reason을 반환*

* **Step 3 (Execution & Response):** 수집된 규정 근거와 전표·도구 결과를 **critic·verify·reporter·finalizer** 노드에서 대조하여 위반 여부를 확정합니다. **Chain of Thought(CoT)** 형태로 노드별 reasoning과 판단요약·조치 권고를 생성하며, 필요 시 HITL로 담당자 검토를 요청합니다.

  * *비고: 판단 근거를 SSE 이벤트와 최종 결과에 함께 제공하여 신뢰성 확보*

#### **2.2 상태 관리 (State Management)**

* **AgentState (LangGraph):** `body_evidence`, `screening_result`, `intended_risk_type`, `plan`, `tool_results`, `score_breakdown`, `hitl_request`, `final_result` 등. HITL 재개 시 `CHECKPOINTER_BACKEND=postgres`면 DB에 체크포인트 저장.

* **Run 타임라인·결과:** `stream_runtime`(메모리)에 run_id별 이벤트 목록·최종 결과 저장. 서버 재시작 시 초기화됨.

* **영구 데이터:** PostgreSQL(dwp_aura) — 전표(fi_doc_header, fi_doc_item), agent_case(스크리닝 결과), rag_chunk(규정 청크·임베딩), analysis_run 메타 등.

### 지식 베이스 및 메모리 전략 (Context & Memory)

LLM이 기억해야 할 단기 대화 내역과 참조해야 할 장기 데이터(RAG) 전략을 수립합니다.

**4.1 RAG (검색 증강 생성) 전략**

* **참조 데이터 소스:** 사내 경비 지출 관리 규정(`규정집/사내_경비_지출_관리_규정_v2.0_확장판.txt`). 규정문서 라이브러리(Streamlit)에서 TXT 업로드·인덱싱 가능.

* **청킹(Chunking) 방식:** 계층/조항/슬라이딩 전략 적용. `chunking_pipeline`, `rag_chunk_lab_service`에서 전략별 미리보기·저장·임베딩. 조항 단위 문맥 유지.

* **임베딩 모델:** OpenAI 호환(`OPENAI_EMBEDDING_MODEL`, 예: text-embedding-3-large). 차원·컬럼은 `OPENAI_EMBEDDING_DIM`, `RAG_EMBEDDING_COLUMN`(embedding_az), `halfvec` 타입.

* **Vector DB:** PostgreSQL pgvector(rag_chunk 테이블). BM25·Dense(RRF)·Rerank(cross-encoder 또는 LLM fallback)로 `policy_service.search_policy_chunks()` 검색.

### 핵심 에이전트 기술 스택

일반적인 웹 개발 스택이 아닌, LLM의 답변 품질과 구조를 제어하기 위한 기술적 의사결정을 기입합니다.

|                     |                              |                                                                                                                                                                     |
| ------------------- | ---------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **구분**              | **선정 전략/기술**                 | **선정 사유 (논리적 근거)**                                                                                                                                                  |
| **LLM Model**       | OpenAI 호환(GPT 계열 등, REASONING_LLM_MODEL 등 .env 설정) | 복합 추론(Reasoning)·다단계 사고(CoT) 수행. 긴 컨텍스트로 전표·규정 전문 동시 처리. |
| **Agent Framework** | LangGraph | 순환·분기·상태(State) 관리가 필요한 감사 워크플로우(start_router→screener/intake→…→finalizer, HITL interrupt/resume) 제어에 사용. |
| **Prompt Strategy** | Chain-of-Thought (CoT) | "규정 확인 → 대조 → 판단" 사고 단계를 노드별 reasoning·이벤트로 투명 공개하여 신뢰도 확보. |
| **Output Parsing**  | Structured Output (Pydantic) | 분석 결과를 정형 JSON(planner_output, execute_output, reporter_output 등)으로 수신하여 UI(추론/근거/탭)에 바인딩. |
| **Monitoring**      | Langfuse  | 에이전트 실행·LLM 호출을 Langfuse 대시보드에서 추적·모니터링. |
