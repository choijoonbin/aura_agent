### 워크플로우 및 오케스트레이션 (Workflow & Logic)

사용자 입력부터 최종 응답까지 Agent의 사고 과정과 행동 순서를 기술합니다.

**2.1 처리 로직 (Processing Pipeline)**

* **Step 1 (Data Ingestion & Parsing):** 시스템에 인입된 Raw 전표 데이터(날짜, 가맹점명, 금액 등)에서 AI 분석에 최적화된 핵심 요소들을 추출하고 정규화합니다.

  * *비고: 비정형 데이터를 AI가 이해할 수 있는 정형 데이터로 변환하는 단계*

* **Step 2 (Rule Matching):** 정규화된 데이터를 기반으로 **pgvector**에 저장된 사내 규정(청크)들과 유사도를 비교하여, 해당 전표와 가장 관련성이 높은 규정 조항들을 실시간으로 탐색합니다.

  * *비고: RAG(검색 증강 생성) 기술을 통해 최신 규정을 즉시 반영*

* **Step 3 (Execution & Response):** 수집된 규정 근거와 전표 데이터를 최종 대조하여 위반 여부를 확정합니다. 이 과정에서 **Chain of Thought(CoT)** 기법을 적용하여 논리적인 추론 결과와 구체적인 조치 권고안을 생성합니다.

  * *비고: 단순히 결과만 주는 것이 아니라 '판단 근거'를 함께 제공하여 신뢰성 확보*

### 지식 베이스 및 메모리 전략 (Context & Memory)

LLM이 기억해야 할 단기 대화 내역과 참조해야 할 장기 데이터(RAG) 전략을 수립합니다.

**4.1 RAG (검색 증강 생성) 전략**

* **참조 데이터 소스:** 사내 경비 지출 관리 규정(`규정집.txt`), 법인카드 사용 지침 매뉴얼, 과거 감사 적발 사례 데이터베이스

* **청킹(Chunking) 방식:** **Semantic Chunking** 적용 (조항 단위로 의미가 끊기지 않도록 문맥 중심 분할, 각 청크당 약 500~800 토큰)

* **임베딩 모델:** `text-embedding-3-small` (OpenAI)

* **Vector DB:** pgvector (PostgreSQL)

### 핵심 에이전트 기술 스택

일반적인 웹 개발 스택이 아닌, LLM의 답변 품질과 구조를 제어하기 위한 기술적 의사결정을 기입합니다.

|                     |                              |                                                                                                                                                                     |
| ------------------- | ---------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **구분**              | **선정 전략/기술**                 | **선정 사유 (논리적 근거)**                                                                                                                                                  |
| **LLM Model**       | GPT-5.0 (또는 상위 모델)           | **복합 추론 성능(Reasoning):** 재무 규정 간의 상충 관계를 파악하고 다단계 사고(CoT)를 수행하는 데 있어 현존 최고 수준의 성능 보유. **긴 컨텍스트 윈도우:** 방대한 전표 데이터와 복잡한 규정 전문을 동시에 처리하면서도 맥락을 놓치지 않는 안정적인 데이터 처리량 보장. |
| **Agent Framework** | LangGraph                    | 단방향 체인을 넘어 순환(Loop) 및 상태(State) 관리가 필수적인 복잡한 감사 워크플로우를 제어하기 위한 최적의 도구                                                                                               |
| **Prompt Strategy** | Chain-of-Thought (CoT)       | AI가 결론만 제시하는 것이 아니라, "규정 확인 -> 대조 -> 판단"의 사고 단계를 투명하게 공개하여 임원진의 신뢰도 확보**Output ParsingStructured Output (Pydantic)**                                                |
| **Output Parsing**  | Structured Output (Pydantic) | 분석 결과를 정형화된 JSON 데이터로 수신하여 프론트엔드 UI(추론/근거/탭 등)에 오차 없이 바인딩하기 위함                                                                                                      |
| **Monitoring**      | LangSmith]                   | 에이전트의 사고 과정과 토큰 비용을 실시간 모니터링하고, 시연 중 발생할 수 있는 이상 응답을 즉시 디버깅하기 위해 선정                                                                                                 |
