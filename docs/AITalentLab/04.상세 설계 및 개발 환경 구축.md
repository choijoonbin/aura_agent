### Agent 페르소나 및 시스템 프롬프트 (Identity)

Agent의 정체성, 역할, 그리고 답변의 톤앤매너를 정의합니다. 실제 LLM의 System Prompt에 들어갈 핵심 내용입니다.

|              |                                                                                                              |
| ------------ | ------------------------------------------------------------------------------------------------------------ |
| **항목**       | **정의 내용**                                                                                                    |
| **Agent 이름** | Aura (Audit & Risk Analysis Agent)                                                                           |
| **주요 역할**    | 기업 내부 경비 규정 준수 여부를 감시하고, 이상 징후 시 규정·전표 기반 논리적 근거(Reasoning)를 제시하며, HITL을 통해 검증 가능한 조치 방안을 제안하는 **지능형 재무 감사 에이전트** |
| **핵심 목표**    | 휴먼 에러 및 부정 전표 차단, 감사 리드타임 단축, LangGraph·도구·RAG 기반의 투명한 재무 거버넌스 확립 |
| **톤앤매너**     | 분석적이고 객관적인 태도를 유지하며, 전문 회계사(CPA) 수준의 격조 있고 정중한 비즈니스 문체 사용                                                    |
| **제약 사항**    | 추측에 근거한 비난은 금지하며, 반드시 **학습된 규정 조항(Knowledge/RAG)**에 기반하여 답변함. 개인정보 유출을 방지하며 확정되지 않은 결론에 대해서는 '권고'의 형식을 취함. 증거 부족·규정 해석 모호 시 HITL로 담당자 검토 요청. 추측 정보 생성 금지("입력에 없는 사실을 만들지 마라"를 시스템 프롬프트에 명시) |

### 워크플로우 및 오케스트레이션 (Workflow & Logic)

사용자 입력부터 최종 응답까지 Agent의 사고 과정과 행동 순서를 기술합니다.

**2.1 처리 로직 (Processing Pipeline)**

전체 흐름은 **Phase 0 (하이브리드 스크리닝)** → **Phase 1 (LangGraph 심층 분석)** 2단계로 구성됩니다.

**Phase 0 — 하이브리드 스크리닝 (Screening)**

전표 생성 시 또는 분석 run 진입 시 수행. LLM 초기 분류 + 결정론적 가드레일 보정으로 케이스 유형을 자율 분류합니다.

* **결정론적 스코어링:** 전표 필드(isHoliday, hrStatus, occurredAt, mccCode, budgetExceeded, amount)에서 시그널을 추출하고 가중 점수 산정. `HOLIDAY_USAGE / LIMIT_EXCEED / PRIVATE_USE_RISK / UNUSUAL_PATTERN / NORMAL_BASELINE` 중 하나로 분류.
* **LLM 분류 + 가드레일 정렬:** LLM(gpt-4o-mini)이 같은 입력을 보고 case_type과 confidence를 제안. 결정론적 시그널과 충돌 시 가드레일이 자동 보정. LLM confidence ≥ 0.75이고 시그널과 모순 없을 때만 LLM 결과 채택.
* **`screening_mode` 기록:** `hybrid`(LLM+결정론적) 또는 `rule`(결정론적만). `SCREENING_MODE` 환경변수로 제어.

**Phase 1 — LangGraph 심층 분석 (Analysis)**

* **Step 1 (start_router → intake):** `case_service.build_analysis_payload()`로 전표 데이터를 정규화하여 `body_evidence`를 생성. start_router에서 사전 스크리닝 여부를 확인하고 분기 (스크리닝 완료 → intake 직행 / 미완료 → screener → intake). intake에서 플래그 추출·멀티모달 OCR 결과 반영.

* **Step 2 (planner):** LLM 플래너(`ENABLE_LLM_PLANNER=true`)가 케이스 유형·flags·스크리닝 결과를 입력으로 받아 조사 계획(plan)과 도구 목록을 결정. 도구별 실행 순서(순차/병렬 그룹)와 파라미터를 구조화된 `PlannerOutput`(Pydantic)으로 반환.

* **Step 3 (execute):** plan 순서대로 LangChain StructuredTool을 호출. `policy_rulebook_probe`·`legacy_aura_deep_audit`는 순차 실행, 나머지는 `ENABLE_PARALLEL_TOOL_EXECUTION=true` 시 병렬 그룹 실행 가능. 각 도구 결과는 `ToolResultEnvelope(tool, ok, facts, summary)` 형태로 수집.

* **Step 4 (critic):** 도구 결과를 검토하여 과잉 주장(unsupported claim)·반례(counter-evidence)를 감지. 최대 2회(`_MAX_CRITIC_LOOP`) 재검토 가능. `check_reasoning_consistency()`로 reasoning 문장과 verdict 필드의 일관성을 자동 검증.

* **Step 5 (verify → HITL):** 하이브리드 스코어링(rule_score + LLM judge) 및 Evidence coverage gate 적용. HITL 필요 여부 판단: 핵심 필드 누락·증거 부족·규정 해석 모호·신호 충돌 시 `hitl_request(reasons, questions, handoff)` 생성 → hitl_pause에서 interrupt. 사용자 응답 제출 후 같은 run으로 resume → hitl_validate → reporter → finalizer만 재실행.

* **Step 6 (reporter → finalizer):** 근거 기반 판단요약·최종 설명 생성. score_breakdown·citation·severity·verdict·권고를 구조화된 `ReporterOutput`·`FinalResult`(Pydantic)로 반환.

**2.2 상태 관리 (State Management)**

* **AgentState (LangGraph TypedDict):** `case_id`, `body_evidence`, `screening_result`, `intended_risk_type`, `flags`, `plan`, `tool_results`, `score_breakdown`, `critique`, `verification`, `hitl_request`, `final_result`, `planner_output`, `execute_output`, `critic_output`, `verifier_output`, `reporter_output`, `visual_audit_results`, `latency_ms`, `evaluation_history` 등 전체 분석 이력 포함.

* **체크포인터 (HITL 재개):** `CHECKPOINTER_BACKEND=postgres` 설정 시 `langgraph-checkpoint-postgres`가 DB에 체크포인트 저장. 재개 시 hitl_validate → reporter → finalizer만 실행. 기본값 `memory`는 단일 프로세스 내에서만 보장.

* **Run 타임라인·결과:** `stream_runtime`(메모리)에 run_id별 이벤트 목록·최종 결과 저장. `runtime_persistence_service`가 DB(analysis_run 테이블)에 run 메타·이벤트를 영속화하여 재시작 후에도 타임라인 조회 가능.

* **영구 데이터:** PostgreSQL(dwp_aura) — 전표(fi_doc_header, fi_doc_item), agent_case(스크리닝 결과·최종 판단), rag_chunk(규정 청크·임베딩·halfvec), analysis_run(run 메타·이벤트), evidence_uploads(증빙 이미지).

### 도구(Tools) 및 함수 명세 (Capability)

Agent가 외부 세상과 상호작용하기 위해 사용할 도구(Function Calling)를 정의합니다.

| **도구명 (Function Name)** | **기능 설명 (Description)** | **입력 파라미터 (Input Schema)** | **출력 데이터 (Output)** |
| ----------------------- | ----------------------- | -------------------------- | ------------------- |
| holiday_compliance_probe | 휴일/휴무/연차 사용 정황 검증 | ToolContextInput(body_evidence: 전표 발생시각·근태 상태 포함) | `{occurredAt, isHoliday, hrStatus, holidayRisk, summary}` |
| budget_risk_probe | 예산 초과 여부·금액 지표 검증 | ToolContextInput(body_evidence: 금액·예산 초과 플래그 포함) | `{amount, budgetExceeded, summary}` |
| merchant_risk_probe | 거래처·MCC 위험도 검증. 선행 holiday_compliance 결과 활용(복합 위험 탐지) | ToolContextInput(body_evidence + prior_tool_results) | `{mccCode, merchantName, merchantRisk, compoundRiskFlags, holidayRiskConsidered, summary}` |
| document_evidence_probe | 전표 라인아이템·문서 증거 수집 | ToolContextInput(body_evidence: 전표·문서 정보) | `{lineItemCount, lineItems, summary}` |
| policy_rulebook_probe | 내부 규정집 관련 조항 조회. prior 도구 결과 기반 문맥 강화(enriched_from_prior) | ToolContextInput(body_evidence + prior_tool_results) | `{policy_refs, ref_count, retrieval_candidates, enriched_from_prior, summary}` |
| legacy_aura_deep_audit | 기존 Aura 심층 분석 호출(선택적). 규정 근거·증거가 충분하면 자동 스킵 | ToolContextInput(body_evidence) | 활성화 시 legacy 심층 분석 결과. `ENABLE_LEGACY_AURA_SPECIALIST=true` 필요 |

**공통 입력 스키마 (`ToolContextInput`, Pydantic):**
- `case_id: str` (필수)
- `body_evidence: dict` (필수, default `{}`)
- `intended_risk_type: str | None`
- `prior_tool_results: list[dict]` (default `[]`)

### 지식 베이스 및 메모리 전략 (Context & Memory)

**4.1 RAG (검색 증강 생성) 전략**

* **참조 데이터 소스:** 사내 경비 지출 관리 규정(`규정집/사내_경비_지출_관리_규정_v2.0_확장판.txt`). 규정문서 라이브러리(Streamlit)에서 TXT 업로드·인덱싱 가능.

* **청킹(Chunking) 전략:** 계층/조항/슬라이딩 3종 전략 적용. `chunking_pipeline`, `rag_chunk_lab_service`에서 전략별 미리보기·저장·임베딩 지원. 조항(article) 단위 문맥 유지 및 공통 증빙 의무 조항(제14조) 우선 채택 로직 포함.

* **임베딩 모델:** OpenAI 호환(`OPENAI_EMBEDDING_MODEL`, 예: text-embedding-3-large). 차원 3072, `halfvec` 타입으로 PostgreSQL pgvector에 저장.

* **검색 전략 (Hybrid Search):**
  1. **BM25 (키워드 검색):** 규정 조항 고유 용어·조항 번호 검색에 강점
  2. **Dense 검색 (벡터 유사도):** 의미 기반 검색. HyDE(가설 문서 임베딩) 옵션(`ENABLE_HYDE_QUERY`) 지원
  3. **RRF (Reciprocal Rank Fusion):** BM25·Dense 결과를 랭킹 기반으로 결합하여 재현율(Recall) 향상
  4. **Rerank:** cross-encoder 또는 LLM fallback(`ENABLE_LLM_RERANK_FALLBACK=true`)으로 최종 순위 보정
  5. **조문(article) 단위 중복 제거:** 동일 조항에서 추출된 청크를 묶어 상위 N개 채택. `adoption_reason` 필드로 채택 근거 기록.

* **검색 API:** `policy_service.search_policy_chunks()` — 위 전략을 순서대로 적용하여 최종 상위 조항 후보 반환.

**4.2 대화 메모리 (Conversation History)**

본 프로젝트는 다중 대화 세션 메모리 대신 **run 단위 이벤트 기반 메모리**를 채택합니다.

* **설계 이유:** 재무 감사 도메인에서 각 전표 분석은 독립적 사건(case)으로 처리되며, 이전 케이스의 판단이 현재 케이스에 오염되는 것을 방지. 대신 AgentState가 run 내 전체 컨텍스트(전표·도구 결과·스크리닝·HITL 이력 등)를 완전히 담아 단일 run 내 모든 노드가 공유.

* **저장 전략:**
  - run_id별 이벤트 목록·최종 분석 결과 → `stream_runtime`(메모리) + `runtime_persistence_service`(DB 영속화)
  - HITL 재개용 체크포인트 → `CHECKPOINTER_BACKEND=postgres` 설정 시 DB 저장
  - 평가 이력(evaluation_history) → AgentState에 누적, finalizer에서 DB 반영

* **실행 이력 조회:** `GET /api/v1/cases/{voucher_key}/runs/{run_id}/timeline` API로 이벤트 타임라인 조회. `GET /api/v1/cases/{voucher_key}/runs` API로 전체 run 목록 조회.

### 핵심 에이전트 기술 스택

일반적인 웹 개발 스택이 아닌, LLM의 답변 품질과 구조를 제어하기 위한 기술적 의사결정을 기입합니다.

|                     |                              |                                                                                                                                                                                                                                                                                                                             |
| ------------------- | ---------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **구분**              | **선정 전략/기술**                 | **선정 사유 (논리적 근거)**                                                                                                                                                                                                                                                                                                          |
| **LLM Model**       | OpenAI 호환(gpt-4o-mini 등, `REASONING_LLM_MODEL` .env 설정) | 복합 추론(Reasoning)·다단계 사고(CoT) 수행. 긴 컨텍스트로 전표·규정 전문 동시 처리. Azure OpenAI 엔드포인트 지원으로 사내 보안 정책 대응 가능. |
| **Agent Framework** | **LangGraph** (StateGraph, conditional_edges, interrupt) | 순환·분기·상태(State) 관리가 필요한 감사 워크플로우(start_router→screener/intake→planner→execute→critic→verify→hitl_pause→reporter→finalizer, HITL interrupt/resume) 제어에 최적. 단순 체인(Chain) 대비 복잡한 조건 분기와 상태 공유가 가능. |
| **Prompt Strategy** | **Chain-of-Thought (CoT)** + Structured Output (Pydantic) | "규정 확인 → 대조 → 판단" 사고 단계를 노드별 reasoning·이벤트로 투명 공개하여 신뢰도 확보. Pydantic 구조화 출력(`PlannerOutput`, `CriticOutput`, `VerifierOutput`, `ReporterOutput`)으로 UI 바인딩 및 일관성 검증. |
| **Hybrid Screening** | LLM 초기 분류 + 결정론적 가드레일 | LLM 단독 분류의 환각·오분류 리스크를 결정론적 시그널 가드레일로 보정. LLM confidence < 0.75이거나 시그널 충돌 시 결정론적 결과 채택. 두 방식의 장점을 결합한 신뢰성 확보. |
| **Hybrid Scoring**  | rule_score + LLM judge 가중 합산 | 결정론적 규칙(isHoliday, hrStatus, MCC 위험도 등) 점수와 LLM judge 점수를 가중 합산. LLM judge 비활성 또는 circuit breaker 발동 시 rule_score로 안전하게 fallback. `LLM_JUDGE_ENABLED` 환경변수로 제어. |
| **Tool Framework**  | **LangChain StructuredTool** | 에이전트가 자율적으로 도구를 선택·호출하는 구조. 공통 `ToolContextInput` 스키마로 body_evidence·prior_tool_results 공유. 병렬 실행(ENABLE_PARALLEL_TOOL_EXECUTION) 지원. |
| **RAG/Vector DB**   | PostgreSQL pgvector + BM25/Dense/RRF/Rerank | 별도 벡터 DB 없이 기존 PostgreSQL에 pgvector 확장으로 임베딩 저장. 운영 부담 최소화. Hybrid Search(BM25+Dense+RRF)로 규정 조항 검색 재현율 향상. |
| **Streaming**       | FastAPI SSE (Server-Sent Events) | 에이전트 노드/도구 실행 이벤트를 실시간 스트리밍. 구조화된 이벤트 스키마(NODE_START/END, PLAN_READY, TOOL_CALL/TOOL_RESULT, SCREENING_RESULT, HITL_REQUESTED, GATE_APPLIED)로 UI 타임라인 자동 구성. |
| **Monitoring**      | **Langfuse** (선택) | `LANGFUSE_ENABLED=true` 설정 시 LangChain CallbackHandler로 각 run의 노드/LLM 호출을 Langfuse 대시보드에서 세션별로 추적·모니터링. 비활성화 시 런타임 영향 없음. |
| **Multimodal**      | pytesseract OCR + Azure Document Intelligence (선택) | 이미지 영수증(PNG/JPG)에서 금액·날짜·업종·가맹점명 자동 추출. 해시 기반 캐싱으로 중복 처리 방지. Azure Document Intelligence 연동 시 정밀도 향상. |

### 개발 환경 구성

**기술 스택 (Tech Stack)**

| 레이어 | 기술 | 버전/비고 |
|--------|------|-----------|
| **Backend API** | FastAPI | Python 3.11+ |
| **Frontend UI** | Streamlit | 멀티페이지 앱 (AI 워크스페이스, 에이전트 스튜디오, RAG 라이브러리, 시연 데이터 제어) |
| **Agent Runtime** | LangGraph + LangChain | StateGraph, conditional_edges, interrupt/resume |
| **LLM** | OpenAI 호환 (Azure OpenAI or OpenAI) | gpt-4o-mini (스크리닝/추론/judge), text-embedding-3-large (임베딩) |
| **Database** | PostgreSQL + pgvector | dwp_aura 스키마, halfvec(3072차원) |
| **Observability** | Langfuse (선택) | LANGFUSE_ENABLED=true 시 활성화 |
| **Graph DB** | Neo4j (선택) | ENABLE_GRAPH_DB=true 시 케이스 관계 탐색 |
| **OCR** | pytesseract / Azure Document Intelligence (선택) | 이미지 영수증 추출 |
| **Checkpoint** | langgraph-checkpoint-postgres (선택) | CHECKPOINTER_BACKEND=postgres 시 HITL 재개 보장 |
