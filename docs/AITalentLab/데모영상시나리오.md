**화면:** 시연 데이터 생성 메뉴 → 영수증 이미지 업로드 → Streamlit 터미널 확대

**멘트:**

**[행동 2] 영수증 이미지 업로드 → 「이미지 분석」 클릭**

**화면 전환:** Streamlit 터미널을 오른쪽 절반으로 확대

**[Streamlit 터미널에서 강조할 로그]**

> ⚠️ OCR·Vision LLM 로그는 **Streamlit 터미널**에 출력됩니다

```
[vllm:analyze] 증빙 이미지 분석 시작 | model=gpt-4o timeout=45.0s
[vllm:ocr] PaddleOCR 실행 시작
[vllm:ocr] PaddleOCR 3.x 완료 | blocks=32 avg_conf=0.98 img=1536x2752
[vllm:ocr] 완료: 32개 텍스트 블록 추출
[vllm:llm] OCR→LLM 필드 매핑 요청 | model=gpt-4o ocr_blocks=32
[vllm:match] LLM 매핑 결과 | merchant=가옹식당 강남점 | date=2026-03-14 | time=23:42 | amount=68000
[vllm:result] 추출완료 | condition=clear avg_conf=0.98 entities=4개
[vllm:result] merchant_name="가옹식당 강남점"(conf=0.94) | date_occurrence="2026-03-14"(conf=0.96) | time_occurrence="23:42"(conf=1.00) | amount_total="68000"(conf=0.97)
[vllm:analyze] PaddleOCR+LLM 경로 완료 | entities=4개
```

**멘트:**
> "영수증 이미지 한 장을 업로드했습니다. 두 단계가 순서대로 작동하는 것이 보이시나요?
>
> **첫 번째**, PaddleOCR — 이미지에서 32개의 텍스트 블록을 픽셀 단위로 추출합니다. 평균 신뢰도 0.98 — 거의 완벽한 추출입니다.
>
> **두 번째**, GPT-4o — 추출된 텍스트 블록들을 의미 기반으로 해석해 필드를 매핑합니다. 단순 OCR이 아니라, '이 숫자가 금액인지 전화번호인지'를 LLM이 맥락으로 판단합니다.
>
> 결과를 보시면 — 가맹점명, 날짜, 시각, 금액 4개 필드가 모두 자동 추출됐습니다. 수작업 입력이 필요 없고, 이 데이터는 이후 전표 정보와의 불일치 감지에도 그대로 활용됩니다."

> 전표가 생성되는 순간, AI가 자동으로 하이브리드 스크리닝을 시작합니다."

**[Streamlit 터미널에서 강조할 로그 — 마우스로 해당 줄 가리키며 설명]**

```
[screener:llm] LLM 분류 요청 | model=gpt-4o-mini response_format={'type': 'json_object'}
[screener:llm] LLM 제안 | model=gpt-4o-mini case=HOLIDAY_USAGE conf=0.9 reason=휴일에 사용한 경비...
[screener:llm] LLM 분류 결과 | llm_case=HOLIDAY_USAGE(conf=0.90) vs rule_case=HOLIDAY_USAGE(score=95) min_conf=0.75
[screener:guardrail] 최종결정 | LLM 채택 → case=HOLIDAY_USAGE | reason=llm_rule_agree
[screener:result] 스크리닝 완료 | case=HOLIDAY_USAGE score=95 severity=CRITICAL llm=HOLIDAY_USAGE(conf=0.90) align=llm_rule_agree
```

> **멘트 포인트:** 정확한 숫자보다 구조를 설명하세요.

**멘트:**
> "스크리닝 엔진 두 개가 동시에 작동합니다. LLM이 전표 맥락을 분석해 HOLIDAY_USAGE — 신뢰도 0.90. 기준치 0.75를 넘었습니다.
>
> 세 번째 줄이 핵심입니다. **규칙 엔진도 score=95로 동일하게 HOLIDAY_USAGE** — 두 엔진이 일치합니다(`llm_rule_agree`). 가드레일이 합의를 확인하고 최종 확정합니다.
>
> severity=**CRITICAL**, 위험 점수 95. 이 하이브리드 구조가 단순 LLM 분류 대비 오분류를 방지하는 핵심 설계입니다."

  4. 위험 신호 종합 평가 (critic)

**멘트:**

> 어떤 도구를 어떤 순서로 사용할지 결정하는 단계입니다. 이것이 ReAct를 넘어선 **Planning 기반 Agentic AI**입니다."

### ▶ 2:00 ~ 2:30 | Execute 노드 — 도구 호출 + RAG

**멘트:**
> "execute 노드에서 계획에 따라 도구를 순차 호출합니다."

**[터미널에서 강조할 로그]**

```
[llm:reasoning] 요청 | node=execute model=<실제 모델명>
[rag:search] case=HOLIDAY_USAGE keywords=[법인카드, 휴일사용, 야간결제, 근태연차, MCC5813](5개) | bm25_w=0.40 dense_w=0.60 group=none
[rag:retrieve] BM25 candidates=12 dense candidates=8
[rag:rerank] 입력 chunks=20 → 상위 5개 선택
[rag:result] 최종선택=3건 채택조항: ['제39조', '제23조', '제12조']
```

> **실제 로그 형식:** `[rag:result] 최종선택=N건 채택조항: [...]` — 조항명은 실행 시마다 달라질 수 있습니다.

**멘트:**
> "RAG 검색을 보시면 단순 키워드 검색이 아닙니다.
>
> '법인카드, 휴일사용, 야간결제, 근태연차, MCC5813' — 케이스 맥락에서 자동 추출한 키워드로 BM25 + Dense 벡터 검색을 병렬로 실행하고, RRF 융합 + Reranker로 정제합니다.
>
> 결과로 휴일 규정, 식대 규정, 사전승인 관련 세 개의 규정 조항이 근거로 선택됩니다."

>
> 화면에 나타난 검토 결과를 바탕으로, AI가 최종 보고서 내용을 스스로 보완합니다.
