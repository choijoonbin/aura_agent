### 핵심 사용자 시나리오

본 문서는 **사용자 관점에서 서비스 시나리오를 정의**하고, PoC 시연 수준에서 **핵심 시나리오의 입출력과 성공 기준**을 수립한다. 현재 프로젝트는 **시연 데이터 제어**로 시나리오별 전표를 생성하고, **AI 워크스페이스**에서 스크리닝·심층 분석·HITL 참여까지 한 흐름으로 시연할 수 있다.

**시연 시 테스트 데이터:** 시연에서는 전체 시나리오(휴일 사용 의심, 한도 초과, 사적 사용 위험, 비정상 패턴, 정상 비교군)를 모두 쓰지 않고, **휴일 사용 의심(HOLIDAY_USAGE)** 과 **정상 비교군(NORMAL_BASELINE)** 위주로 테스트 데이터를 생성해 시연한다. 고위험·정상 대비를 명확히 보여 주기 위한 선택이다.

**테스트 데이터 생성 시 스크리닝:** 시연 데이터 제어에서 전표를 **생성할 때** 백엔드가 곧바로 스크리닝을 실행해, 해당 전표의 `agent_case`에 `case_type`·`screening_reason_text` 등을 저장한다. 따라서 워크스페이스 전표 목록에는 **이미 스크리닝된 결과**(고위험/정상 배지)가 반영된 상태로 보인다.

**화면 동작:** **화면에는 「분석 실행」 버튼만 있다.** 「분석 실행」을 누르면 **사전 스크리닝된 전표**는 run 내에서 screener 노드를 건너뛰고 **intake부터** 진행되며, 스크리닝 결과는 SCREENING_RESULT 이벤트로 스트림에만 발행된다. 스크리닝이 아직 없는 전표는 start_router 후 screener → intake 순으로 진행된다. 스크리닝 전용 API(`POST …/screen`)는 백엔드·자동화용으로만 제공된다.

**시나리오 1: [시연 데이터 생성 및 분석 실행 시 스크리닝 결과 확인 — 고위험 케이스 자동 분류]**

* **ID:** SC-001

* **상황:** 시연·검증 시 진행자가 **휴일 사용 의심·정상 비교군** 등 시연용 시나리오로 전표를 생성하면, **생성 시점에 백엔드에서 스크리닝이 수행**되어 리스트/상세에 이미 케이스 유형·스크리닝 사유가 반영된다. 이어서 **「분석 실행」**을 누르면 사전 스크리닝된 전표는 run 내에서 screener를 건너뛰고 intake부터 심층 분석이 진행되며, 스크리닝 결과는 이벤트로 확인할 수 있다.

* **목표:** 테스트 데이터 **생성 시점**에 전표 원시 데이터만으로 위험 유형이 스크리닝되고, 그 결과(케이스 유형·스크리닝 사유)가 **리스트·상세·분석 run 이벤트**에서 일관되게 확인되며, 고위험 vs 정상을 식별할 수 있게 한다.

* **사전 조건:** 시연 데이터 제어(Streamlit 또는 API) 사용 가능, **분석 실행** API(`POST /api/v1/cases/{voucher_key}/analysis-runs`) 사용 가능. 전표 생성 시 스크리닝이 자동 수행되며, 분석 시작 시 **사전 스크리닝된 경우** 그래프는 **start_router → intake → … → finalizer**(screener 생략), **미스크리닝 건**은 **start_router → screener → intake → …** 순으로 실행된다.

* **상세 흐름**

| 단계 | 사용자 행동 | 시스템 동작 | 비고 |
|------|-------------|-------------|------|
| 1 | 시연 데이터 제어에서 **휴일 사용 의심**, **정상 비교군** 시나리오를 선택해 각각 전표 N건 생성 | 전표가 DB에 삽입된 뒤 **백엔드에서 곧바로 스크리닝**이 실행되어 agent_case에 case_type·screening_reason_text 등이 저장됨. voucher_key는 DEMO-{시나리오}-{순번} 형식 | 생성 시점 스크리닝 |
| 2 | AI 워크스페이스에서 전표 목록·상세 확인 | **이미 스크리닝된 결과**가 리스트/상세에 반영됨(케이스 유형 배지, 스크리닝 사유). 전표 선택 후 **「분석 실행」** 클릭 | 단일 뷰 |
| 3 | **「분석 실행」** 클릭 후 실시간 사고 흐름 또는 이벤트 확인 | 사전 스크리닝된 전표는 **screener 노드 없이** intake부터 진행. SCREENING_RESULT 이벤트는 **사전 적용 결과**로 스트림에 발행됨. 이어서 planner → … → finalizer 순으로 심층 분석 진행 | 스크리닝 로직은 생성 시 1회만 실행 |
| 4 | 케이스 리스트 또는 상세에서 분류 결과 확인 | case_type, screening_reason_text가 리스트·상세·run 이벤트에서 일관되게 확인 가능 | 근거 텍스트 제공 |

* **입력 예시**

휴일 사용 의심(HOLIDAY_USAGE) 시나리오로 생성된 전표:  
결제 시각 토요일 23:xx, hrStatus=LEAVE, mccCode=5813, budgetExceeded=N, 적요 "휴일 야간 식대" 등.

* **기대 출력**

`case_type`: HOLIDAY_USAGE(또는 동일 위험군). `screening_reason_text`에 "주말/휴일 발생", "근태 상태 LEAVE", "심야 시간대 사용", "고위험 업종 MCC 5813" 등 결정론적 근거가 포함됨. 리스트/상세에서 해당 전표가 "휴일 사용 의심" 등으로 배지 표시됨.

* **성공 기준**

  * **기준 1:** 테스트 데이터 **생성 시점**에 전표 필드만으로 스크리닝이 수행되고, intended_risk_type(시스템이 미리 주는 위험 유형 힌트) 없이 케이스 유형이 분류되는가?
  * **기준 2:** 스크리닝 결과(케이스 유형·스크리닝 사유)가 **리스트·상세·분석 run 이벤트**에서 일관되게 확인 가능한가?

**시나리오 2: [심층 분석 실행 및 근거 기반 판단 — HITL 검토 후 재분석]**

* **ID:** SC-002

* **상황:** 실무자가 고위험(또는 관심) 전표에 대해 "왜 위반인가?"에 대한 **규정 근거와 조치 가이드**를 받고, 필요 시 HITL로 검토에 참여한 뒤 최종 판단을 확정하고 싶을 때.

* **목표:** LangGraph 심층 분석(intake→planner→execute→critic→verify→reporter→finalizer)을 실행하고, 규정 조항 인용·점수·판단요약을 확인하며, HITL 요청 시 응답 제출 후 **재분석(이어하기)**으로 최종 결과가 갱신되는지 시연한다.

* **사전 조건:** RAG 규정집 인덱싱 완료, 시연 전표 생성(생성 시 스크리닝 자동 수행됨). Streamlit AI 워크스페이스 또는 분석/HITL API 사용 가능. 사전 스크리닝된 전표는 분석 run에서 screener를 건너뛰고 intake부터 진행된다.

* **상세 흐름**

| 단계 | 사용자 행동 | 시스템 동작 | 비고 |
|------|-------------|-------------|------|
| 1 | AI 워크스페이스에서 전표 선택 후 "분석 실행" 클릭 | 분석 run 생성. (사전 스크리닝된 전표는 screener 생략) intake→planner→execute→… 순차 실행. SSE로 NODE_START/END, TOOL_CALL/TOOL_RESULT, SCREENING_RESULT(사전 적용 시) 실시간 전달 | 실시간 사고 흐름 |
| 2 | 실시간 사고 흐름 탭에서 노드·도구 진행 상황 확인 | intake(입력 정규화), planner(조사 계획), execute(도구 호출), critic(반례 검토), verify(게이트·HITL 판단), reporter(설명·요약), finalizer(최종 확정) 이벤트 표시 | 설명 가능한 실행 |
| 3 | 최종 결과 탭에서 규정 조항·점수·판단요약 확인 | policy_rulebook_probe 등 도구 결과 기반 규정 조항 인용, score_breakdown, 판단요약(verdict·권고) 표시 | 근거 기반 의사결정 |
| 4 | HITL 요청 발생 시 질문에 응답 후 "제출" → (선택) "분석 이어하기" | hitl_pause에서 interrupt. 응답 제출 시 같은 run으로 resume → hitl_validate → reporter → finalizer만 재실행. 최종 설명·판단요약 갱신 | 인간-AI 협업(HITL) |

* **입력 예시**

시연 시에는 **휴일 사용 의심(HOLIDAY_USAGE)** 전표를 주로 사용: 결제 시각 주말/심야, hrStatus=LEAVE, mccCode=5813, 적요 "휴일 야간 식대" 등. (필요 시 한도 초과 등 다른 시나리오 전표로도 동일 흐름 시연 가능.)

* **기대 출력**

규정 조항(예: 업무시간 외 결제·휴일 사용 관련 조항 등) 인용, score_breakdown(도구별 점수), "휴일/심야 사용 위반 … 환수 또는 소명 권고" 수준의 **판단요약**. HITL 발생 시: reasons·questions·handoff가 표시되고, 사용자 응답 제출 후 재분석 시 **갱신된 최종 설명·판단요약**.

* **성공 기준**

  * **기준 1:** AI가 위반 근거를 **규정 조항과 도구 결과**로 제시하고, 논리적인 후속 조치(환수/소명 등)를 권고하는가?
  * **기준 2:** 사용자가 실시간 이벤트(SSE)와 최종 근거를 신뢰하고, **HITL 참여 후 재분석 결과**까지 한 흐름으로 확인 가능한가?

### 시나리오 우선순위 매트릭스

| 시나리오 | 비즈니스 가치 | 구현 난이도 | PoC 시연 포함 |
|----------|----------------|-------------|----------------|
| SC-001 (스크리닝) | **높음** (고위험 케이스 선별·자동 분류) | 낮음(결정론적 스코어링) | 네 |
| SC-002 (심층 분석·HITL) | **매우 높음** (근거 제시·의사결정 지원·HITL 협업) | 중간(LangGraph·도구·RAG) | 네 |

시연 시 권장 순서: **휴일 사용 의심**·**정상 비교군**으로 테스트 데이터를 생성하면(생성 시점에 스크리닝 자동 수행), 리스트에서 **SC-001**으로 스크리닝 결과(고위험 vs 정상) 확인 → 전표 선택 후 **「분석 실행」**으로 **SC-002** 심층 분석(규정 인용·판단요약·HITL) 확인. 분석 run 내에서는 사전 스크리닝된 전표에 대해 스크리닝 로직을 다시 실행하지 않는다.
