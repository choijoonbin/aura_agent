### 핵심 사용자 시나리오

본 문서는 **사용자 관점에서 서비스 시나리오를 정의**하고, PoC 시연 수준에서 **핵심 시나리오의 입출력과 성공 기준**을 수립한다. 현재 프로젝트는 **시연 데이터 제어**로 시나리오별 전표를 생성하고, **AI 워크스페이스**에서 스크리닝·심층 분석·HITL 참여까지 한 흐름으로 시연할 수 있다.

**시연 전략:** 5종의 케이스 유형(HOLIDAY_USAGE / LIMIT_EXCEED / PRIVATE_USE_RISK / UNUSUAL_PATTERN / NORMAL_BASELINE) 중 **휴일 사용 의심(HOLIDAY_USAGE)** 과 **정상 비교군(NORMAL_BASELINE)** 을 핵심 시연 케이스로 선정. 고위험 전표와 정상 전표의 극명한 대비를 통해 AI의 자율 분류·판단·HITL 협업 능력을 효과적으로 어필한다. 나머지 3종(LIMIT_EXCEED / PRIVATE_USE_RISK / UNUSUAL_PATTERN)은 시스템이 완전히 지원하며, 시연 시간 여유가 있을 때 추가 시연한다.

**테스트 데이터 생성 시 스크리닝:** 시연 데이터 제어에서 전표를 **생성할 때** 백엔드가 곧바로 하이브리드 스크리닝(LLM 초기 분류 + 결정론적 가드레일)을 실행해, 해당 전표의 `agent_case`에 `case_type`·`screening_reason_text`·`screening_mode` 등을 저장한다. 따라서 워크스페이스 전표 목록에는 **이미 스크리닝된 결과**(고위험/정상 배지)가 반영된 상태로 보인다.

**화면 동작:** **화면에는 「분석 실행」 버튼만 있다.** 「분석 실행」을 누르면 **사전 스크리닝된 전표**는 run 내에서 screener 노드를 건너뛰고 **intake부터** 진행되며, 스크리닝 결과는 SCREENING_RESULT 이벤트로 스트림에만 발행된다. 스크리닝이 아직 없는 전표는 start_router 후 screener → intake 순으로 진행된다.

**시나리오 1: [시연 데이터 생성 및 고위험 케이스 자동 분류 — 하이브리드 스크리닝]**

- **ID:** SC-001
- **상황:** 시연·검증 시 진행자가 **휴일 사용 의심·정상 비교군** 등 시연용 시나리오로 전표를 생성하면, **생성 시점에 백엔드에서 하이브리드 스크리닝**(LLM 초기 분류 + 결정론적 가드레일 보정)이 수행되어 리스트/상세에 이미 케이스 유형·스크리닝 사유가 반영된다.
- **목표:** AI가 **전표 원시 데이터만으로**(의도적 위험 유형 힌트 없이) 케이스 유형을 자율 분류하고, 하이브리드 스크리닝 결과(케이스 유형·사유·모드)가 **리스트·상세·분석 run 이벤트**에서 일관되게 확인되며, 고위험 vs 정상을 식별할 수 있게 한다.
- **사전 조건:** 시연 데이터 제어(Streamlit 또는 API) 사용 가능. 전표 생성 시 하이브리드 스크리닝이 자동 수행됨.
- **상세 흐름**


| 단계  | 사용자 행동                                                     | 시스템 동작                                                                                                                                                           | 비고         |
| --- | ---------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------- |
| 1   | 시연 데이터 제어에서 **휴일 사용 의심**, **정상 비교군** 시나리오를 선택해 각각 전표 N건 생성 | 전표가 DB에 삽입된 뒤 **백엔드에서 곧바로 하이브리드 스크리닝** 실행. LLM 분류 결과와 결정론적 점수를 가드레일로 정렬하여 agent_case에 case_type·screening_reason_text·screening_mode·hybrid_case_align_reason 저장 | 생성 시점 스크리닝 |
| 2   | AI 워크스페이스에서 전표 목록·상세 확인                                    | **이미 스크리닝된 결과**가 리스트/상세에 반영됨(케이스 유형 배지, 스크리닝 사유). HOLIDAY_USAGE는 **고위험** 배지, NORMAL_BASELINE은 **정상** 배지로 명확하게 구분                                                 | 단일 뷰       |
| 3   | 전표 선택 후 **「분석 실행」** 클릭                                     | 사전 스크리닝된 전표는 **screener 노드 없이** intake부터 진행. SCREENING_RESULT 이벤트(사전 적용 결과)가 스트림에 발행됨                                                                            | 중복 스크리닝 방지 |
| 4   | 케이스 리스트 또는 상세에서 분류 결과 확인                                   | case_type, screening_reason_text, screening_mode(hybrid/rule)가 리스트·상세·run 이벤트에서 일관되게 확인 가능                                                                       | 근거 텍스트 제공  |


- **입력 예시**

휴일 사용 의심(HOLIDAY_USAGE) 시나리오로 생성된 전표:
결제 시각 토요일 23:xx, hrStatus=LEAVE, mccCode=5813(고위험 업종), budgetExceeded=N, 적요 "휴일 야간 식대"

정상 비교군(NORMAL_BASELINE) 시나리오로 생성된 전표:
결제 시각 평일 11~13시, hrStatus=WORK, mccCode=5816(일반 업종), budgetExceeded=N

- **기대 출력**

HOLIDAY_USAGE: `case_type=HOLIDAY_USAGE`, `screening_mode=hybrid`, `screening_reason_text`에 "주말/휴일 발생", "근태 상태 LEAVE", "심야 시간대 사용(23시)", "고위험 업종 MCC 5813" 등 결정론적 근거 + LLM 사유 포함.
NORMAL_BASELINE: `case_type=NORMAL_BASELINE`, `score` 낮음, 정상 배지 표시.

- **성공 기준**
  - **기준 1:** 전표 필드만으로(의도적 위험 유형 힌트 없이) 하이브리드 스크리닝이 수행되고, LLM과 결정론적 결과가 가드레일로 정렬되어 케이스 유형이 분류되는가?
  - **기준 2:** 스크리닝 결과(케이스 유형·스크리닝 사유·모드)가 리스트·상세·분석 run 이벤트에서 일관되게 확인 가능한가?
  - **기준 3:** HOLIDAY_USAGE(고위험)와 NORMAL_BASELINE(정상)이 명확하게 구분되어 배지로 표시되는가?

**시나리오 2: [심층 분석 실행 및 근거 기반 판단 — LangGraph 에이전트 + HITL 협업]**

- **ID:** SC-002
- **상황:** 실무자가 고위험(또는 관심) 전표에 대해 "왜 위반인가?"에 대한 **규정 근거와 조치 가이드**를 받고, 필요 시 HITL로 검토에 참여한 뒤 최종 판단을 확정하고 싶을 때.
- **목표:** LangGraph 심층 분석(intake→planner→execute→critic→verify→hitl_pause(선택)→hitl_validate→reporter→finalizer)을 실행하고, 규정 조항 인용·점수·판단요약을 확인하며, HITL 요청 시 응답 제출 후 **재분석(이어하기)**으로 최종 결과가 갱신되는지 시연한다.
- **사전 조건:** RAG 규정집 인덱싱 완료(청킹·임베딩·pgvector 저장), 시연 전표 생성(생성 시 스크리닝 자동 수행됨). Streamlit AI 워크스페이스 또는 분석/HITL API 사용 가능.
- **상세 흐름**


| 단계  | 사용자 행동                               | 시스템 동작                                                                                                                                                                      | 비고             |
| --- | ------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------- |
| 1   | AI 워크스페이스에서 전표 선택 후 「분석 실행」 클릭       | 분석 run 생성. 사전 스크리닝된 전표는 screener 생략. intake→planner→execute→critic→verify 순차 실행. SSE로 NODE_START/END, PLAN_READY, TOOL_CALL/TOOL_RESULT, SCREENING_RESULT 실시간 전달            | 실시간 사고 흐름      |
| 2   | 실시간 사고 흐름 탭에서 노드·도구 진행 상황 확인         | planner가 케이스 유형에 맞는 도구 목록(예: holiday_compliance_probe, document_evidence_probe, policy_rulebook_probe) 결정 후, execute에서 LangChain StructuredTool 순차/병렬 호출. critic에서 과잉 주장 검토 | 다단계 추론(CoT)    |
| 3   | 최종 결과 탭에서 규정 조항·점수·판단요약 확인           | policy_rulebook_probe 결과 기반 규정 조항 인용(citation), score_breakdown(정책 신호·도구별 가산점·가중치), 판단요약(verdict·권고·severity) 표시                                                            | 근거 기반 의사결정     |
| 4   | HITL 요청 발생 시 질문에 응답 후 「제출」→「분석 이어하기」 | hitl_pause에서 interrupt. 응답 제출 시 같은 run으로 resume → hitl_validate → reporter → finalizer만 재실행. 갱신된 최종 설명·판단요약 확인                                                              | 인간-AI 협업(HITL) |
| 5   | NORMAL_BASELINE 전표로 동일 흐름 반복         | HITL 없이 자동 완료. 낮은 점수(LOW severity)·"이상 없음" 수준의 판단요약 확인                                                                                                                      | 정상 vs 위반 대비    |


- **입력 예시**

시연 시에는 **휴일 사용 의심(HOLIDAY_USAGE)** 전표를 주로 사용: 결제 시각 주말/심야, hrStatus=LEAVE, mccCode=5813, 적요 "휴일 야간 식대"
이어서 **정상 비교군(NORMAL_BASELINE)** 전표: 평일 낮, hrStatus=WORK, mccCode=5816, 이상 없음

- **기대 출력**

**HOLIDAY_USAGE:** 규정 조항(예: 제11조 업무시간 외 결제·휴일 사용 관련 조항) 인용, score_breakdown(isHoliday+35, hrStatus+20, isNight+10, merchantRisk HIGH+20, policy_refs 3건+22 …), 심각도 HIGH/CRITICAL, "휴일/심야 사용 위반 — 환수 또는 소명 권고" 수준의 판단요약. HITL 발생 시 reasons·questions·handoff 표시, 응답 제출 후 갱신된 최종 판단요약.

**NORMAL_BASELINE:** score LOW, HITL 미발생, "정상 처리 범위 — 특이사항 없음" 수준의 판단요약. 빠른 자동 완료로 AI의 효율성 어필.

- **성공 기준**
  - **기준 1:** AI가 위반 근거를 **규정 조항(citation)과 도구 결과(score_breakdown)**로 제시하고, 논리적인 후속 조치(환수/소명 등)를 권고하는가?
  - **기준 2:** 사용자가 실시간 이벤트(SSE)와 최종 근거를 신뢰하고, **HITL 참여 후 재분석 결과**까지 한 흐름으로 확인 가능한가?
  - **기준 3:** NORMAL_BASELINE 전표에서 HITL 없이 자동 완료되어, 위험 전표 대비 처리 흐름 차이가 명확하게 드러나는가?

**시나리오 4: [RAG 규정집 인덱싱 및 청킹 품질 검증 — 청킹 실험실]**

- **ID:** SC-004
- **상황:** 관리자 또는 개발자가 사내 경비 지출 관리 규정 TXT를 업로드하고, **청킹 전략(계층/조항/슬라이딩)**별 미리보기 및 품질 리포트를 확인한 뒤 최적 전략을 선택하여 pgvector에 인덱싱한다.
- **목표:** RAG 규정집 인덱싱의 전 과정(업로드→청킹 전략 선택→임베딩→저장→검색 품질 확인)을 한 화면에서 시연하고, 청킹 전략이 규정 조항 검색 품질에 미치는 영향을 체감한다.
- **상세 흐름**


| 단계  | 사용자 행동                  | 시스템 동작                                                                                          | 비고          |
| --- | ----------------------- | ----------------------------------------------------------------------------------------------- | ----------- |
| 1   | 규정문서 라이브러리에서 규정 TXT 업로드 | `chunking_pipeline.py`가 계층/조항/슬라이딩 전략으로 청크 분할 미리보기 생성                                           | 전략별 미리보기    |
| 2   | 청킹 전략 선택 후 인덱싱 실행       | OpenAI 호환 임베딩(text-embedding-3-large, 3072차원, halfvec) 생성 후 PostgreSQL `rag_chunk` 테이블에 저장      | pgvector 저장 |
| 3   | 품질 리포트 탭에서 검색 품질 확인     | BM25·Dense·RRF·Rerank 방식별 검색 결과 및 `retrieval_quality.py` 기반 Faithfulness·Answer Relevance 지표 확인 | 검색 품질 평가    |


- **성공 기준**
  - **기준 1:** 규정 TXT가 조항 단위로 의미 있게 청킹되고, pgvector에 임베딩이 저장되는가?
  - **기준 2:** BM25+Dense+RRF 하이브리드 검색이 단일 시맨틱 검색 대비 관련 조항 검색 재현율(Recall)을 향상시키는가?

**시연 권장 순서:**

1. **시연 데이터 생성** → HOLIDAY_USAGE + NORMAL_BASELINE 전표 각 1건 생성 (SC-001 — 하이브리드 스크리닝 결과 즉시 확인)
2. **HOLIDAY_USAGE 전표 분석 실행** → 실시간 사고 흐름·도구 호출·규정 조항 인용·HITL 참여 (SC-002 메인)
3. **NORMAL_BASELINE 전표 분석 실행** → HITL 없이 자동 완료·낮은 점수 확인 (SC-002 대비 효과)
4. (선택) SC-003/SC-004 시연 — RAG 청킹 실험실 또는 멀티모달 OCR 데모

**2개 케이스(HOLIDAY_USAGE + NORMAL_BASELINE)로 충분한 이유:**

- **극명한 대비**: 고위험 전표(HITL 발생·복잡한 추론)와 정상 전표(자동 완료·낮은 점수)의 대비가 AI 판단력을 가장 명확하게 보여줌
- **흐름의 완결성**: 스크리닝 → 심층 분석 → HITL → 재분석 → 최종 확정의 전 과정이 2케이스로 커버됨
- **시간 효율**: 10분 발표 기준으로 핵심 어필에 집중 가능. 단, 평가자 질문 시 LIMIT_EXCEED 등 나머지 유형도 지원함을 언급하여 시스템 완결성 어필
