# 프로젝트 개요

## 프로젝트명

**Aura Agentic AI 기반 법인카드 전표 리스크 탐지·감사 워크벤치(PoC)**  
대상 업무: **법인카드 전표 점검(휴일/심야 사용, 한도 초과, 업종 위험, 증빙-전표 불일치 검증)**

## 기술 스택

- Frontend: Streamlit (AI 워크스페이스, 시연데이터 생성/제어, 규정문서 라이브러리)
- Backend: FastAPI (분석 실행, SSE 스트림, HITL 제출/재개, 이벤트 조회)
- Agent: LangGraph + LangChain (StateGraph, conditional_edges, interrupt/resume)
- DB: PostgreSQL + pgvector (case/run/event/result + RAG chunk)
- RAG: BM25 + Dense + RRF + Rerank (+ LLM fallback rerank)
- OCR/멀티모달: evidence extraction + 비교 파이프라인 (해시 캐시, 신뢰도 기반 차단)
- 관측/옵션: SSE runtime, Langfuse(선택), Neo4j(선택)

## 주요 기능

- Phase 0 하이브리드 스크리닝: 규칙 점수 + LLM 분류 + 가드레일 보정으로 5개 케이스 자동 분류
- Phase 1 Agent 분석: planner-execute-critic-verify-reporter-finalizer 파이프라인 실행
- RAG 근거 제시: 조항 단위 인용 + adoption_reason으로 판단 근거 설명 가능성 확보
- HITL 협업: interrupt 후 같은 run_id로 재개하여 최종 확정(승인/보류)
- 증빙 교차검증: 영수증 추출값과 전표값 비교 후 불일치 시 fail-closed로 검토 강제
- 감사 추적성: SSE 이벤트와 DB 영속 로그를 통한 재현 가능한 증적 관리

## 기대 효과

- 고위험 전표 우선 선별로 재무 감사 리드타임 단축
- 규정 인용 기반 판단으로 설명 가능성·감사 대응력 강화
- 완전 자동이 아닌 HITL 기반 안전한 Agentic 업무 자동화 구현

# 서비스 아키텍처

## 클라이언트(UI)

- AI 워크스페이스: 케이스 선택, 분석 실행, 실시간 스트림, HITL 입력, 최종 결과/근거 확인
- 시연데이터 생성/제어: 시나리오별 테스트 전표 생성/삭제, 분석 재현 데이터 관리
- 규정문서 라이브러리: 규정 업로드, 청킹 전략 실험, RAG 인덱싱/품질 확인

## API/서버

- FastAPI 기반 단일 API 서버
- 주요 엔드포인트
  - 분석 시작: `/api/v1/cases/{voucher_key}/analysis-runs`
  - 스트림: `/api/v1/analysis-runs/{run_id}/stream`
  - 이벤트 조회: `/api/v1/analysis-runs/{run_id}/events`
  - HITL 제출/재개: `/api/v1/analysis-runs/{run_id}/hitl`, `/review-submit`

## Agent 런타임

- LangGraph StateGraph
  - `start_router -> (screener|intake) -> planner -> execute -> critic -> verify -> (hitl_pause/hitl_validate) -> reporter -> finalizer`
- Tool 기반 증거 수집
  - `holiday_compliance_probe`, `budget_risk_probe`, `merchant_risk_probe`, `document_evidence_probe`, `policy_rulebook_probe`, `legacy_aura_deep_audit(선택)`

## 데이터/스토리지

- PostgreSQL
  - 케이스/런/이벤트/최종결과 영속화
- pgvector
  - 규정 청크 임베딩 저장 및 하이브리드 검색
- 증빙 저장소
  - 업로드 파일 + 추출/비교 결과 캐시

## 실서비스 전환 가정(명시)

PoC 기준 연동 가정은 **배치 적재 경로**입니다.  
즉, SAP 전표를 실시간 API/JCo로 직접 조회하지 않고, DB 적재된 전표를 기준으로 Agent 분석을 수행합니다.

## 핵심 데이터 흐름

1. 전표 준비(시연데이터 생성 또는 DB 적재)
2. Phase 0 스크리닝(case_type/severity 저장)
3. 분석 run 생성 및 LangGraph 실행
4. planner에서 조사 계획 수립, execute에서 도구 호출
5. critic/verify에서 근거 점검 및 게이트 판정
6. HITL 필요 시 interrupt, 사용자 입력 후 same-run resume
7. reporter/finalizer로 최종 상태 확정 및 DB 영속화
8. UI에서 SSE 타임라인/근거/결과 조회

# 핵심 기능 및 Agent 구현 내용

## 1) Phase 0 하이브리드 스크리닝

- 입력 신호: `isHoliday`, `hrStatus`, `occurredAt`, `mccCode`, `budgetExceeded`, `amount`
- 분류 방식: rule score + LLM 제안 결합
- 가드레일
  - `llm_confidence < 0.75` 또는 규칙 신호 충돌 시 rule 우선
  - 과탐 방지를 위한 NORMAL_BASELINE 보호 규칙 적용

## 2) LangGraph 기반 오케스트레이션

- 단순 체인이 아닌 **상태 기반 분기·루프·재개** 구조
- `start_router`로 사전 스크리닝 여부에 따라 경로 최적화
- `critic` 루프(최대 2회)와 verify 게이트를 분리해 안전성 강화
- `hitl_pause -> hitl_validate`로 사람-에이전트 협업 사이클 구현

## 3) Tool-First 증거 수집 + 계약 기반 출력

- 모든 도구를 `ToolContextInput -> ToolResultEnvelope` 계약으로 고정
- planner가 케이스 유형별 실행 도구를 동적으로 구성
- 병렬 실행 가능한 도구는 병렬화(`ENABLE_PARALLEL_TOOL_EXECUTION`)하여 지연 완화

## 4) RAG 근거 인용 파이프라인

- 규정 텍스트를 계층/조항/슬라이딩 청킹 후 pgvector 저장
- BM25 + Dense + RRF + Rerank로 검색 정확도 개선
- 조항 단위 중복 제거 + reference normalizer + adoption_reason 기록
- 결과적으로 근거 없는 결론 대신 조항 기반 판단 문장 생성

## 5) Verify/HITL Fail-Closed 안전장치

- 자동 확정 차단 조건
  - `no_citation`, `contradictory_evidence`, coverage 미달, 필수 입력 누락
- HITL 요청 생성
  - 질문/필수입력은 규정 문맥 기반으로 동적 생성
  - UI 과밀 방지: 질문/필수입력 최대 2개 제한
- 최근 개선
  - HOLIDAY_USAGE 비출장 케이스에서 출장 조항/질문 오매칭 배제 필터 강화

## 6) 멀티모달 증빙 비교

- 증빙 업로드 후 추출값(금액/일자/시간/가맹점)과 전표값 비교
- 불일치 시 REVIEW_REQUIRED로 보수 처리
- 증빙이 있는 review-submit은 same-run 재개 경로를 유지하면서도 증빙 비교 결과를 최종 판정에 반영하도록 보강

## 7) 관측 가능성(Observability)

- SSE 이벤트: `NODE_START/END`, `TOOL_CALL/RESULT`, `SCREENING_RESULT`, `HITL_REQUESTED`, `RUN_COMPLETED`
- 동일 이벤트를 DB에 저장해 사후 감사 추적 가능
- run_id 단위로 재현 가능한 실행 증적 확보

## KPI-1. E2E 재현성(증거 기반)

- 목표: 대표 시나리오 end-to-end 100% 재현

## KPI-2. HITL/게이트 안정성(same-run + fail-closed)

- 목표: HITL 재개 same-run 100%, 근거 부족 자동확정 0건
- 결과
  - same-run resume: **100% (1/1)**
  - fail-closed: blocking taxonomy 발견 시 자동확정 금지 규칙 적용
- 근거: hitl API 응답(`source_run_id == resumed_run_id`) + verify 게이트 규칙

- Faithfulness: **0.72 -> 0.89**
- Answer Relevance: **0.81 -> 0.92**

# 기술 문제 해결 과정(발표용 핵심 3건)

## 1) HITL 재개 불안정

- 문제: memory 체크포인터 환경에서 재시작/멀티워커 시 재개 불안정
- 해결: `CHECKPOINTER_BACKEND=postgres` 경로 표준화 + same-run resume 보장
- 효과: interrupt 지점부터 재개 가능, 운영 재현성/감사성 강화

## 2) 근거 부족/모순 시 오판 위험

- 문제: 규정 인용 부족 또는 증빙 모순 상황에서 자동 완료 위험
- 해결: verify 단계 fail-closed 정책(`no_citation`, `contradictory_evidence`)으로 HITL 강제
- 효과: 보수적 판정 체계 확립, 감사 리스크 감소

## 3) 규정 매칭 오탐(휴일 케이스에 출장 질문)

- 문제: HOLIDAY_USAGE에서 출장 조항(제25조) 유입으로 부적절 질문 생성
- 해결: 정책 정렬 필터 + HITL 질문 필터를 이중 적용
- 효과: 케이스-규정 정합성 향상, 리뷰 질문 품질 개선

# Agentic AI 관점 차별화 포인트(평가 어필)

1. **Stateful Agent Orchestration**  
단순 프롬프트 체인이 아니라 상태(State) 기반 분기/루프/재개를 구현한 LangGraph 오케스트레이션을 실서비스 형태로 적용.

2. **Human-in-the-Loop as First-Class Node**  
HITL을 UI 보조 기능이 아니라 그래프 노드(`hitl_pause/hitl_validate`)로 설계해, 사람 개입을 런타임 제어 흐름에 내재화.

3. **Fail-Closed Governance**  
근거 부족/증거 모순 시 자동완료를 금지하고 담당자 검토로 전환하는 안전 우선 설계. Agentic 자율성과 통제 가능성을 동시에 달성.

4. **Tool-Augmented Reasoning**  
Planner가 도구를 선택하고 Execute가 증거를 수집, Critic/Verify가 교차검증하는 다단계 추론 파이프라인으로 “생성형 답변”이 아닌 “증거 기반 판단”을 수행.

5. **Retrieval + Citation Discipline**  
RAG 검색 결과를 조항 단위로 정규화/중복제거하고 adoption_reason까지 남겨, LLM 출력을 감사 가능한 citation 중심 결과로 변환.

6. **Observable Agent Runtime**  
SSE + DB 이중 기록으로 run 단위 실행 과정을 재현 가능하게 만들어, 평가/감사/디버깅 모두에 활용 가능한 운영형 에이전트 패턴을 구현.

# 결론

본 PoC는 Agentic AI를 단순 대화형 기능이 아닌, **실행 제어 가능한 감사 워크플로우**로 구현했습니다.  
핵심 성과는 다음 3가지입니다.

- 상태 기반 오케스트레이션 + HITL 재개까지 포함한 운영형 Agent 구조 구현
- 규정 인용/증빙 비교/fail-closed 정책으로 신뢰 가능한 판단 체계 확보
- SSE/DB 증적 중심의 재현 가능한 E2E 데모 완성

다음 단계는 KPI 확장(MRR/nDCG, p95 latency, 토큰/비용)과 운영 데이터셋 기반 정량 검증 자동화입니다.
