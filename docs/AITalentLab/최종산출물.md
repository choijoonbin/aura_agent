# 프로젝트 개요

## 프로젝트명

Aura Agentic AI 기반 실시간 재무 리스크 탐지·감사 워크벤치(POC). SAP 전표를 스크리닝/심층분석하여 규정 위반 근거(citation)와 조치 가이드를 제공하고, 필요 시 HITL로 최종 확정합니다.

## 기술 스택

* Frontend: Streamlit (AI 워크스페이스, 시연 데이터 생성, RAG 청킹 실험실 등)

* Backend: FastAPI (분석 실행/이벤트 조회/HITL 제출 API)

* Agent: LangGraph + LangChain (planner/execute/critic/verify/reporter/finalizer, interrupt/resume)

* DB: PostgreSQL + pgvector (rag_chunk, analysis_run/event, agent_case 등)

* RAG 검색: BM25 + Dense + RRF + Rerank, (옵션) HyDE

* OCR(증빙): pytesseract (미설치 시 데모 stub), (확장) Azure Document Intelligence

* 옵션: Neo4j(관계 탐색), Langfuse(관측/추적)

## 주요 기능

* 하이브리드 스크리닝(Phase 0): 규칙 점수 + LLM 분류(가드레일)로 5종 케이스 분류(HOLIDAY_USAGE/LIMIT_EXCEED/PRIVATE_USE_RISK/UNUSUAL_PATTERN/NORMAL_BASELINE)

* 심층 분석(Phase 1): LangGraph 워크플로로 도구(probe) 계획·실행·비평·검증 후 리포트 생성

* RAG 기반 근거 제시: 사내 규정 조항을 검색·채택(adoption_reason 포함)하여 판단 근거(citation)로 제공

* HITL 승인/보류: interrupt/resume + 체크포인터(Postgres)로 동일 run에서 재개, 최종 상태 확정

* 관측 가능성: SSE로 NODE_START/END, TOOL_CALL/RESULT, HITL_REQUESTED 등 이벤트 스트리밍 + DB 감사 로그 적재

* 증빙(OCR) 기반 교차검증: 영수증 이미지에서 엔티티 추출 후 전표 필드와 불일치 탐지

## 기대 효과

* 대량 전표 사후 점검의 리드타임/인력 소모 감소(고위험 케이스 우선순위화)

* 규정 조항·점수·도구 결과를 근거로 한 설명 가능한 판단(감사 추적성/거버넌스 강화)

* HITL 기반의 안전한 자동화(무인 처리 대신, 케이스 위험도에 따른 인간 개입)


# 서비스 아키텍처

## 클라이언트(UI)

* Streamlit AI 워크스페이스 — 전표/케이스 선택, 분석 실행, SSE 타임라인·근거·결과 확인, HITL 응답 입력

* Streamlit 시연 데이터 생성/관리 — 시나리오별 전표 생성(HOLIDAY_USAGE/NORMAL_BASELINE 등)

* Streamlit 규정 문서 라이브러리 — 규정 TXT 업로드, 청킹 전략(계층/조항/슬라이딩) 미리보기 및 인덱싱

## API/서버

* FastAPI — 분석 실행(run 생성), SSE 스트림 제공, 이벤트 조회, HITL 제출, (옵션) 스크리닝 API 제공

## Agent 런타임

* LangGraph StateGraph — start_router→(screener/intake)→planner→execute→critic→verify→(hitl_pause/hitl_validate)→reporter→finalizer

* LangChain Tools(Probe) — DB/RAG/OCR 등을 조회하여 facts/summary를 표준 봉투(ToolResultEnvelope)로 반환

## 데이터/스토리지

* PostgreSQL(dwp_aura) — 전표/케이스, run 메타/이벤트, HITL, 결과 영속화(감사 추적)

* pgvector(rag_chunk) — 규정 청크 임베딩 저장(halfvec), 하이브리드 검색(BM25/Dense/RRF/Rerank) 지원

* 증빙 업로드 저장 — 영수증 이미지 및 추출 엔티티(비교/검증용)

## 옵션 컴포넌트

* Neo4j(선택) — 케이스 간 관계 탐색/설명 경로 시각화(ENABLE_GRAPH_DB=true)

* Langfuse(선택) — 실행 추적/관측 대시보드(LANGFUSE_ENABLED=true)

## 핵심 데이터 흐름

1. (시연/운영) 전표 데이터 준비: 시연 데이터 생성 또는 DB 적재(SAP 연동은 향후)

2. Phase 0 스크리닝: 전표 필드 기반 규칙 점수 + LLM 분류(가드레일)로 case_type/severity 산출 및 저장

3. 분석 run 생성: UI/Api에서 run 생성 → LangGraph 실행 시작

4. intake: 전표 정규화 + 플래그/증빙(OCR) 결과 반영 → AgentState 갱신

5. planner: 조사 계획(도구 목록/순서/병렬 그룹) 생성(Pydantic 구조화 출력)

6. execute: Probe 도구 호출(규정 RAG/증빙/휴일/예산/가맹점 등) → ToolResultEnvelope로 수집

7. critic/verify: 과잉 주장/모순 점검, 하이브리드 스코어링+게이트 적용 → 자동 완료 또는 HITL 요청

8. HITL 발생 시 interrupt: 사용자 입력 제출 → 동일 run resume → reporter/finalizer만 재실행

9. 결과 제공/저장: SSE로 타임라인 스트리밍 + DB에 이벤트/최종 결과 영속화 → UI에서 근거/점수/조치 확인


# 핵심 기능 및 Agent 구현 내용

### 하이브리드 스크리닝(Phase 0)

전표 필드(isHoliday, hrStatus, occurredAt, mccCode, budgetExceeded, amount)에서 신호를 추출해 rule_score를 만들고, LLM 분류 결과(confidence)와 가드레일로 정렬하여 5종 케이스 유형을 결정합니다.

* 결정 로직 — LLM confidence < 0.75 또는 결정론 신호와 충돌 시 rule 기반 결과를 우선 채택

* 결과 저장 — case_type, screening_reason, screening_mode(hybrid/rule), llm_confidence 등을 케이스에 기록

### LangGraph 기반 심층 분석(Phase 1) 워크플로

start_router→(screener/intake)→planner→execute→critic→verify→(hitl_pause/hitl_validate)→reporter→finalizer 순서로 실행되며, SSE로 단계별 이벤트를 스트리밍합니다.

* interrupt/resume — HITL 필요 시 interrupt 후 동일 run에서 resume(체크포인터 backend=postgres 권장)

* 구조화 출력 — Planner/Reporter/FinalResult를 Pydantic 모델로 고정하여 UI/로그/저장 형식 일관성 확보

### Probe 도구 기반 증거 수집(Execute)

planner가 케이스 유형/플래그에 따라 도구 실행 계획을 만들고, execute가 LangChain StructuredTool을 순차/병렬로 호출하여 facts/summary를 수집합니다.

* holiday_compliance_probe — 휴일/근태(hrStatus) 충돌 및 휴일 리스크 검증

* budget_risk_probe — 예산 초과/금액 구간 기반 리스크 점수 산정

* merchant_risk_probe — MCC 위험도(예: 5813) 기반 가맹점/업종 리스크(휴일 결과와 결합)

* document_evidence_probe — 전표 라인아이템/증빙 수집 및 누락 점검

* policy_rulebook_probe — 규정 조항 RAG 검색 후 채택 조항/채택 사유(adoption_reason) 반환

* legacy_aura_deep_audit(선택) — 기존 심층 파이프라인 호출(충분한 근거 확보 시 자동 스킵 가능)

### RAG(규정 근거 검색)

규정 TXT를 청킹(계층/조항/슬라이딩) 후 pgvector에 저장하고, BM25+Dens+RRF+Rerank로 하이브리드 검색하여 조항 단위 근거를 제공합니다.

* 중복 제거/우선 채택 — 조항(article) 단위로 중복 제거, 공통 증빙 의무 조항(예: 제14조) 우선 채택 로직 포함

* 정규화 — policy reference 포맷을 정규화하여 환각/오표기 방지

### Hybrid Scoring & Verify/HITL 게이트

rule_score와 LLM judge 점수(llm_score)를 가중 합산해 final_score 및 severity(LOW~CRITICAL)를 산출하고, 증거 커버리지/모순/무근거를 게이트로 처리합니다.

* 예: final_score ≥ 75 → CRITICAL

* 예: abs(rule_score-llm_score) ≥ 20 또는 fidelity < 40이면 재평가 우선(조건 충족 시) → 이후 HITL 여부 결정

* Fail-closed HITL — no_citation/contradictory_evidence/필수 입력 누락 등은 자동 확정 금지 후 HITL 요청

### SSE 기반 관측 가능성(Observability)

NODE_START/END, TOOL_CALL/RESULT, SCREENING_RESULT, HITL_REQUESTED, RUN_COMPLETED 등을 구조화 이벤트로 스트리밍하고, 동일 이벤트를 DB에도 적재하여 재현 가능한 감사 근거를 제공합니다.

### 멀티모달 증빙(OCR) 추출 및 대조

영수증 이미지에서 금액/날짜/가맹점명 등을 추출해 전표 필드와 대조하고, 불일치(예: 금액 오차 5% 초과)는 contradictory_evidence로 처리하여 안전하게 HITL로 승격합니다.

* pytesseract 미설치 환경 — 파일 해시 기반 데모 stub로 동작(시연/테스트 목적)

[작성 가이드] 최종 발표용으로는 실제 대표 시나리오 1개(HOLIDAY_USAGE)의 입력(JSON)→주요 SSE 이벤트 5줄→최종 결과(JSON) 1개를 이 문서 말미에 부록 형태로 붙이면 구현 완성도가 더 잘 드러납니다.



# KPI 달성도

## 달성 항목

* E2E 재현 가능성(단일 시나리오) — 목표: 1개 시나리오를 처음부터 끝까지 재현 → 결과: HOLIDAY_USAGE 시나리오 run_id 기준 SSE/DB/최종 리포트까지 증거 확보 (100%)

* RAG 응답 품질(주관 지표) — 목표: Faithfulness/Answer Relevance 개선 → 결과: Faithfulness 72%→89%, Answer Relevance 81%→92% (목표 수치 미정)

* 자동화 테스트 — 목표: 회귀 방지 테스트 구축 → 결과: pytest 기준 184개 테스트 통과 (목표 수치 미정)

## 미달성 항목

* 리스크 탐지 정확도(정량) — 목표: 95% → 결과: 89% (93.7%)

원인: PoC 단계에서 실제 운영 데이터/라벨셋 규모가 제한되어 도메인 커버리지와 튜닝(규정/패턴/모델) 여지가 남아 있음.

* Retrieval 정량 지표(MRR/nDCG/Recall) — 목표: before/after 수치 제시 → 결과: 문서에 실행 커맨드와 데이터셋(N=5)만 명시, 실제 측정값 미기재 (0%)

원인: evaluate 스크립트 실행 결과를 리포트에 확정 수치로 반영하기 전 상태로 보임(수치 캡처/로그 첨부 필요).

* 성능/비용(지연, 토큰) 정량 — 목표: p50/p95 latency 및 run당 비용 측정 → 결과: 측정 항목/방법은 정의, 실제 수치 미기재 (0%)

원인: 대표 시나리오별 측정(run 수 N)과 기준선(before) 정의가 추가로 필요.

[작성 가이드] KPI는 최종 발표에서 3개만 강하게 가져가도 충분합니다. 추천 조합: (1) E2E 재현성(증거 기반) (2) HITL/게이트 안정성(강제 HITL 비율, fail-closed 0건 누락) (3) 성능(p95 latency) 또는 비용(run당 토큰).



# 기술 문제 해결 과정

### HITL 이후 resume가 불안정한 문제(체크포인터)

1. 문제 상황 — HITL 입력 후 분석을 재개(resume)할 때, 환경에 따라 전체 노드가 재실행되거나 정확한 재개 지점이 보장되지 않음

2. 원인 — CHECKPOINTER_BACKEND=memory 기본값은 단일 프로세스/단일 실행 컨텍스트에만 적합하여, 멀티 워커/재시작 상황에서 상태 복원이 어려움

3. 해결 방법 — CHECKPOINTER_BACKEND=postgres + langgraph-checkpoint-postgres로 체크포인트를 DB에 저장하고, HITL 직후 상태에서 reporter/finalizer만 재실행되도록 워크플로를 설계

4. 결과 — 동일 run_id로 interrupt/resume이 가능해지고, 감사 추적과 재현성이 강화됨

### LLM 분류 결과와 규칙 신호 충돌/불확실성 처리(가드레일)

1. 문제 상황 — LLM이 케이스 유형을 제안했지만 전표 필드 기반 신호와 충돌하거나 confidence가 낮아 오탐/과탐 위험이 존재

2. 원인 — LLM 단독 분류는 입력 노이즈/도메인 편향에 취약하며, 규정 기반 도메인에서는 '설명 가능성'과 '보수적 처리'가 필수

3. 해결 방법 — 결정론 스코어(rule_score)와 LLM confidence 임계값(예: 0.75)을 사용해 하이브리드 정렬(hybrid_case_align) 적용, 불확실 시 rule로 fallback

4. 결과 — 스크리닝의 일관성 향상, 애매한 케이스는 심층 분석/검증 단계에서 추가 근거 수집 또는 HITL로 안전하게 승격

### 근거 부족/모순(no_citation, contradictory_evidence) 시 안전한 처리(fail-closed)

1. 문제 상황 — 규정 조항 인용이 없거나, 도구/OCR 결과가 서로 모순될 때 LLM이 임의 결론을 낼 위험(환각) 존재

2. 원인 — RAG 누락/검색 실패, OCR 신뢰도 부족, 입력 데이터 누락 등이 결합되면 '근거 기반' 판단이 어려움

3. 해결 방법 — no_citation/contradictory_evidence를 강제 HITL 트리거로 정의, OCR 엔티티는 최소 신뢰도(VISUAL_MIN_CONFIDENCE) 미만 시 비교 제외, 금액 불일치(>5%)는 차단

4. 결과 — 자동 확정 오판 리스크 감소, HITL 질문 수/필수 입력 수를 제한(MAX_HITL_QUESTIONS/INPUTS)해 운영 부담도 관리

### UI OCR bbox 하이라이트 좌표 어긋남 문제

1. 문제 상황 — 영수증 이미지 위 bbox 하이라이트가 화면 크기/렌더링에 따라 어긋나 사용자 신뢰를 저해

2. 원인 — 이미지 원본 좌표계와 UI 렌더링 좌표계 간 스케일 차이(픽셀 기준 좌표를 그대로 사용)

3. 해결 방법 — bbox를 normalized coordinates(상대좌표)로 변환해 저장/표시하고, UI에서 렌더링 크기에 맞춰 재스케일

4. 결과 — 다양한 화면/해상도에서 하이라이트 정합성 개선, 시연 품질 향상

### RAG 품질 한계(계층 문서 청킹/검색 개선)

1. 문제 상황 — 규정이 장/조/항 구조인데 조 단위 청킹만으로는 노이즈가 섞여 정확한 조항 인용이 어려움("89% ceiling" 이슈)

2. 원인 — 과도하게 큰 청크는 검색 precision을 낮추고, 너무 작은 청크는 상위 문맥을 잃어 해석 오류를 유발

3. 해결 방법 — 계층/조항/슬라이딩 청킹 전략을 실험 가능한 형태로 분리하고, Child 검색 후 Parent 문맥 제공(Parent Document Retrieval) 패턴을 고려; 하이브리드 검색(BM25+Dense+RRF+Rerank) 적용

4. 결과 — Faithfulness/Answer Relevance 지표가 개선되었고, 조항 단위 근거 제공 구조가 강화됨
