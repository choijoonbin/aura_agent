### Agent 페르소나 및 시스템 프롬프트 (Identity)

Agent의 정체성, 역할, 그리고 답변의 톤앤매너를 정의한다.

| 항목 | 정의 내용 |
| --- | --- |
| **Agent 이름** | Aura (Audit & Risk Analysis Agent) |
| **주요 역할** | 기업 경비 전표의 규정 준수 여부를 분석하고, 근거 기반으로 승인/보류 판단을 지원하는 지능형 감사 에이전트 |
| **핵심 목표** | 스크리닝 자동화, 심층분석 표준화, HITL 기반 최종 검증 가능성 확보 |
| **톤앤매너** | 객관적·보수적·근거 중심 설명. 확정 불가 항목은 단정 대신 "추가 검토 필요"로 표현 |
| **제약 사항** | 비근거 추론 금지, 규정 미인용 판단 금지, 증거 충돌 시 fail-closed(HITL 승격) 적용 |

### 워크플로우 및 오케스트레이션 (Workflow & Logic)

**2.1 처리 로직 (Processing Pipeline)**

- **Step 1 (Routing):** `start_router`에서 `body_evidence.case_type` 존재 여부로 분기
  - 존재: 사전 스크리닝 결과 사용, `intake`로 직행
  - 미존재: `screener` 실행 후 `intake`
- **Step 2 (Intake/Plan):** `intake`에서 신호 플래그 추출, `planner`에서 케이스별 도구 계획 수립
- **Step 3 (Execute/Critic):** `execute`에서 Tool 호출, `critic`에서 반례 및 과잉 주장 점검
- **Step 4 (Verify/HITL):** `verify`에서 점수/게이트 평가, 필요 시 `hitl_pause(interrupt)`
- **Step 5 (Resume/Finalize):** 응답 제출 시 `hitl_validate → reporter → finalizer`로 동일 run 재개

실제 그래프: `start_router → (screener|intake) → intake → planner → execute → critic → verify → (hitl_pause|reporter) → finalizer`

**2.2 상태 관리 (State Management)**

- **AgentState(LangGraph):** `body_evidence`, `flags`, `plan`, `tool_results`, `score_breakdown`, `verification`, `hitl_request`, `final_result`
- **런타임 저장:** `services/stream_runtime.py`로 run_id별 timeline/result 메모리 저장
- **영속 저장:** PostgreSQL의 전표/케이스/분석 이력 테이블 사용
- **체크포인트:** `CHECKPOINTER_BACKEND=postgres` 사용 시 interrupt-resume 안정성 향상

### 도구(Tools) 및 함수 명세 (Capability)

| 도구명 (Function Name) | 기능 설명 (Description) | 입력 파라미터 (Input Schema) | 출력 데이터 (Output) |
| --- | --- | --- | --- |
| `holiday_compliance_probe` | 휴일/심야/근태 충돌 확인 | `ToolContextInput(case_id, body_evidence, prior_tool_results)` | `isHoliday`, `holidayRisk`, 판정 사유 |
| `budget_risk_probe` | 예산 초과 여부 확인 | 동일 | `budgetExceeded`, 금액 지표 |
| `merchant_risk_probe` | 업종(MCC) 위험도 평가 | 동일 | `merchantRisk`, `compoundRiskFlags` |
| `document_evidence_probe` | 전표 라인 증거 수집 | 동일 | `lineItemCount`, `lineItems` |
| `policy_rulebook_probe` | 규정 조항 검색/채택 | 동일 | `policy_refs`, `ref_count`, `adoption_reason` |
| `legacy_aura_deep_audit` | 기존 Aura 브리지(옵션) | 동일 | legacy 분석 결과 또는 비활성 상태 |

공통 계약:
- 입력: `ToolContextInput` (Pydantic)
- 출력: `{tool, ok, facts, summary}` 형태를 기본으로 후속 노드에서 소비

### 지식 베이스 및 메모리 전략 (Context & Memory)

**4.1 RAG (검색 증강 생성) 전략**

- **소스:** `규정집/사내_경비_지출_관리_규정_v2.0_확장판.txt`
- **청킹:** 계층/조항/슬라이딩 전략 (`services/chunking_pipeline.py`)
- **저장소:** PostgreSQL `rag_chunk` + pgvector(halfvec)
- **검색:** BM25 + Dense + RRF + Rerank (`services/policy_service.py`)
- **정합성 보정:** 케이스-조항 필터로 문맥 불일치 조항 억제 (`services/policy_case_alignment.py`)

**4.2 대화 메모리 (Conversation History)**

- 본 PoC는 챗봇형 장기 메모리보다 **run 중심 상태 추적**에 초점
- HITL 응답/재개 기록은 run 단위로 조회 가능하며, UI에서 이전 제출값 재표시 지원
- 장기 운영 단계에서는 run 이력의 DB 영속화 범위를 확대하는 것이 다음 과제

### 핵심 에이전트 기술 스택

| 구분 | 선정 전략/기술 | 선정 사유 (논리적 근거) |
| --- | --- | --- |
| **LLM Model** | OpenAI 호환 모델(환경변수 기반) | 규정 추론, 구조화 출력, 질문 생성 품질 확보 |
| **Agent Framework** | LangGraph | 분기/루프/interrupt-resume 등 상태 기반 워크플로우에 적합 |
| **Tool Framework** | LangChain StructuredTool | 도구 I/O 계약 고정, Planner-Execute 연결 단순화 |
| **Prompt Strategy** | 노드별 역할 프롬프트 + 구조화 JSON 강제 | 노드 책임 분리, 출력 파싱 안정화 |
| **RAG Stack** | pgvector + Hybrid Retrieval | 규정 검색 재현율/정밀도 균형 |
| **Observability** | SSE 이벤트 + run timeline + 대시보드 | 노드/도구/게이트 실행 과정을 시연 가능하게 제공 |

### 개발 환경 구성 (실행 기준)

- **Backend:** `uvicorn main:app --reload --host 0.0.0.0 --port 8010`
- **Frontend:** `streamlit run app.py --server.port 8502`
- **DB:** PostgreSQL(`dwp_aura` 스키마 재사용)
- **필수 패키지:** `requirements.txt`
- **옵션 패키지:** 체크포인터/리랭커 전용 requirements 파일

정리: 본 설계는 "LLM 단독 응답"이 아니라 **오케스트레이션 + 도구 + RAG + HITL**을 결합한 Agentic AI 아키텍처를 실제 코드로 구현하는 데 목적이 있다.
