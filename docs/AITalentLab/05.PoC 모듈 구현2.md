### 핵심 구현 내용

이번 PoC의 핵심은 "분석 결과 문장"보다, 그 문장을 만들기 위한 **Agentic 실행 체계**를 실제로 구현한 점이다.

**1.1 에이전트 워크플로우 (Agent Workflow)**

- **구현 기능:** LangGraph 기반 상태 머신
  - `start_router → (screener|intake) → intake → planner → execute → critic → verify → (hitl_pause|reporter) → finalizer`
- **동작 원리:**
1. `start_router`가 사전 스크리닝 여부를 판정해 중복 연산을 방지
2. `planner`가 케이스별 도구 계획을 생성
3. `execute`가 계획된 Tool을 실행하고 결과를 누적
4. `critic/verify`가 근거 충실도와 게이트를 점검
5. `hitl_pause`는 `interrupt()`로 대기, 응답 시 `hitl_validate`를 거쳐 같은 run에서 확정
- **주요 기술:** LangGraph conditional edge, interrupt/resume, Pydantic 구조화 출력

**1.2 도구(Tool) 및 함수 연동**

- **구현 기능:** 6개 Probe 도구 기반 근거 수집
  - `holiday_compliance_probe`
  - `budget_risk_probe`
  - `merchant_risk_probe`
  - `document_evidence_probe`
  - `policy_rulebook_probe`
  - `legacy_aura_deep_audit`(옵션)
- **동작 원리:** planner에서 계획한 순서대로 execute가 호출하며, 각 도구는 공통 입력(`ToolContextInput`)을 사용
- **주요 기술:** LangChain StructuredTool, 도구 공통 스키마, 케이스별 도구 선택

**1.3 데이터 및 메모리 (RAG & Context)**

- **구현 기능:** 규정집 기반 RAG 검색 및 조항 채택
- **동작 원리:**
1. 규정 문서를 청킹/임베딩 후 `rag_chunk` 저장
2. 검색 시 BM25 + Dense + RRF + Rerank 적용
3. 조항 단위 후보를 `policy_refs`로 반환하고 채택 사유(`adoption_reason`) 제공
4. 케이스-조항 정합 필터로 오탐 조항을 억제
- **주요 기술:** PostgreSQL pgvector, hybrid retrieval, alignment filtering

### 핵심 동작 검증 (재현 가능 패키지)

#### 3.1 실행 환경

- Backend: `uvicorn main:app --reload --host 0.0.0.0 --port 8010`
- Frontend: `streamlit run app.py --server.port 8502`
- DB: PostgreSQL 연결 필요(.env)

#### 3.2 검증 시나리오 A: 생성 시점 스크리닝

- 입력: `시연데이터 생성(beta)`에서 `HOLIDAY_USAGE` 생성
- 기대 동작:
1. 생성 즉시 `agent_case`에 `case_type/screening_reason_text` 저장
2. 워크스페이스 목록에서 배지/사유 확인
- 확인 포인트: 분석 전에도 스크리닝 결과가 조회 가능

#### 3.3 검증 시나리오 B: 분석 + HITL + 재개

- 입력: 워크스페이스에서 전표 선택 후 `분석 실행`
- 기대 동작:
1. SSE에 `NODE_START/TOOL_CALL/TOOL_RESULT/SCREENING_RESULT` 표시
2. HITL 발생 시 질문/사유 노출
3. 응답 제출 후 같은 run에서 재개 및 최종 결과 갱신
- 확인 포인트: 재개 후 질문/이전 응답 이력이 UI에서 확인 가능

#### 3.4 검증 시나리오 C: 케이스-조항 정합성

- 입력: `LIMIT_EXCEED` 전표 분석
- 기대 동작:
1. 한도/승인 중심 조항 우선 채택
2. 맥락 불일치 조항은 필터링
- 확인 포인트: `policy_refs` 구성과 검토 질문의 맥락 일치

### Verify/HITL 기준 명시 (코드 기준)

- 점수는 `score_breakdown`으로 관리되며, verify 단계에서 HITL 필요 여부를 결정
- `required_inputs`는 "실제 누락값"만 남기도록 필터링
- `review_questions`는 최대 개수 제한 및 케이스 정합성 필터 적용
- `NORMAL_BASELINE`은 예외적으로 불필요한 HITL을 억제하는 보호 로직 보유

주의: 임계치 세부 수치는 운영 중 튜닝 대상이며, 본 문서에는 "구조와 판단 경로"를 중심으로 기록한다.

### Tool/Output 계약 명시

- 공통 입력: `ToolContextInput(case_id, body_evidence, intended_risk_type, prior_tool_results)`
- 공통 출력: `tool`, `ok`, `facts`, `summary`
- Reporter/Finalizer 소비 계약:
1. 최소 1개 이상의 근거 조항 또는 도구 결과
2. 근거 부족/충돌 시 보수적 판정 또는 HITL 승격
3. 최종 응답에 score/근거/판단요약 포함

정리: 이 PoC는 "설계 설명"을 넘어서, 실행 로그(SSE), run 상태, HITL 재개, 규정 채택 결과로 **구현 완료를 재현 가능하게 증명**할 수 있는 수준까지 구현되었다.
