## 1. 최종 아키텍처 요약

### 완성된 아키텍처 핵심

**FastAPI + Streamlit + LangGraph + PostgreSQL(pgvector) Python 단일 스택**으로 기존 React(Front) + Java(Backend) + Aura Platform 3계층 구조를 단일 코드베이스로 통합. 재무 전표 분석 전 과정(스크리닝 → 심층 분석 → HITL 협업 → 최종 확정)이 하나의 에이전트 런타임에서 실행됩니다.

### 최종 산출물 형태

**Streamlit 멀티페이지 AI 워크벤치** — 재무 실무자가 전표를 선택하고 AI의 실시간 사고 흐름을 관찰하며, 필요 시 HITL로 판단에 직접 참여하는 **지능형 감사 워크스테이션**.

### Agent 구조 (Phase 0 + Phase 1)

```
[사용자: 전표 선택 + 「분석 실행」]
         │
         ▼
[Phase 0: 하이브리드 스크리닝 — 전표 생성 시 자동 수행]
  LLM 초기 분류 (gpt-4o-mini, JSON 응답)
  + 결정론적 시그널 스코어링 (isHoliday, hrStatus, MCC, budgetExceeded 등)
  + 가드레일 정렬 (신뢰도 < 0.75 또는 시그널 충돌 → 결정론적 우선)
  → case_type 결정 (5종: HOLIDAY_USAGE / LIMIT_EXCEED / PRIVATE_USE_RISK
                        / UNUSUAL_PATTERN / NORMAL_BASELINE)
         │
         ▼
[Phase 1: LangGraph 심층 분석]

  start_router ──→ (사전 스크리닝 O) → intake
               └──→ (사전 스크리닝 X) → screener → intake

  intake (플래그 추출·OCR 반영)
    ↓
  planner (LLM 기반 조사 계획·도구 목록 결정 — PlannerOutput)
    ↓
  execute (LangChain StructuredTool 순차/병렬 호출)
    ├── holiday_compliance_probe
    ├── budget_risk_probe        ← 병렬 그룹
    ├── document_evidence_probe
    ├── merchant_risk_probe (holiday 결과 활용)
    └── policy_rulebook_probe (BM25+Dense+RRF+Rerank, prior 결과 강화)
         [legacy_aura_deep_audit — 선택, 충분한 근거 시 자동 스킵]
    ↓
  critic (과잉 주장·반례 검토, 최대 2회 루프, 일관성 검증)
    ↓
  verify (하이브리드 점수 산정 + Evidence Coverage Gate)
    ├─→ [HITL 필요] hitl_pause (interrupt) → [사용자 응답] → hitl_validate
    └─→ [HITL 불필요] reporter
    ↓
  reporter (근거 기반 판단요약·최종 설명 — ReporterOutput)
    ↓
  finalizer (최종 결과·점수·이력 확정 — FinalResult)

[결과: SSE 스트림 + DB 영속화 + Streamlit 타임라인/결과/근거 탭]
```

## 3. 창출된 핵심 가치

### 3-1. 비즈니스 가치

* **고위험 케이스 자동 선별:** AI가 전표 필드만으로 5종 케이스 유형을 자율 분류(하이브리드 스크리닝). 재무 실무자는 AI가 고위험으로 선별한 케이스만 집중 검토하면 되어, 단순 반복 검토 업무 대폭 감소.

* **규정 근거 기반 판단:** "왜 위반인가?"에 대해 내부 규정집 조항(citation)과 score_breakdown을 근거로 제시. 담당자 숙련도에 의존하지 않는 객관적·일관된 판단 기준 확립.

* **투명한 감사 증적:** 모든 AI 판단 근거·이벤트·HITL 이력이 DB와 런타임에 기록. run 이벤트 API·SSE 스트림·에이전트 스튜디오를 통해 감사 시 객관적 증빙 자료로 활용 가능.

* **하이브리드 거버넌스:** AI 자율 판단(NORMAL_BASELINE → 자동 완료)과 인간 검토(HITL 개입)를 케이스 위험도에 따라 자동 분기. "AI가 모든 것을 결정"이 아닌, AI가 선별·근거 제시 → 인간이 최종 확정하는 신뢰 기반 워크플로우.

### 3-2. 기술적 가치

* **Agentic AI 구현:** LangGraph StateGraph로 순환·분기·HITL interrupt/resume을 갖춘 실제 에이전트 오케스트레이션 구현. 단순 체인(Chain)이나 Q&A 봇이 아닌, **자율적으로 계획(planner)·실행(execute)·검토(critic)·검증(verify)·보고(reporter)**하는 Multi-Step Agent.

* **하이브리드 방식의 신뢰성:** LLM 단독의 환각 리스크를 결정론적 가드레일로 보정(하이브리드 스크리닝), 규칙 점수와 LLM judge 점수를 가중 합산(하이브리드 스코어링). AI 판단의 예측 가능성과 일관성을 동시에 확보.

* **실전 RAG 파이프라인:** 단순 벡터 검색을 넘어 BM25+Dense+RRF+Rerank 하이브리드 검색으로 규정 조항 검색 품질(Faithfulness 89%, Answer Relevance 92%) 달성. 청킹 실험실로 전략별 품질 비교 가능.

* **관찰 가능한 AI (Explainable AI):** SSE 기반 실시간 이벤트 스트리밍으로 노드별 사고 흐름 투명 공개. score_breakdown·citation·reasoning·`hybrid_case_align_reason` 등 판단 근거의 완전한 추적 가능.

* **Python 단일 스택 통합:** React+Java+Aura Platform 3계층을 FastAPI+Streamlit+LangGraph로 단일 코드베이스화. 개발·배포·유지보수 복잡도 대폭 감소. 기존 PostgreSQL(dwp_aura) 스키마 재사용으로 마이그레이션 없이 PoC 구축.

## 4. 운영 및 보안 고려 사항

| 항목 | PoC 현황 | 운영 전환 시 필요 조치 |
|------|----------|----------------------|
| **인증·권한** | 미적용 (tenant_id=1, user_id=1 고정) | OAuth/JWT 기반 사용자 인증, 역할별 접근 제어(재무팀/감사팀/관리자) |
| **입력 검증** | FastAPI Pydantic 유효성 검증, ToolContextInput 스키마 강제 | 추가 PII 마스킹, 입력 크기 제한 |
| **AI 가드레일** | 하이브리드 스크리닝 가드레일, fail-closed HITL(no_citation/contradictory_evidence), LLM judge circuit breaker | 프롬프트 인젝션 탐지 레이어 추가 |
| **데이터 보안** | PoC용 시연 데이터만 사용 | 실 전표 데이터 암호화, DB 접근 계정 분리(최소 권한) |
| **장애 대응** | LLM 호출 실패 시 Pydantic 기본값 fallback, fallback_used 기록 | LLM 서비스 SLA 모니터링, 다중 모델 fallback 체계(primary → fallback model) |
| **HITL 재개** | memory 체크포인터(단일 프로세스) | `CHECKPOINTER_BACKEND=postgres` + `langgraph-checkpoint-postgres`로 멀티 워커 환경에서도 HITL 재개 보장 |
| **관찰 가능성** | Langfuse 선택적 통합(`LANGFUSE_ENABLED`), SSE 이벤트·run 타임라인 DB 기록 | Langfuse 또는 LangSmith 전사 모니터링 도입 |

## 5. 회고 및 향후 확장

### 기술적 성과 및 한계

**성과:**
- LangGraph interrupt/resume 기반 HITL 흐름 구현 — 단일 run에서 사람 검토 후 재분석까지 이어지는 완결된 에이전트 워크플로우
- 하이브리드 스크리닝/스코어링으로 LLM 단독 대비 신뢰성 향상
- BM25+Dense+RRF+Rerank Hybrid Search로 RAG Faithfulness 72%→89% 달성
- 184개 자동화 테스트로 회귀 방지 체계 구축
- Python 단일 스택으로 기존 3계층 구조 대비 개발 속도·유지보수성 향상

**한계:**
- 탐지 정확도 89%(목표 95%) — 더 많은 실제 전표 데이터와 도메인 특화 파인튜닝 필요
- HITL 재개: `CHECKPOINTER_BACKEND=postgres` 미설정 시 멀티 워커 환경에서 전체 재실행
- SAP 실시간 인터페이스 미연동: PoC는 PostgreSQL 직접 접근, 운영은 SAP API/배치 연동 필요
- OCR 정밀도: pytesseract 기반으로 이미지 품질에 따라 추출 정확도 편차. Azure Document Intelligence 연동 필요
- 인증 미적용: PoC는 단일 테넌트 고정, 운영 전환 시 멀티테넌트 인증 체계 필요
