## 1. 최종 아키텍처 요약

### 완성된 아키텍처 핵심

**FastAPI + Streamlit + LangGraph + PostgreSQL(pgvector) Python 단일 스택**으로 기존 React(Front) + Java(Backend) + Aura Platform 3계층 구조를 단일 코드베이스로 통합. 재무 전표 분석 전 과정(스크리닝 → 심층 분석 → HITL 협업 → 최종 확정)이 하나의 에이전트 런타임에서 실행됩니다.

### 최종 산출물 형태

**Streamlit 멀티페이지 AI 워크벤치** — 재무 실무자가 전표를 선택하고 AI의 실시간 사고 흐름을 관찰하며, 필요 시 HITL로 판단에 직접 참여하는 **지능형 감사 워크스테이션**.

### Agent 구조 (Phase 0 + Phase 1)

```
[사용자: 전표 선택 + 「분석 실행」]
         │
         ▼
[Phase 0: 하이브리드 스크리닝 — 전표 생성 시 자동 수행]
  LLM 초기 분류 (gpt-4o-mini, JSON 응답)
  + 결정론적 시그널 스코어링 (isHoliday, hrStatus, MCC, budgetExceeded 등)
  + 가드레일 정렬 (신뢰도 < 0.75 또는 시그널 충돌 → 결정론적 우선)
  → case_type 결정 (5종: HOLIDAY_USAGE / LIMIT_EXCEED / PRIVATE_USE_RISK
                        / UNUSUAL_PATTERN / NORMAL_BASELINE)
         │
         ▼
[Phase 1: LangGraph 심층 분석]

  start_router ──→ (사전 스크리닝 O) → intake
               └──→ (사전 스크리닝 X) → screener → intake

  intake (플래그 추출·OCR 반영)
    ↓
  planner (LLM 기반 조사 계획·도구 목록 결정 — PlannerOutput)
    ↓
  execute (LangChain StructuredTool 순차/병렬 호출)
    ├── holiday_compliance_probe
    ├── budget_risk_probe        ← 병렬 그룹
    ├── document_evidence_probe
    ├── merchant_risk_probe (holiday 결과 활용)
    └── policy_rulebook_probe (BM25+Dense+RRF+Rerank, prior 결과 강화)
         [legacy_aura_deep_audit — 선택, 충분한 근거 시 자동 스킵]
    ↓
  critic (과잉 주장·반례 검토, 최대 2회 루프, 일관성 검증)
    ↓
  verify (하이브리드 점수 산정 + Evidence Coverage Gate)
    ├─→ [HITL 필요] hitl_pause (interrupt) → [사용자 응답] → hitl_validate
    └─→ [HITL 불필요] reporter
    ↓
  reporter (근거 기반 판단요약·최종 설명 — ReporterOutput)
    ↓
  finalizer (최종 결과·점수·이력 확정 — FinalResult)

[결과: SSE 스트림 + DB 영속화 + Streamlit 타임라인/결과/근거 탭]
```

### 4개 Streamlit UI 화면


| 화면             | 주요 기능                                                                                |
| -------------- | ------------------------------------------------------------------------------------ |
| **AI 워크스페이스**  | 전표 목록·상세 조회, 분석 실행, 실시간 사고 흐름(SSE 이벤트 타임라인), HITL 질문-응답, 최종 결과·근거·score_breakdown 확인 |
| **에이전트 스튜디오**  | run 이력 조회, 노드별 이벤트 타임라인, 오케스트레이션 그래프 설정 확인                                           |
| **규정문서 라이브러리** | 규정 TXT 업로드, 청킹 전략(계층/조항/슬라이딩) 실험실, 임베딩 저장, 검색 품질 리포트(Faithfulness·Answer Relevance)  |

## 2. 재현 가능한 실행 가이드 (가장 중요)

> 목표: **처음 보는 사람이 이 문서만으로 동일 결과를 재현**할 수 있도록, “표준 실행 경로(로컬)”를 1개로 고정하고 커맨드/환경변수/시드/데모 시나리오/기대 결과를 함께 제공합니다.

### 2-1. Repo 구조(핵심 엔트리)

- **FastAPI 엔트리**: `main.py`
- **Streamlit 엔트리**: `app.py`
- **환경변수 템플릿**: `.env.example` (→ `.env`로 복사 후 값 채우기)
- **Neo4j(선택)**: `docker-compose.neo4j.yml`

### 2-2. 사전 준비(필수 의존)

- **PostgreSQL**: 기존 `dwp_aura` 스키마 재사용(테이블: `dwp_aura.agent_activity_log`, `dwp_aura.case_analysis_run` 등)
- **Python**: `python3` 사용
- **OpenAI 호환 LLM 키**: 최소 스크리닝/플래너/리포터가 동작하도록 `OPENAI_API_KEY` 필요  
  - Azure OpenAI를 쓰는 경우: `OPENAI_BASE_URL`, `OPENAI_API_VERSION`을 함께 설정(프로젝트는 OpenAI 호환 클라이언트를 사용)

### 2-3. 필수 환경변수(최소 셋)

아래는 `.env.example` 기준으로 **데모 재현에 필요한 최소 환경변수**입니다.


| 구분          | 키                             | 예시                                                           | 필수  |
| ----------- | ----------------------------- | ------------------------------------------------------------ | --- |
| DB          | `DATABASE_URL`                | `postgresql://dwp_user:dwp_password@localhost:5432/dwp_aura` | ✅   |
| LLM         | `OPENAI_API_KEY`              | `...`                                                        | ✅   |
| API/UI      | `APP_PORT` / `STREAMLIT_PORT` | `8010` / `8502`                                              | ✅   |
| HITL 재개(표준) | `CHECKPOINTER_BACKEND`        | `memory`(기본) / `postgres`(권장)                                | ✅   |
| Neo4j(선택)   | `ENABLE_GRAPH_DB` + `NEO4J_`* | `true`, `bolt://localhost:7687`                              | ⛔   |


권장(재현성 강화를 위해 문서에서 “표준”으로 지정):

- **표준 경로(운영에 가까운 재현성)**: `CHECKPOINTER_BACKEND=postgres`
- **PoC 단일 프로세스 빠른 데모**: `CHECKPOINTER_BACKEND=memory`

### 2-4. 로컬 실행(표준 커맨드)

1. 환경변수 설정

- `.env.example` → `.env`로 복사
- `.env`에서 `DATABASE_URL`, `OPENAI_API_KEY`를 채움

1. 의존성 설치

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

1. (선택) Neo4j 기동

```bash

```

1. FastAPI 기동

```bash
uvicorn main:app --reload --host 0.0.0.0 --port 8010
```

1. Streamlit 기동(별도 터미널)

```bash
streamlit run app.py --server.port 8502
```

## 3. 데모 시나리오(1개) — 처음부터 끝까지 기대 결과

> 시나리오: **HOLIDAY_USAGE(휴일/휴무 중 사용 의심)** → 분석 수행 → **HITL 요청 발생** → 승인 응답 제출 → 같은 run으로 재개 → 최종 결과 확정

### 3-1. (옵션 A) UI로 실행(권장: 평가자 관찰용)

1. 브라우저에서 `http://localhost:8502` 접속  
2. 좌측 메뉴에서 **시연 데이터 제어** 진입 → `HOLIDAY_USAGE` 1건 생성  
3. **AI 워크스페이스**로 이동 → 생성된 전표 선택  
4. `분석 시작` 클릭  
5. **SSE 타임라인**에서 노드 진행 확인: `intake → planner → execute → critic → verify`  
6. `HITL` 팝업(또는 HITL 요청 카드) 노출 확인  
7. 검토 입력 예시
  - 승인: `approved=true`
  - 추가 사실: `approval_status=APPROVED`
  - 코멘트: “휴일 식사비는 프로젝트 비상 대응 회의로 발생. 사전승인 완료.”
8. `제출/재개` 클릭  
9. 같은 `run_id`로 `hitl_validate → reporter → finalizer`가 진행되고 **최종 결과(COMPLETED_AFTER_HITL)**가 표시됨

| **평가 지표 (KPI)**           | **목표(Plan)** | **실제(Actual)**   | **측정 방법(재현 가능)**                                                                      |

## 3. 창출된 핵심 가치

### 3-1. 비즈니스 가치

- **고위험 케이스 자동 선별:** AI가 전표 필드만으로 5종 케이스 유형을 자율 분류(하이브리드 스크리닝). 재무 실무자는 AI가 고위험으로 선별한 케이스만 집중 검토하면 되어, 단순 반복 검토 업무 대폭 감소.
- **규정 근거 기반 판단:** "왜 위반인가?"에 대해 내부 규정집 조항(citation)과 score_breakdown을 근거로 제시. 담당자 숙련도에 의존하지 않는 객관적·일관된 판단 기준 확립.
- **투명한 감사 증적:** 모든 AI 판단 근거·이벤트·HITL 이력이 DB와 런타임에 기록. run 이벤트 API·SSE 스트림·에이전트 스튜디오를 통해 감사 시 객관적 증빙 자료로 활용 가능.
- **하이브리드 거버넌스:** AI 자율 판단(NORMAL_BASELINE → 자동 완료)과 인간 검토(HITL 개입)를 케이스 위험도에 따라 자동 분기. "AI가 모든 것을 결정"이 아닌, AI가 선별·근거 제시 → 인간이 최종 확정하는 신뢰 기반 워크플로우.

### 3-2. 기술적 가치

- **Agentic AI 구현:** LangGraph StateGraph로 순환·분기·HITL interrupt/resume을 갖춘 실제 에이전트 오케스트레이션 구현. 단순 체인(Chain)이나 Q&A 봇이 아닌, **자율적으로 계획(planner)·실행(execute)·검토(critic)·검증(verify)·보고(reporter)**하는 Multi-Step Agent.
- **하이브리드 방식의 신뢰성:** LLM 단독의 환각 리스크를 결정론적 가드레일로 보정(하이브리드 스크리닝), 규칙 점수와 LLM judge 점수를 가중 합산(하이브리드 스코어링). AI 판단의 예측 가능성과 일관성을 동시에 확보.
- **실전 RAG 파이프라인:** 단순 벡터 검색을 넘어 BM25+Dense+RRF+Rerank 하이브리드 검색으로 규정 조항 검색 품질(Faithfulness 89%, Answer Relevance 92%) 달성. 청킹 실험실로 전략별 품질 비교 가능.
- **관찰 가능한 AI (Explainable AI):** SSE 기반 실시간 이벤트 스트리밍으로 노드별 사고 흐름 투명 공개. score_breakdown·citation·reasoning·`hybrid_case_align_reason` 등 판단 근거의 완전한 추적 가능.
- **Python 단일 스택 통합:** React+Java+Aura Platform 3계층을 FastAPI+Streamlit+LangGraph로 단일 코드베이스화. 개발·배포·유지보수 복잡도 대폭 감소. 기존 PostgreSQL(dwp_aura) 스키마 재사용으로 마이그레이션 없이 PoC 구축.

### 4-1. 운영/보안 체크리스트(“체크리스트 수준” 구체화)

- **인증/권한**
  - PoC: 인증 미적용(고정 `tenant_id=1`, `user_id=1`)을 문서에 명시
  - 운영: OAuth/JWT + RBAC(재무/감사/관리자), 테넌트 격리(tenant_id 기반 row-level 격리)
- **PII 처리**
  - 입력/증빙에서 PII 가능 필드 목록화(사번, 전화번호, 카드번호 등)
  - 로그/SSE에 PII가 그대로 노출되지 않도록 마스킹 정책 적용(운영 전)
  - 증빙 파일 저장소 암호화 및 접근 제어(운영 전)
- **에러 분류/재시도/타임아웃**
  - LLM: `SCREENING_LLM_TIMEOUT_SECONDS`, `LLM_JUDGE_TIMEOUT_MS`, `OPENAI_EMBEDDING_MAX_RETRIES` 등 기준을 설정/문서화
  - DB: connection timeout, 재시도, 실패 시 사용자 메시지(“재시도/문의”) 표준화
  - 외부 연동(향후): 타임아웃/서킷브레이커/큐잉 기준 수치화
- **감사 로그 보존**
  - `agent_activity_log` 보존 기간(예: 90일/1년) 및 삭제 정책 정의(운영 전)
  - run_id 단위 추적 가능(요청/도구/게이트/HITL/결과) 여부를 정기 점검

## 5-A. HITL 재개 이슈(멀티워커에서 전체 재실행) — PoC/운영 회피 1안

문서에 명시한 한계(멀티워커 + `CHECKPOINTER_BACKEND=memory`에서 재개 실패) 대응으로, **현 PoC 스코프에서 가장 현실적인 1안**을 표준으로 지정합니다.

- **PoC 운영 방침(1줄)**: 본 PoC 데모는 **단일 워커(단일 프로세스)로 고정 운영**하며, 멀티워커 환경이 필요할 경우 `CHECKPOINTER_BACKEND=postgres`를 표준으로 사용합니다.

- **표준 운영/데모 방안**: `CHECKPOINTER_BACKEND=postgres` 사용 + 단일 run_id 기반 재개 유지  
  - 장점: 멀티워커(또는 재시작)에서도 체크포인트로 `hitl_pause` 이후부터 재개 가능
  - 실행 지침: `.env`에서 `CHECKPOINTER_BACKEND=postgres`로 설정하고, 프로젝트에 포함된 체크포인터 설치 가이드를 따라 의존성을 추가 설치(예: `requirements-checkpoint-postgres.txt`)
