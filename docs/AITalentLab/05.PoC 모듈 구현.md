핵심 기능 PoC (Proof of Concept) 구현

### 핵심 구현 내용

이번 PoC 단계에서 실제 코드로 구현된 핵심 기능들을 **동작 원리**와 **사용 기술** 중심으로 상세히 기술합니다.

**1.1 에이전트 워크플로우 (Agent Workflow)**

* **구현 기능:** 전표 단위 분석 run에서 start_router → (screener 또는 intake) → intake → planner → execute → critic → verify → hitl_pause/reporter → finalizer 순의 LangGraph 워크플로우. 사전 스크리닝된 전표는 screener 노드를 건너뛰고 intake부터 진행.

* **동작 원리:** 사용자가 「분석 실행」을 누르면 FastAPI가 `body_evidence`를 LangGraph에 주입. **start_router**에서 `body_evidence.case_type` 존재 여부로 분기 — 있으면 screening_result를 상태에 넣고 **intake**로 직행, 없으면 **screener**에서 전표 원시 데이터만으로 케이스 유형 분류 후 intake로 진행. intake에서 플래그 추출, **planner**에서 케이스 유형별 조사 계획(plan) 및 도구 목록 결정, **execute**에서 LangChain 도구 순차/병렬 호출, **critic**에서 과잉 주장·반례 검토, **verify**에서 점수·HITL 필요 여부 판단. HITL 필요 시 **hitl_pause**에서 interrupt 후 담당자 응답 제출 시 같은 run으로 resume → hitl_validate → reporter → finalizer. **reporter**·**finalizer**에서 근거 기반 판단요약·최종 결과 생성.

* **주요 기술:** LangGraph(StateGraph, conditional_edges, interrupt), LangChain Core, AgentState(TypedDict), Pydantic 구조화 출력(planner_output, execute_output, reporter_output 등), SSE 스트리밍(NODE_START/END, TOOL_CALL/TOOL_RESULT, SCREENING_RESULT, HITL_REQUESTED). 체크포인터(memory 또는 postgres)로 HITL 재개 시 상태 복원.

**1.2 도구(Tool) 및 함수 연동**

* **구현 기능:** 전표·규정 기반 위험 검증을 위한 6개 Probe 도구. holiday_compliance_probe(휴일/휴무 정황), budget_risk_probe(예산 초과), merchant_risk_probe(MCC 위험도), document_evidence_probe(전표 라인·증거 수집), policy_rulebook_probe(규정집 조항 조회), legacy_aura_deep_audit(기존 Aura 심층 분석, 선택).

* **동작 원리:** planner가 케이스 유형(휴일/한도/업종 등)과 플래그에 따라 도구 목록(plan)을 생성. execute 노드에서 plan 순서대로 LangChain StructuredTool을 호출. 각 도구는 `ToolContextInput`(body_evidence, prior_tool_results 등)을 받아 DB·RAG·전표 데이터를 조회하고, facts·summary 형태로 결과 반환. policy_rulebook_probe·legacy_aura_deep_audit는 순차 실행, 나머지는 병렬 그룹 후 순차 실행 가능(ENABLE_PARALLEL_TOOL_EXECUTION). 도구 결과는 critic·verify·reporter에서 점수 산정·규정 인용·판단요약에 사용됨.

* **주요 기술:** LangChain StructuredTool, Pydantic ToolContextInput, `agent_tools.py` 내 비동기 handler 함수, `policy_service.search_policy_chunks()`(BM25/Dense/RRF/Rerank), MCC 위험 집합(utils.config get_mcc_sets).

**1.3 데이터 및 메모리 (RAG & Context)**

* **구현 기능:** 사내 경비 지출 규정 텍스트의 청킹·임베딩·저장 및 분석 시 규정 조항 검색(RAG). 규정문서 라이브러리(Streamlit)에서 TXT 업로드·전략별 청킹 미리보기·품질 리포트. run 단위 타임라인·최종 결과는 메모리(stream_runtime)에 저장.

* **동작 원리:** 규정 TXT는 `chunking_pipeline`으로 계층/조항/슬라이딩 전략에 따라 청크 분할 후 PostgreSQL `rag_chunk` 테이블에 저장. OpenAI 호환 임베딩(embedding_az, halfvec)으로 벡터 저장. 분석 시 `policy_rulebook_probe`가 케이스·키워드로 `policy_service.search_policy_chunks()`를 호출해 BM25·Dense(RRF)·Rerank 후 상위 조항 후보를 반환. 조문(article) 단위로 채택·adoption_reason을 붙여 에이전트에 전달. 대화형 세션 메모리 대신 run_id별 이벤트 목록·최종 결과를 stream_runtime에 보관하고, 이벤트 API·SSE로 조회.

* **주요 기술:** PostgreSQL pgvector(rag_chunk), halfvec 임베딩, OpenAI Embeddings, chunking_pipeline·rag_chunk_lab_service(청킹 전략), policy_service(BM25·Dense·RRF·Rerank), stream_runtime(메모리 타임라인·결과).

### 주요 문제 해결 및 기술 리서치

구현 과정에서 마주친 기술적 문제와 이를 해결하기 위해 **찾아본 자료(리서치)** 및 **적용한 방법**을 기록합니다.

| **이슈 구분** | **문제 상황 및 원인** | **리서치 및 해결 과정 (Reference & Solution)** |
| --------- | ------------------------------------ | ----------------------------------------------------------------------------------------- |
| **워크플로우** | (해결됨) 테스트 데이터 생성 시 스크리닝까지 하기로 했으나, 초기에는 분석 run에서도 screener가 매번 실행되어 사전 스크리닝된 전표에서 중복 연산이 발생했음 | • **리서치:** LangGraph conditional_edges로 진입 분기 가능 여부 확인 • **적용:** 분석 시에는 스크리닝 로직 제거 방향으로 설계 변경. start_router 노드 추가 — body_evidence.case_type 존재(사전 스크리닝됨) 시 screener를 실행하지 않고 screening_result만 주입한 뒤 intake로 직행. 미스크리닝 건만 screener 실행. intake에서 사전 스크리닝 시 SCREENING_RESULT 이벤트 1회 발행해 UI 호환 유지 |
| **HITL 재개** | HITL 응답 제출 후 재실행 시 체크포인트 없어 screener부터 전부 재실행됨 | • **리서치:** LangGraph checkpointing(memory vs postgres) 문서 • **적용:** CHECKPOINTER_BACKEND=postgres 사용 시 langgraph-checkpoint-postgres로 DB에 체크포인트 저장. 재개 시 hitl_validate → reporter → finalizer만 실행. PoC 기본은 memory(단일 프로세스 내에서만 재개 시 이어서 실행) |
| **도구 연동** | planner가 생성한 plan과 execute 노드의 도구 호출 스키마 불일치 | • **리서치:** LangChain StructuredTool, ToolContextInput 스키마 • **적용:** ToolContextInput(Pydantic)으로 body_evidence·prior_tool_results·intended_risk_type 등 통일. execute 노드에서 plan의 tool 이름으로 레지스트리에서 도구 조회 후 동일 스키마로 호출 |
| **RAG 검색** | 규정 조항이 많을 때 단일 벡터 검색만으로 관련도 낮은 조문 상위 노출 | • **리서치:** BM25·Dense 하이브리드, RRF(Reciprocal Rank Fusion), cross-encoder rerank • **적용:** policy_service에서 BM25·Dense 후 RRF 결합, (선택) sentence-transformers rerank 또는 LLM fallback. 조문(article) 단위로 중복 제거 후 상위 N개 채택 |
| **CoT 일관성** | 노드 reasoning 텍스트와 구조화 출력(verdict 등)이 서로 모순되는 경우 | • **리서치:** 출력 검증·일관성 체크 패턴 • **적용:** check_reasoning_consistency()로 reasoning 문장과 recommend_hold/verdict 등 필드 비교, 모순 시 conflict_description 반환해 재생성 또는 HITL 유도 |

### 핵심 동작 검증

위에서 구현한 기능이 의도대로 동작하는지 보여주는 **대표적인 실행 결과**를 첨부합니다.

**[검증 시나리오:** 휴일 사용 의심 전표에 대한 분석 실행(스크리닝 결과 확인 + 심층 분석 + 규정 인용)**]**

* **입력:** 시연 데이터 제어에서 「휴일 사용 의심」 시나리오로 생성한 전표 1건 선택 후 「분석 실행」 클릭. (전표: 주말/심야 발생, hrStatus=LEAVE, mccCode=5813, budgetExceeded=N 등)

* **에이전트 동작:**

  1. start_router에서 body_evidence.case_type 존재 → screening_result 주입, intake로 직행. (또는 미스크리닝 시 screener 실행 후 intake)

  2. intake에서 플래그 추출, SCREENING_RESULT 이벤트 발행(사전 적용 시). planner에서 holiday_compliance_probe, document_evidence_probe, policy_rulebook_probe 등 plan 생성.

  3. execute에서 holiday_compliance_probe → document_evidence_probe → policy_rulebook_probe 순 호출. policy_rulebook_probe가 규정집에서 관련 조항(예: 업무시간 외·휴일 사용) 검색 후 채택 조항·adoption_reason 반환.

  4. critic에서 도구 결과 교차 검토, verify에서 점수·HITL 필요 여부 판단. (필요 시 hitl_pause에서 interrupt)

  5. reporter·finalizer에서 근거 기반 판단요약·최종 결과 생성. SSE로 NODE_START/END, TOOL_CALL/TOOL_RESULT, SCREENING_RESULT 등 스트리밍.

* **최종 결과:**

  * 실시간 사고 흐름 탭에 스크리닝 완료(사전 적용)·intake·planner·execute(도구별)·critic·verify·reporter·finalizer 이벤트 표시.
  * 최종 결과 탭에 규정 조항 인용(예: 제11조 2항 등), score_breakdown, 판단요약(휴일/심야 사용 위반·환수 또는 소명 권고 등) 표시.
  * (HITL 발생 시) 담당자 응답 제출 후 「분석 이어하기」로 hitl_validate → reporter → finalizer만 재실행되어 갱신된 판단요약 확인 가능.
