### 핵심 구현 내용

이번 PoC 단계에서 실제 코드로 구현된 핵심 기능들을 **동작 원리**와 **사용 기술** 중심으로 상세히 기술합니다.

**1.1 에이전트 워크플로우 (Agent Workflow)**

* **구현 기능:** 전표 단위 분석 run에서 start_router → (screener 또는 intake) → planner → execute → critic → verify → hitl_pause(선택)/hitl_validate → reporter → finalizer 순의 LangGraph 워크플로우. 사전 스크리닝된 전표는 screener 노드를 건너뛰고 intake부터 진행.

* **동작 원리:** 사용자가 「분석 실행」을 누르면 FastAPI가 `body_evidence`를 LangGraph에 주입. **start_router**에서 `body_evidence.case_type` 존재 여부로 분기 — 있으면 screening_result를 상태에 넣고 **intake**로 직행, 없으면 **screener**에서 하이브리드 스크리닝(LLM 초기 분류 + 결정론적 가드레일 보정) 후 intake로 진행. intake에서 플래그 추출·멀티모달 OCR 결과 반영, **planner**에서 케이스 유형별 조사 계획(plan) 및 도구 목록 결정, **execute**에서 LangChain StructuredTool 순차/병렬 호출, **critic**에서 과잉 주장·반례 검토 (최대 2회 루프), **verify**에서 하이브리드 점수 산정·HITL 필요 여부 판단. HITL 필요 시 **hitl_pause**에서 interrupt 후 담당자 응답 제출 시 같은 run으로 resume → hitl_validate → reporter → finalizer. **reporter**·**finalizer**에서 근거 기반 판단요약·최종 결과 생성.

* **주요 기술:** LangGraph(StateGraph, conditional_edges, interrupt/resume), LangChain Core, AgentState(TypedDict), Pydantic 구조화 출력(PlannerOutput, ExecuteOutput, CriticOutput, VerifierOutput, ReporterOutput), SSE 스트리밍(NODE_START/END, PLAN_READY, TOOL_CALL/TOOL_RESULT, SCREENING_RESULT, HITL_REQUESTED, GATE_APPLIED). 체크포인터(memory 또는 postgres)로 HITL 재개 시 상태 복원.

* **핵심 파일:** `agent/langgraph_agent.py`(그래프 빌드), `agent/langgraph_nodes.py`(노드 디스패처), `agent/langgraph_nodes_ingest.py`(start_router/screener/intake/planner), `agent/langgraph_nodes_execute.py`(execute), `agent/langgraph_nodes_review.py`(critic/verify), `agent/langgraph_nodes_outcome.py`(hitl_pause/hitl_validate/reporter/finalizer)

**1.2 하이브리드 스크리닝 (Hybrid Screening)**

* **구현 기능:** `rule`(결정론적 시그널 기반)과 `hybrid`(LLM 초기 분류 + 결정론적 가드레일 보정) 2종 스크리닝 모드. 기본값 `hybrid`. 5종 케이스 유형(HOLIDAY_USAGE / LIMIT_EXCEED / PRIVATE_USE_RISK / UNUSUAL_PATTERN / NORMAL_BASELINE) 자율 분류.

* **동작 원리:**
  1. 전표 필드(isHoliday, hrStatus, occurredAt, mccCode, budgetExceeded, amount)에서 시그널 추출 및 결정론적 점수 산정(isHoliday +35, is_leave +20, isNight +10, budgetExceeded +10, MCC 위험도 +15~+30)
  2. LLM(gpt-4o-mini, JSON 응답 형식)이 동일 입력을 보고 case_type·confidence 제안
  3. **가드레일 정렬:** LLM 제안이 결정론적 시그널과 충돌하거나 confidence < 0.75이면 결정론적 결과 채택 (예: 휴일 시그널 없이 HOLIDAY_USAGE 제안 → 결정론적 결과 우선)
  4. 결과에 `screening_mode`, `hybrid_case_align_reason`, `llm_confidence` 기록

* **주요 기술:** `agent/screener.py`(`run_screening()`, `_run_hybrid()`, `_align_hybrid_case_type()`), OpenAI/AzureOpenAI client, `utils/config.py`(`SCREENING_LLM_OVERRIDE_MIN_CONFIDENCE=0.75`, `MCC_HIGH_RISK`, `MCC_LEISURE`, `MCC_MEDIUM_RISK`)

**1.3 도구(Tool) 및 함수 연동**

* **구현 기능:** 전표·규정 기반 위험 검증을 위한 6개 Probe 도구. `holiday_compliance_probe`, `budget_risk_probe`, `merchant_risk_probe`, `document_evidence_probe`, `policy_rulebook_probe`, `legacy_aura_deep_audit`(선택).

* **동작 원리:** planner가 케이스 유형(휴일/한도/업종 등)과 플래그에 따라 도구 목록(plan)을 생성. execute 노드에서 plan 순서대로 LangChain StructuredTool을 호출. 각 도구는 `ToolContextInput`(body_evidence, prior_tool_results 등)을 받아 DB·RAG·전표 데이터를 조회하고, `ToolResultEnvelope(tool, ok, facts, summary)` 형태로 결과 반환. `policy_rulebook_probe`·`legacy_aura_deep_audit`는 prior 결과를 참조하므로 순차 실행. 나머지는 `ENABLE_PARALLEL_TOOL_EXECUTION=true` 시 병렬 그룹 실행. `merchant_risk_probe`는 `holiday_compliance_probe` 결과에 의존(복합 위험 탐지).

* **도구 스킵 로직:** `policy_refs` ≥ 2건이고 `lineItemCount` > 0이며 입력 누락 없고 예산 초과 없으면 `legacy_aura_deep_audit`를 자동 스킵하여 불필요한 호출 억제.

* **주요 기술:** LangChain StructuredTool, Pydantic `ToolContextInput`, `agent/agent_tools.py`(도구 handler 함수), `policy_service.search_policy_chunks()`(BM25/Dense/RRF/Rerank), `utils/config.py`(`get_mcc_sets()`)

**1.4 데이터 및 메모리 (RAG & Context)**

* **구현 기능:** 사내 경비 지출 규정 텍스트의 청킹·임베딩·저장 및 분석 시 규정 조항 검색(RAG). 규정문서 라이브러리(Streamlit)에서 TXT 업로드·전략별 청킹 미리보기·품질 리포트. run 단위 타임라인·최종 결과는 메모리(stream_runtime) + DB 영속화.

* **동작 원리:** 규정 TXT는 `chunking_pipeline`으로 계층/조항/슬라이딩 전략에 따라 청크 분할 후 PostgreSQL `rag_chunk` 테이블에 저장. OpenAI 호환 임베딩(text-embedding-3-large, 3072차원, halfvec) 생성 후 HNSW 인덱스(m=16, ef_construction=64)로 저장. 분석 시 `policy_rulebook_probe`가 케이스·키워드로 `policy_service.search_policy_chunks()`를 호출해 BM25·Dense(RRF)·Rerank 후 상위 조항 후보를 반환. 조문(article) 단위로 채택·`adoption_reason`을 붙여 에이전트에 전달. `policy_ref_normalizer.py`로 규정 참조 형식 통일(환각 방지).

* **주요 기술:** PostgreSQL pgvector(rag_chunk), halfvec(3072차원), OpenAI Embeddings, `services/chunking_pipeline.py`·`services/rag_chunk_lab_service.py`(청킹 전략), `services/policy_service.py`(BM25·Dense·RRF·Rerank), `services/policy_ref_normalizer.py`(정규화), `services/stream_runtime.py`(메모리 타임라인)

**1.5 멀티모달 증빙 추출 (Multimodal OCR)**

* **구현 기능:** 업로드된 이미지 영수증(PNG/JPG)에서 금액·승인일자·업종·가맹점명을 자동 추출하고 전표 데이터와 대조.

* **동작 원리:** `services/evidence_extraction.py`에서 파일 SHA-256 해시 기반 캐싱 확인. pytesseract 설치 시 OCR 수행, 미설치 시 알려진 영수증 해시(demo stub)로 구조화 엔티티 반환. `services/evidence_compare_service.py`가 추출 엔티티와 전표(body_evidence) 필드를 대조하여 불일치 감지. intake 노드에서 `body_evidence.extracted_entities`를 읽어 `visual_audit_results`로 AgentState에 반영. 증빙 불일치 시 verify 노드의 HITL 승격 가능성 증가.

* **주요 기술:** `services/evidence_extraction.py`(OCR·캐싱), `services/evidence_compare_service.py`(대조), `utils/ocr_paddle.py`(PaddleOCR 선택), pytesseract, Azure Document Intelligence(프로덕션 연동 시)

**1.6 하이브리드 스코어링 (Hybrid Scoring)**

* **구현 기능:** 결정론적 규칙 기반 점수(rule_score)와 LLM judge 점수(llm_score)를 가중 합산하여 최종 점수(final_score)·심각도 산정. Circuit breaker로 LLM judge 오류 시 안전하게 fallback.

* **동작 원리:** `agent/langgraph_scoring.py`에서 정책 신호 기본점 + 도구 기반 가산점 + 복합 위험 승수(compound_multiplier) + 금액 승수(amount_multiplier)로 rule_score 산정. `LLM_JUDGE_ENABLED=true` 시 LLM judge가 동일 증거를 보고 0~100 점수와 reasoning 반환. rule_score·llm_score 차이가 `LLM_JUDGE_CONFLICT_THRESHOLD`(기본 20점) 이상이면 재평가 유도. Circuit breaker(`llm_judge_circuit_window=10`, `failure_threshold=0.5`)가 LLM judge 연속 실패 시 자동 차단, rule_score로 안전하게 fallback.

| 점수 구성 요소 | 점수 |
|---------------|------|
| isHoliday | +35 |
| hrStatus 충돌 (휴가/결근) | +20 |

### 주요 문제 해결 및 기술 리서치

구현 과정에서 마주친 기술적 문제와 이를 해결하기 위해 **찾아본 자료(리서치)** 및 **적용한 방법**을 기록합니다.

| **이슈 구분** | **문제 상황 및 원인** | **리서치 및 해결 과정 (Reference & Solution)** |
| ----------- | --------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **워크플로우** | 테스트 데이터 생성 시 스크리닝까지 수행했으나, 초기 설계에서는 분석 run에서도 screener가 매번 실행되어 사전 스크리닝된 전표에서 중복 연산 발생 | **리서치:** LangGraph conditional_edges로 진입 분기 가능 여부 확인. **적용:** `start_router` 노드 추가 — `body_evidence.case_type` 존재(사전 스크리닝됨) 시 screener를 실행하지 않고 screening_result만 주입한 뒤 intake로 직행. 미스크리닝 건만 screener 실행. intake에서 사전 스크리닝 시 SCREENING_RESULT 이벤트 1회 발행해 UI 호환 유지 |
| **HITL 재개** | HITL 응답 제출 후 재실행 시 체크포인트 없어 screener부터 전부 재실행됨 | **리서치:** LangGraph checkpointing(memory vs postgres) 공식 문서 검토. **적용:** `CHECKPOINTER_BACKEND=postgres` 사용 시 `langgraph-checkpoint-postgres`로 DB에 체크포인트 저장. 재개 시 hitl_validate → reporter → finalizer만 실행. PoC 기본은 memory(단일 프로세스 내에서 이어서 실행 보장) |
| **도구 연동** | planner가 생성한 plan과 execute 노드의 도구 호출 스키마 불일치 | **리서치:** LangChain StructuredTool, ToolContextInput 스키마 설계. **적용:** `ToolContextInput`(Pydantic)으로 body_evidence·prior_tool_results·intended_risk_type 등 통일. execute 노드에서 plan의 tool 이름으로 레지스트리에서 도구 조회 후 동일 스키마로 호출 |
| **RAG 검색 품질** | 규정 조항이 많을 때 단일 벡터 검색만으로 관련도 낮은 조문 상위 노출. 키워드 검색과 시맨틱 검색 결과 불일치 | **리서치:** BM25·Dense 하이브리드, RRF(Reciprocal Rank Fusion), cross-encoder rerank 기법 리서치. **적용:** `policy_service.search_policy_chunks()`에서 BM25·Dense 후 RRF 결합, (선택) sentence-transformers rerank 또는 LLM fallback(`ENABLE_LLM_RERANK_FALLBACK=true`). 조문(article) 단위로 중복 제거 후 상위 N개 채택 |
| **CoT 일관성** | 노드 reasoning 텍스트와 구조화 출력(verdict 등)이 서로 모순되는 경우 발생 | **리서치:** LLM 출력 검증·일관성 체크 패턴 조사. **적용:** `check_reasoning_consistency()`로 reasoning 문장과 recommend_hold/verdict 등 필드 비교. 모순 시 `conflict_description` 반환해 재생성 또는 HITL 유도(`no_citation`, `contradictory_evidence` 감지 시 fail-closed HITL 강제) |
| **스크리닝 환각** | LLM 스크리닝이 시그널 없는 케이스 유형을 높은 확신도로 제안하는 경우 발생(예: 휴일 시그널 없이 HOLIDAY_USAGE 제안) | **리서치:** LLM 분류 신뢰도 threshold 및 결정론적 보정 패턴 조사. **적용:** `_align_hybrid_case_type()`에 시그널별 가드레일 규칙 추가. 신뢰도 < 0.75이거나 시그널 충돌 시 결정론적 결과 채택. `hybrid_case_align_reason` 필드로 보정 사유 기록 |

### 핵심 동작 검증

위에서 구현한 기능이 의도대로 동작하는지 보여주는 **대표적인 실행 결과**를 첨부합니다.

**[검증 시나리오: 휴일 사용 의심 전표에 대한 분석 실행 (스크리닝 + 심층 분석 + 규정 인용)]**

* **입력:** 시연 데이터 제어에서 「휴일 사용 의심」 시나리오로 생성한 전표 1건 선택 후 「분석 실행」 클릭. (전표: 주말/심야 발생, hrStatus=LEAVE, mccCode=5813, budgetExceeded=N 등)

* **에이전트 동작:**

  1. start_router에서 body_evidence.case_type 존재 → screening_result 주입, intake로 직행. (또는 미스크리닝 시 screener 실행 후 intake)

  2. intake에서 플래그 추출 (isHoliday=true, isNight=true, hrStatus=LEAVE 등), SCREENING_RESULT 이벤트 발행. planner에서 holiday_compliance_probe, document_evidence_probe, policy_rulebook_probe 등 plan 생성.

  3. execute에서 holiday_compliance_probe → document_evidence_probe → policy_rulebook_probe 순 호출. policy_rulebook_probe가 규정집에서 관련 조항(예: 업무시간 외·휴일 사용) 검색 후 채택 조항·adoption_reason 반환.

  4. critic에서 도구 결과 교차 검토, verify에서 하이브리드 점수 산정·HITL 필요 여부 판단. (필요 시 hitl_pause에서 interrupt → 담당자 응답 → resume)

  5. reporter·finalizer에서 근거 기반 판단요약·최종 결과 생성. SSE로 NODE_START/END, TOOL_CALL/TOOL_RESULT, SCREENING_RESULT, GATE_APPLIED 등 스트리밍.

* **최종 결과:**

실시간 사고 흐름 탭에 스크리닝 완료(사전 적용)·intake·planner·execute(도구별)·critic·verify·reporter·finalizer 이벤트 표시. 최종 결과 탭에 규정 조항 인용(예: 제11조 업무시간 외 결제·휴일 사용 관련 조항), score_breakdown(isHoliday +35, hrStatus +20, isNight +10, merchantRisk HIGH +20, policy_refs 3건 +22 등), 심각도 HIGH/CRITICAL, "휴일/심야 사용 위반 — 환수 또는 소명 권고" 수준의 판단요약 표시. (HITL 발생 시) 담당자 응답 제출 후 「분석 이어하기」로 hitl_validate → reporter → finalizer만 재실행되어 갱신된 판단요약 확인 가능.

* **휴일 사용 의심 케이스로 검토 필요 항목으로 분석된 결과 화면 (SSE 스트림 수신 및 점수 산정 결과 화면 캡처)**

### Verify/HITL 정량 기준 (코드 기준 명시)

**2.1 점수 산정 로직** (`agent/langgraph_scoring.py`)

- 정책 신호 기본점 (위 점수표 참조)
- 도구 기반 가산 (위 점수표 참조)
- 가중치/보정: `compound_multiplier`(고위험 신호 2/3/4개 이상 시 ×1.15/×1.3/설정최대), `amount_multiplier`(금액 구간별 연속 함수, 최대 `SCORE_AMOUNT_MULTIPLIER_MAX`)
- `policy_weight = 0.75 - 0.25 * evidence_completeness`, `evidence_weight = 1 - policy_weight`
- 고위험(≥3) + 증거완결도 < 0.4이면 보수 패널티 적용

**2.2 Evidence Coverage Gate** (`services/evidence_verification.py`)

- 기본 임계값: `hold < 0.5`, `caution < 0.7`, `regenerate_citations < 1.0`
- 동적 상향: severity/final_score/compound_multiplier에 따라 hold·caution threshold 증가

**2.3 verify 노드 HITL/재평가 조건** (`agent/langgraph_nodes_review.py`)

- HITL 생성 기본: `build_hitl_request(...)` 결과가 존재하면 `needs_hitl=true`
- Fail-closed 강제 HITL: `no_citation`, `contradictory_evidence` 감지 시 강제
- NORMAL_BASELINE 자동 READY 예외: blocking 이슈/누락/핵심도구 실패가 없으면 `hitl_request` 제거
- 재평가(planner 재진입): `abs(rule_score - llm_score) ≥ 20` 또는 `fidelity < 40`이고 `retry_count < max_retries`일 때 HITL보다 재평가 우선

### Tool/Probe I/O 계약 (스키마 고정)

**3.1 공통 입력 스키마** (`ToolContextInput`, Pydantic)

- `case_id: str` (필수)
- `body_evidence: dict` (필수, default `{}`)
- `intended_risk_type: str | None`
- `prior_tool_results: list[dict]` (default `[]`)

**3.2 공통 출력 봉투** (`ToolResultEnvelope`)

- `tool: str`
- `ok: bool`
- `facts: dict`
- `summary: str`

**3.3 각 도구 facts 계약 (현재 구현)**

- `holiday_compliance_probe`: `occurredAt`, `isHoliday`, `hrStatus`, `holidayRisk`
- `budget_risk_probe`: `amount`, `budgetExceeded`
- `merchant_risk_probe`: `mccCode`, `merchantName`, `merchantRisk`, `compoundRiskFlags`, `holidayRiskConsidered`
- `document_evidence_probe`: `lineItemCount`, `lineItems`
- `policy_rulebook_probe`: `policy_refs`, `ref_count`, `retrieval_candidates`, `enriched_from_prior`
- `legacy_aura_deep_audit`: 활성화 시 bridge 결과, 비활성화 시 `disabled=true`

**3.4 빈값 처리 규칙 (핵심)**

- `body_evidence` 누락 필드: default/빈 dict 처리 후 보수적으로 HITL 승격
- `required_inputs`/`review_questions`: 최대 2개 제한 (UI 표시 복잡도 제어)
- `NORMAL_BASELINE`: critical required input 없으면 HITL 무시 가능 (자동 완료 우선)
- LLM 응답 파싱 실패 시: Pydantic 기본값으로 graceful fallback, `fallback_used=true` 기록
