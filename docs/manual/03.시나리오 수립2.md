### 핵심 사용자 시나리오

본 문서는 사용자 관점 시나리오를 정의하되, **AI Master 평가 관점(Agentic AI 구현 증빙)**에서 바로 점검 가능한 형태로 구성한다.

- 핵심 시연 목표: 단순 분류가 아니라 **자율형 워크플로우(LangGraph)**, **도구 실행(LangChain Tool)**, **근거 검색(RAG)**, **HITL 재개**를 한 흐름으로 증명
- 시연 데이터 생성 시점에 스크리닝이 먼저 반영되고, 워크스페이스 분석 실행 시 심층 분석이 이어지는 구조
- PoC 시연 권장 조합: 필수 2개 + 확장 2개

---

### 시연 시나리오 포트폴리오 (평가 관점)

| 구분 | 시나리오 | 목적 | 평가 포인트 |
|---|---|---|---|
| 필수 | HOLIDAY_USAGE(휴일 사용 의심) | 고위험 탐지 + 규정 인용 + HITL 유도 | 규정 정합성, 질문 생성 품질 |
| 필수 | NORMAL_BASELINE(정상 비교군) | 정상건 자동 통과 경로 검증 | 과검출 억제, False Positive 제어 |
| 확장 | LIMIT_EXCEED(한도 초과 의심) | 금액/승인 규정 중심 분석 | 케이스-조항 정합 필터 정확성 |
| 확장 | PRIVATE_USE_RISK 또는 UNUSUAL_PATTERN | 업종/패턴 리스크 검증 | 도구 합의도, 근거 충실도 |

설명: 필수 2개만으로도 데모는 가능하지만, 평가자가 Agentic AI 완성도를 볼 때는 **서로 다른 리스크 메커니즘(시간/금액/업종/패턴)**을 최소 3종 이상 보여주는 구성이 유리하다.

---

**시나리오 1: [생성 시점 스크리닝 + 목록 반영 확인]**

- **ID:** SC-001
- **상황:** 시연 데이터 제어에서 전표를 생성하면 백엔드가 즉시 스크리닝을 수행하고, 워크스페이스 목록/상세에 분류 결과가 반영된다.
- **목표:** 스크리닝 결과(`case_type`, `screening_reason_text`)가 생성→조회→분석까지 일관되게 유지됨을 확인
- **사전 조건:** `시연데이터 생성(beta)` 또는 `시연데이터제어(legacy)`에서 케이스별 전표 생성 가능

| 단계 | 사용자 행동 | 시스템 동작 | 검증 기준 |
|---|---|---|---|
| 1 | 케이스 유형 선택 후 데이터 생성 | 전표 저장 + 스크리닝 실행 + `agent_case` 반영 | 생성 직후 유형/사유 조회 가능 |
| 2 | 워크스페이스 목록/상세 확인 | 배지/사유 표시 | 목록과 상세 값 일치 |
| 3 | 분석 실행 | 사전 스크리닝건은 start_router 후 intake 직행 | 중복 스크리닝 방지 |

- **성공 기준**
1. 생성 직후 스크리닝 결과가 UI에서 확인된다.
2. 분석 실행 시 사전 스크리닝 결과가 재사용된다.

---

**시나리오 2: [심층 분석 + 규정 근거 + HITL 재개]**

- **ID:** SC-002
- **상황:** 고위험 전표를 분석하고, 검토자 응답(HITL) 후 같은 run에서 결과를 갱신한다.
- **목표:** `planner→execute→critic→verify→hitl_pause/hitl_validate→reporter→finalizer` 흐름과 근거 기반 최종 판단 증명
- **사전 조건:** 규정집 인덱싱 완료, 분석 대상 전표 존재

| 단계 | 사용자 행동 | 시스템 동작 | 검증 기준 |
|---|---|---|---|
| 1 | 분석 실행 클릭 | SSE로 노드/도구 이벤트 스트리밍 | NODE/TOOL 이벤트 수신 |
| 2 | HITL 팝업 응답 제출 | interrupt-resume 수행 | 같은 run_id 재개 |
| 3 | 재분석 완료 확인 | 최종 판단/근거/점수 갱신 | 응답 반영 확인 |

- **성공 기준**
1. 규정 조항 + adoption_reason + score_breakdown이 함께 제공된다.
2. HITL 제출 후 동일 run 재개가 가능하다.

---

**시나리오 3: [한도 초과 규정 정합성 검증]**

- **ID:** SC-003
- **상황:** LIMIT_EXCEED 전표에서 휴일 전용 조항이 과다 채택되는지 확인
- **목표:** 케이스-조항 정합 필터(`services/policy_case_alignment.py`)가 비관련 조항을 억제하는지 검증

- **성공 기준**
1. 한도 초과 핵심 조항(예: 금액/승인 관련)이 우선 채택된다.
2. 맥락 불일치 조항(예: 휴일 제약)이 불필요하게 채택되지 않는다.

---

**시나리오 4: [정상 비교군 안전성 검증]**

- **ID:** SC-004
- **상황:** NORMAL_BASELINE 전표 분석 시 불필요한 HITL·위험 판정이 과도하게 발생하지 않는지 확인
- **목표:** 정상건 자동 진행 경로와 보수적 예외 처리 균형 검증

- **성공 기준**
1. 정상건은 blocking 이슈 없을 때 READY/통과 경로를 유지한다.
2. 근거 부족/충돌이 있을 때만 제한적으로 HITL이 발생한다.

### 평가자 관점 체크리스트

1. LangGraph 분기/재개가 실제로 동작했는가(이벤트/이력으로 검증 가능한가)
2. 도구 결과가 최종 판단 문장과 논리적으로 연결되는가
3. RAG 조항 채택이 케이스 맥락과 정합한가
4. HITL이 형식적 절차가 아니라 판단 품질을 실제로 개선하는가
5. 정상/이상 케이스 모두에서 과검출/누락을 통제하는가

위 5개를 충족하면, 본 시나리오 문서는 "기능 나열"이 아니라 "Agentic AI 완성도"를 설명하는 평가 대응 문서로 기능한다.
